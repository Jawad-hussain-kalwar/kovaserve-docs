title: Sovereign deployment
description: For regulated industries, public sector, and any environment with strict data-residency requirements. Same KovaServe semantics, customer-controlled infrastructure end-to-end.

A sovereign deployment runs KovaServe entirely inside your perimeter — your Kubernetes cluster, your object storage, your PostgreSQL, your TLS certificates, your network. KovaServe never sees the data; the semantic model is identical to managed cloud. This is the deployment path for regulated industries, public sector, and any environment with strict data-residency requirements.

## What "sovereign" means here

Three properties define a sovereign deployment:

1. **Data residency.** Every byte of customer data — context manifests, artifacts, checkpoints, KV exports, usage records — stays in customer-controlled storage.
2. **No control-plane callback.** The KovaServe control plane runs locally. There is no phone-home, no telemetry to KovaServe servers.
3. **License-controlled.** A signed offline license file gates feature access. KovaServe never reaches the deployment to verify.

## Install

The base install is identical to [self-hosting on Kubernetes](/docs/guides/deployment/self-hosting-kubernetes). Three additional configuration steps make it sovereign:

### 1. Offline license file

```yaml
license:
  secretName: kovaserve-license
```

The license file is a signed JWT covering the licensed tenants, features, expiry, and signing key. Generated by KovaServe's licensing tooling and delivered out-of-band; mounted as a Kubernetes Secret. The gateway verifies the signature on startup and refuses to start with an invalid or expired license.

### 2. Disable telemetry

```yaml
telemetry:
  outbound:
    enabled: false      # default in sovereign mode
```

When `telemetry.outbound.enabled` is `false`, the gateway emits **no** outbound HTTP — no metrics, no error reports, no version checks. The deployment is fully air-gapped from KovaServe's infrastructure.

### 3. Data-boundary defaults

```yaml
defaults:
  policy_scope:
    data_boundary: eu_only       # or "us_only", "national", etc.

objectStorage:
  buckets:
    eu_only: ks-context-artifacts
```

Every Tenant / Project created without an explicit PolicyScope inherits the deployment's default. All artifact uploads route to the configured regional bucket.

## Regional bucket layout

For deployments with multi-region buckets:

```yaml
objectStorage:
  buckets:
    default: ks-context-artifacts
    eu_only: ks-context-artifacts-eu
    us_only: ks-context-artifacts-us
    canada_only: ks-context-artifacts-ca

defaults:
  policy_scope:
    data_boundary: canada_only
```

The PolicyScope `data_boundary` field selects which bucket artifacts route to. See [policy scopes](/docs/guides/policy/policy-scopes) and [managing artifacts](/docs/guides/context/managing-artifacts).

## Compliance and audit

Sovereign deployments inherit every KovaServe compliance feature:

- **Audit trail.** Every model call, tool call, policy decision, and outcome is recorded in PostgreSQL and exportable as JSONL.
- **Context manifest immutability.** Every manifest is written once and is forever queryable for audit.
- **Cost identity** (`gross - savings = net`) holds exactly for all historical records.
- **Pricing snapshots** on `usage_records` rows make replay-of-history audit trivial.

The KovaServe-side compliance posture (SOC 2, ISO 27001) does not apply to sovereign deployments — the customer's posture does. KovaServe's licensing terms cover the software; deployment compliance is the customer's responsibility.

## Operations

Sovereign deployments are operated by the customer (or by a KovaServe partner under contract). Upgrades, scaling, disaster recovery, and on-call are not provided by KovaServe in this mode.

For partner-operated sovereign deployments — where a neocloud or systems integrator runs KovaServe on behalf of end-customers — the same chart and license model applies; the partner provides operations.

## Switching to sovereign from managed cloud

Application code does not change. Migrate by:

1. Installing the sovereign cluster.
2. Provisioning the same Tenant / Project / Environment structure.
3. Re-issuing keys.
4. Switching `KOVASERVE_BASE_URL` on the application side.

Run history does not migrate automatically. Export historical data from managed cloud as JSONL via `kova usage export` if you need it on the sovereign side — every row is self-contained for audit, so replays work across deployment modes.
