The **control plane** is Layer 2 of the KovaServe layered model and the commercial center of the system. It sits above the open [execution substrate](/docs/concepts/execution-substrate) and below application and runtime layers, and its job is to convert substrate signals into the semantics that make long-running AI execution productizable: continuity, checkpoint-aware resumability, budget-aware execution control, governance, observability, and managed/private deployment portability. It is the system of record for every entity in the canonical execution model — `Run`, `Session`, `Step`, `ModelCall`, `ContextManifest`, `ContextSegment`, `ContextArtifact`, `CompactionRecord`, `Checkpoint`, `MemoryScope`, `MemoryEntry`, `ToolCatalog`, `ToolContextDisposition`, `PolicyScope`, `Budget`, `ContextBudget`, `PlacementHint`, `StateHandle`.

The control plane ships as a modular monolith — a single Python/FastAPI process composed of cleanly bounded modules that talk through `Protocol` interfaces — paired with per-node agents and per-process workers where physical separation is required. The semantic surface is identical in [managed cloud](/docs/concepts/managed-cloud) and [private deployment](/docs/concepts/private-deployment); only the operational topology differs.

## Subsystems

The control plane is composed of the following subsystems. Each subsystem has a clear ownership boundary and a stable API to its neighbors.

### API Gateway

The Gateway is the canonical entry point into the control plane. It hosts both the OpenAI-compatible inference surface and the KovaServe-native control-plane surface.

| Responsibility | Detail |
|---|---|
| Authentication | Bearer keys against the [Tenant Management](#tenant-management) hierarchy. |
| Tenancy resolution | Resolves `(organization_id, project_id, environment_id)` from the key on every call. |
| Request normalization | Strips the [`kovaserve` extension envelope](/docs/api/index#extension-envelope) before forwarding to the substrate; surfaces it as canonical metadata. |
| Runtime metadata ingestion | Accepts the 16-field runtime metadata set used by [BYO runtime](/docs/concepts/byo-runtime) integrations. |
| Idempotency handling | Honors the [idempotency contract](/docs/api/idempotency) per `(tenant_id, idempotency_key)`. |
| Streaming | SSE conventions, including final-chunk cost delivery. |
| Front-door policy checks | Coarse `PolicyScope` checks before deeper subsystems are invoked. |
| Tracing | OpenTelemetry trace correlation across the entire control plane. |

The Gateway is a thin discriminator and ingress, not a business-logic layer. Discrimination rule: `kovaserve` extension field present on `/v1/chat/completions` engages the v2 path with full canonical execution; absent engages anonymous v1 mode (still metered, still tenanted, no `Run` row).

### Canonical Execution Service

Owns authoritative records for [`Run`](/docs/concepts/runs), [`Session`](/docs/concepts/sessions), [`Step`](/docs/concepts/steps), and [`ModelCall`](/docs/concepts/model-calls). It is the semantic ledger of execution.

| Responsibility | Detail |
|---|---|
| Entity creation | Run on explicit `POST /v1/ks/runs` or implicit creation from the first inference call carrying `kovaserve.run_id`. |
| State transitions | The 12-state [Run FSM](/docs/concepts/run-state-machine), the [Step FSM](/docs/concepts/step-state-machine), and the lifecycle timestamps. |
| Lineage | `parent_run_id` for [sub-runs](/docs/concepts/sub-runs); `summary_artifact_id` for compact handoff back to parent. |
| External references | `external_run_id` and `external_runtime` for runtimes integrated via [Runtime Integration Adapters](#runtime-integration-adapters). |
| Timeline reconstruction | Embedded Steps and ModelCalls on `GET /v1/ks/runs/{id}` so clients reconstruct without N+1 calls. |
| Deferred Step admission | Step rows are created only after admission succeeds — a rejected admission inserts no Step. |

Every Step, ModelCall, Checkpoint, and timeline event in the system traces back to exactly one Run owned by this service.

### Global Context Manager (GCM v2)

GCM is the producer of [`ContextManifest`](/docs/concepts/context-manifests) records. It is the execution-boundary owner of context shape, identity, and accounting; it is **not** a prompt composition engine and **not** a retrieval framework.

| Responsibility | Detail |
|---|---|
| Validate | Inbound context blocks against the canonical `ContextType × MemoryScope` matrix. |
| Normalize | Segment metadata — digest, token count, byte count, checkpoint relevance, scope, retention. |
| Identify | Assign segment IDs and content digests. |
| Order | Deterministically order segments per ADR-003. |
| Identify (legacy) | Compute `stable_prefix_id` for Pass 4 router compatibility. |
| Identify (compatibility-aware) | Compute `stable_context_id` from the digest cascade plus renderer version, model family, tokenizer fingerprint, and tenant/project/context scope. |
| Account | Stable, volatile, active, and offloaded token counts. |
| Materialize | Artifact-backed segments by reference (preview text, never full payload). |
| Produce | The durable [`ContextManifest`](/docs/concepts/context-manifests) consumed by Admission, Router, Checkpoint Manager, and the runtime. |
| Expose | Checkpoint-relevant segment refs to the Checkpoint Manager. |
| Render | Deterministic model-dispatch output with multi-tier prefix breakpoints (tools / system / history-prefix / summary) so Pass 4-aligned routing and Anthropic-style prompt cache hints work from the same data structure. |

GCM is **not** a broad retrieval framework, **not** a domain-specific prompt assembler, **not** an agent memory authoring UX, **not** a planner, and **not** an in-process summarizer. Frameworks own retrieval; runtimes own prompt logic; the Memory Store is a typed KV backend; the [Compaction Lineage Service](#compaction-lineage-service) owns the lineage contract.

### Context Manifest Registry

Durable storage for every produced `ContextManifest`, queryable by `run_id`, `step_id`, `model_call_id`, and `checkpoint_id`. Every [`ModelCall`](/docs/concepts/model-calls) carries a `context_manifest_id` foreign key so the audit trail links every model call to the exact manifest it consumed.

### Context Artifact Store

Object-storage-backed externalized context. [`ContextArtifact`](/docs/concepts/context-artifacts) shape: `name + version + mime_type + bytes` (Google ADK convergence). Used for raw tool outputs, file diffs, workspace snapshots, large transcripts, retrieved documents, compaction inputs, and pre/post-compaction summaries. Artifact bytes live in MinIO (managed) or customer-supplied S3-compatible storage (private). Manifests reference artifacts via `artifact_ref_id` on the segment.

### Context Admission Service

Pre-dispatch admission against [`ContextBudget`](/docs/concepts/context-budgets). Decisions and their semantics:

| Decision | Meaning |
|---|---|
| `ALLOW` | Manifest fits the budget; dispatch proceeds. |
| `WARN` | Approaching limits; informational only — never blocks. |
| `OFFLOAD_REQUIRED` | Transient; the runtime offloads to artifacts and the manifest is re-admitted. |
| `COMPACT_REQUIRED` | Transient; the runtime compacts and the manifest is re-admitted. |
| `DEGRADE` | Transient; `max_tokens` reduced or optional segments dropped, then re-admitted. |
| `HARD_STOP` | Terminal; the request is rejected before GPU dispatch. |

The decision hits the wire as a [`ContextBudgetDecision`](/docs/concepts/context-budget-state-machine) and as the FSM action being taken. Admission runs against the *assembled* manifest before the run-level [`Budget`](/docs/concepts/budgets) precheck, so cost decisions are made on real token counts, not raw client message estimates.

### Compaction Lineage Service

[`CompactionRecord`](/docs/concepts/compaction-records) storage. Records source manifest, source segment IDs, output segment IDs, method, model used, token counts before and after, quality flags. The runtime may produce summaries and report records, or KovaServe may run a managed compactor — the lineage contract is the same in both cases. The contract is what makes resume reconstructible: a Checkpoint can rebuild logical context from manifest + artifacts + compaction lineage even when KV restore is unavailable.

### Memory Store Service

A typed namespace + key store with [`MemoryScope`](/docs/concepts/memory-scopes) retention. [`MemoryEntry`](/docs/concepts/memory-entries) row shape: `(tenant_id, project_id, scope_type, scope_id, namespace, key, value, retention, ttl, embedding?)`. LangGraph `Store`-compatible; backs Anthropic memory tool's caller-provided persistence contract. Scopes:

| Scope | Lifetime |
|---|---|
| `ephemeral` | Invocation-scoped, dropped after the call. |
| `session` | Session lifetime. |
| `user` | Cross-session per user. |
| `tenant` | Cross-user per tenant. |
| `global` | Cross-tenant within deployment, admin-controlled. |

### Tool Catalog Registry

[`ToolCatalog`](/docs/concepts/tool-catalogs) is a hash-identified, immutable, versioned bundle of tool definitions. Referenced by `tool_catalog_id` so prompts that share a catalog stay on the same cache breakpoint tier. Registration computes the hash and returns the id; the bundle is never mutated in place — a new version is always a new id.

### Tool Context Governance

Evaluates whether tool definitions and tool outputs may enter context, and how. Output is a [`ToolContextDisposition`](/docs/concepts/tool-context-disposition):

| Disposition | Meaning |
|---|---|
| `inline_full` | Full payload enters context inline. |
| `inline_preview` | A bounded preview enters context; the full payload is artifact-backed. |
| `summary_required` | A required summary is the inline form; full payload is artifact-only. |
| `artifact_ref` | Only an artifact reference enters context; no inline form. |
| `drop` | The output is recorded but never enters context. |
| `blocked` | The tool call itself is denied. |

Decisions carry `risk_level`, `checkpoint_recommended`, `max_inline_tokens`, `artifact_required`, and `redaction_policy`. They are deterministic and auditable.

### KV Event System

Ingests and normalizes the `kv_events` stream from [Layer 1](/docs/concepts/execution-substrate). Responsibilities: durable ingestion, event enrichment and validation, replayable streams, and fan-out to KV Index, Metering, Dashboard, and audit consumers. The system is event-driven on real substrate truth — never synthetic polling approximations. See [state events](/docs/concepts/events/state).

### KV Index Service

Maintains near-real-time authoritative lookup of reusable state. Maps stable identities to nodes and tiers; tracks freshness, completeness, compatibility, and residency; answers lookup queries for routing and resume planning; resolves activation [`StateHandle`](/docs/concepts/state-handles)s. See [KV Index](/docs/concepts/kv-index).

### Cache-Aware Router (manifest-aware)

Chooses execution placement using continuity, context reuse value, and economics — not blind load-balancing or raw prefix hash. Score inputs:

| Input | Source |
|---|---|
| Cache hit potential | KV Index lookup on `stable_prefix_id`. |
| Manifest reuse estimate | `stable_context_id` match across breakpoint tiers. |
| Expected prefill savings tokens | Manifest token accounting. |
| Rehydrate / restore cost | KV state restore latency + artifact reload bytes. |
| Queue cost | Substrate node telemetry. |
| Isolation penalty | Tenancy and PolicyScope scope filter. |
| Model availability | Substrate model lifecycle signals. |
| Node health | Substrate telemetry. |
| Placement hints | Runtime-supplied `PlacementHint`. |
| Resume priority | Whether the request is a resume from `Checkpoint`. |
| Policy constraints | `policy_scope.context_share_scope`, `data_boundary`. |
| Run-level Budget pressure | Metering layer. |
| `ContextBudget` admission state | Degraded runs prefer cache reuse more aggressively. |
| Session affinity | Session Registry, when bound. |

Outputs: selected route, fallback route(s), explanation rationale (`route.rationale`), warm/partial/cold classification, `resume_feasible` flag. See [manifest-aware routing](/docs/concepts/manifest-aware-routing).

### Session Registry

Persists continuity over time. Manages [`Session`](/docs/concepts/sessions)s and their active state handles, tracks the Session's `current_manifest_id` and bounded manifest history, tracks `CompactionRecord`s associated with the Session, tracks `ContextArtifact` references retained at Session scope, treats summaries as first-class typed segments, binds Sessions to Runs and Checkpoints (and parent Runs for sub-run lineage), and supports Session survival across node restarts.

### Checkpoint Manager

Makes resumability first-class. **Logical context is the primary capability; KV cache export/import is an optimization layered on top.** A checkpoint must remain useful even when KV restore is unavailable. Responsibilities:

| Responsibility | Detail |
|---|---|
| Create checkpoint manifests | Bound to a frozen `ContextManifest` snapshot. |
| Bind references | Run / Step / Session / state / artifact / compaction refs. |
| Validate compatibility | Model family, tokenizer fingerprint, renderer version, scope. |
| Plan restore | Logical context manifest first, KV state if available. |
| Track outcomes | Restore attempts, latencies, failure reasons. |
| Expose StateHandles | Typed `kv_prefix | context_manifest | checkpoint | artifact_bundle | memory_entry | runtime_state` references. |

Resume order is logical-first: load manifest → validate manifest and artifacts → validate model/tokenizer compatibility → plan KV restore if available → reconstruct active context from manifest+artifacts if KV restore is unavailable → route based on feasibility → transition through `RECOVERING → RESUMING → RUNNING`. See [resume flow](/docs/concepts/resume-flow).

### State Handle Registry

Resolves typed [`StateHandle`](/docs/concepts/state-handles)s. Discriminator values: `kv_prefix`, `context_manifest`, `checkpoint`, `artifact_bundle`, `memory_entry`, `runtime_state`. Generalizes the original "activation handle" concept so cross-session reuse extends beyond raw KV prefixes to manifests, artifacts, memory entries, and runtime state.

### Tiering / State Manager

Preserves continuity under pressure. Observes tier occupancy and pressure, plans demotion / promotion / archive / prefetch, instructs Layer 1 actions, preserves state identity across movement, exposes transfer and restore activity to observability and metering. The Tiering Manager is a **planner/controller** in Layer 2; raw movement execution belongs to the Layer 1 Transfer/Restore Worker adjacency.

### Policy / Governance Layer

| Responsibility | Detail |
|---|---|
| Model ACLs | Which models a [`PolicyScope`](/docs/concepts/policy-scopes) may dispatch against. |
| Rate limits | Per Organization / Project / Environment. Most-specific applicable limit wins. |
| Share scope | `policy_scope.context_share_scope` — controls cache and manifest reuse boundaries (`tenant`, `project`, `environment`, `key`). |
| Tool policy | Integrated with `ToolContextDisposition` outputs. |
| Data boundary | Artifact and context residency constraints. |
| Environment / deployment restrictions | Region pinning, sovereign overlays. |
| Decisions | Explainable allow / deny / degrade — fast, deterministic, auditable. |

### Metering / Cost / Budget Layer

The economic system of record. Active control objects, not just billing records.

| Responsibility | Detail |
|---|---|
| Ingest raw usage | With OpenAI-convergent `cached_tokens`. |
| Aggregate | By tenant / project / model / day. |
| Attach | To [`Budget`](/docs/concepts/budgets)s and [`ContextBudget`](/docs/concepts/context-budgets)s. |
| Forecast | Exhaustion windows. |
| Emit | Warnings and hard stops as runtime-visible decisions. |
| Track savings | Cache reuse value, manifest reuse value, compaction savings. |
| Track context economics | Active tokens, stable tokens, volatile tokens, offloaded tokens, artifact storage bytes. |
| Roll up | `cost_per_completed_run` per Run, per Project, per Organization. |
| Snapshot pricing | `input_price_snapshot`, `output_price_snapshot`, `discount_snapshot` per usage row, so audit is row-self-contained. |
| Preserve identity | `gross = net + savings` exactly across rows and scopes. |

`Budget`s govern run-level cost / tokens / steps / time / tools. `ContextBudget`s govern admission control on context size. They coexist; they evaluate at different points.

### Tenant Management

Owns the [tenancy](/docs/concepts/tenancy) hierarchy: [`Organization`](/docs/concepts/organizations) → [`Project`](/docs/concepts/projects) → [`Environment`](/docs/concepts/environments) → [`Key`](/docs/concepts/keys). Resolves scope on every request; manages role bindings; partitions partner-safe deployments; supports both managed multitenancy and private single-organization topologies.

### Runtime Integration Adapters

Mandatory for [BYO runtime mode](/docs/concepts/byo-runtime). Ingest metadata from external runtimes, map external execution IDs into canonical entities, preserve checkpoint / budget / placement hints, and normalize semantics without taking over orchestration. Adapters exist for LangGraph, Temporal, and a generic runtime envelope; the wire shape is uniform across them.

### Dashboard / Admin Backend

Surfaces every control-plane truth that operators and developers need:

| View | Purpose |
|---|---|
| Run explorer | Per-Run timeline, Step list, ModelCalls, Checkpoints, branches, sub-runs. |
| Session explorer | Manifest history, current_manifest_id, retained artifacts. |
| Checkpoint browser | Tree view, restore feasibility, attached StateHandles. |
| State / routing view | Where reusable state lives, routing decisions, fallbacks. |
| Usage and budget dashboards | Per-Run, Project, Organization roll-up. |
| Context manifest timeline | Per Run — segments, sizes, tier classification. |
| Compaction lineage visualization | Source-to-output relationships. |
| Artifact offload visibility | Bytes, types, retention. |
| Context budget burn | Per Run — admission decisions over time. |
| Tool catalog inventory | Versions, invalidation events. |
| Policy decision inspection | Per call — what was allowed, denied, degraded, why. |
| Fleet / node health | From substrate telemetry. |
| Tenant / Project admin | Hierarchy management, key issuance. |

### SDK / CLI support APIs

The control-plane endpoints that back the [Python SDK](/docs/sdk) and the [`kova` CLI](/docs/cli). Wire shapes for Run, Session, Checkpoint, Budget, Context, Memory, Tool catalog, Events.

## The hot path

The control plane orders the standard model call so context-dependent decisions run after context is assembled — never against rough message estimates.

```
1.  Authn / scope resolution           ── Gateway: resolve tenant / project / PolicyScope
2.  Run / Session resolution            ── Canonical Execution Service (no Step yet)
3.  Context inputs gathering            ── Session history + Memory Store + ContextArtifact previews
4.  Manifest assembly (GCM v2)          ── ContextSegments, stable_prefix_id, stable_context_id, token counts
5.  Tool context disposition            ── ToolContextPolicy → ToolContextDisposition
6.  Context admission                   ── ContextBudget decision; transformations applied; manifest updated
7.  Run-budget precheck                 ── Budget evaluates against admitted manifest
8.  Step creation                       ── Step row created and bound to manifest
9.  Routing                             ── Cache-Aware Router with KV Index + scope filters + budget pressure
10. Substrate execution                 ── Layer 1 executes; emits real lifecycle / state events
11. Recording                           ── ModelCall stored with context_manifest_id FK; metering, ContextBudget
                                            consumption, ExecutionEvents, Session updates, dashboards update
```

## What the control plane does not own

The boundary is precise.

- **Not the substrate.** Model execution mechanics, KV cache, scheduler — all Layer 1.
- **Not the runtime.** Worker lease ownership, sandbox lifecycle, workspace and `WorkspaceArtifact` model, runtime-local state, retry/resume/branch UX — all [Layer 3](/docs/concepts/runtime-wedge).
- **Not prompt composition.** Frameworks own retrieval; runtimes own prompt logic.
- **Not a planner.** Runtimes own planning.
- **Not an agent memory authoring UX.** The Memory Store is a typed KV backend.
- **Not a context engineering platform.** The control plane ships execution-boundary primitives — manifests, segments, artifacts, budgets, compaction lineage, memory scope, tool catalog versioning, sub-run summary handoff, tool context disposition — and not a prompt-composition studio.
- **Not bespoke per customer.** The same semantic model serves every Organization on managed cloud, every customer on private deployment, and every partner on a partner fleet.

## Deployment

The control plane is one Helm chart. On [managed cloud](/docs/concepts/managed-cloud) it runs in KovaServe's hosted Kubernetes cluster against hosted Postgres / Redis / NATS / MinIO. On [private deployment](/docs/concepts/private-deployment) it runs in the customer's k3s or conformant Kubernetes cluster against customer-supplied Postgres / Redis / NATS / S3-compatible storage. The reference 3-VM split is one control node plus two GPU nodes; the chart parameterizes the layout.

Subsystem boundaries map cleanly to deployment: every Layer 2 subsystem is a module in the FastAPI monolith, every Layer 1 component is a per-node agent or worker, and every cross-process call goes through gRPC, NATS, or ZMQ — never an in-process event bus.
