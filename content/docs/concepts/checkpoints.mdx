A **Checkpoint** is a durable, validated marker on a [Run](/docs/concepts/runs) that captures enough state to resume execution from that point on a different node, after a crash, or after an explicit pause. KovaServe's checkpoint design has one defining property: **logical context is the primary capability; KV cache export and import is an optimization layered on top.** A checkpoint must remain useful even when KV restore is unavailable.

The Checkpoint is the entity to reach for when you need:

- Recovery from a node failure mid-Run without restarting from zero.
- A safe point taken before a risky Step (a destructive tool call, an irreversible write).
- A handoff point so the next Run in a [Session](/docs/concepts/sessions) can pick up where the previous one left off, on a different node and possibly a different replica.
- Auditable restore lineage — which manifest, which artifacts, which compaction history produced a given resumed state.

Checkpoints are managed by the **Checkpoint Manager**. They bind to a frozen [`ContextManifest`](/docs/concepts/context-manifests) snapshot and to a typed family of [StateHandle](/docs/concepts/state-handles)s.

## Logical-context-first restore

KovaServe departs from KV-first checkpoint systems by treating **logical context as the source of truth** for what a Run was thinking about, and treating GPU KV cache state as a performance accelerator that may or may not be available at restore time.

A Checkpoint is anchored to a frozen [`ContextManifest`](/docs/concepts/context-manifests) — the full segment list, token counts, artifact references, and compaction lineage that produced the prompt at the moment of capture. From the manifest plus the referenced [ContextArtifacts](/docs/concepts/context-artifacts), the active context can always be reconstructed, even when:

- the original GPU node is gone,
- the KV export is missing or corrupt,
- the model has been redeployed with a different tokenizer fingerprint,
- the renderer version has rolled forward.

KV cache restore is attempted when compatible and skipped silently when not. A resumed Run is **functionally indistinguishable** whether it came back via KV restore or via manifest replay — only the latency profile differs. Full restore order is documented on the [resume flow](/docs/concepts/resume-flow) page.

## Relationships

```
Run                                  (1..N checkpoints)
└── Checkpoint
    ├── context_manifest_id          (frozen ContextManifest snapshot)
    ├── state_handles[]              (typed; see state-handles)
    ├── compatibility_vector         (model family, tokenizer fp,
    │                                 renderer version, scope)
    ├── storage_refs[]               (S3 / MinIO URIs for artifacts and
    │                                 KV exports)
    └── run_ref / step_ref / session_ref / parent_run_ref
```

A Checkpoint is bound to exactly one Run and, transitively, one Session. Checkpoints are not shared across Runs; a resumed Run is a new Run that *references* the Checkpoint of the prior Run as its restore source.

## What a Checkpoint binds

| Field | Purpose |
|---|---|
| `run_id` / `step_id` / `session_id` | Canonical execution references. |
| `parent_run_id` | The originating Run when the Checkpoint was created; preserved across resume. |
| `context_manifest_id` | The frozen manifest snapshot. The audit anchor for restore. |
| `state_handles[]` | Typed references to reusable execution state. See [StateHandles](/docs/concepts/state-handles). |
| `compatibility_vector` | Model family, tokenizer fingerprint, renderer version, policy scope. Validated before restore. |
| `storage_refs[]` | Object storage URIs for artifacts and any KV exports. |

A single Checkpoint typically holds a handful of StateHandles: one `context_manifest`, optionally one `kv_prefix`, zero or more `artifact_bundle`s, and — for runtime-aware checkpoints — one `runtime_state`. The Checkpoint Manager validates each StateHandle's compatibility independently.

## Lifecycle

A Checkpoint travels `REQUESTED → CAPTURING → STORED → VALIDATED → RESTORING → RESTORED`, with terminal failure states for capture, validation, and restore-attempt failure. The full FSM, ref-counted concurrent-restore semantics, and validated transitions live on the [Checkpoint state machine](/docs/concepts/checkpoint-state-machine) page.

The associated Run-side transitions — `CHECKPOINTING` on capture, `RECOVERING → RESUMING → RUNNING` on restore — are documented on the [Run state machine](/docs/concepts/run-state-machine) and the [resume flow](/docs/concepts/resume-flow).

## Compatibility vector

Each Checkpoint records a compatibility vector validated on every restore:

| Component | Source |
|---|---|
| `model_id` | The serving model at capture time. |
| `model_family` | Coarse compatibility class (e.g. `qwen3-coder`). |
| `tokenizer_fingerprint` | Hash of the tokenizer assets. Mismatches force manifest replay. |
| `renderer_version` | The context renderer version. Forward-compatible renderers are explicitly allow-listed. |
| `policy_scope` | The [PolicyScope](/docs/concepts/policy-scopes) under which the Checkpoint was created. The resuming Run must be under a compatible scope. |

A mismatch on `model_family` or `policy_scope` rejects restore outright. Mismatches on `tokenizer_fingerprint` or `renderer_version` cause KV restore to be skipped while manifest replay proceeds. See [renderer versions](/docs/concepts/renderer-versions) for the detail of how each field is computed and validated.

## ID format

Checkpoint IDs are bare 26-character Crockford ULIDs, with no prefix:

```
01J8X9Y2Z3K4M5N6P7Q8R9S0T7
```

This matches the convention for `run_id`, `step_id`, `model_call_id`, and `context_manifest_id`. Only `session_id` deviates (`ses_<24hex>`); see [Identity and IDs](/docs/concepts/identity-and-ids).

## Wire surface — extension envelope

Checkpoint references travel on the inference path through the `kovaserve` extension envelope:

```json
{
  "kovaserve": {
    "checkpoint_id": "01J8X9Y2Z3K4M5N6P7Q8R9S0T7",
    "resume_marker": "step:7"
  }
}
```

| Field | Purpose |
|---|---|
| `checkpoint_id` | Identifies the Checkpoint to restore from at request time. |
| `resume_marker` | Optional opaque marker the runtime can read to position itself within the restored context. |

Both fields flow end-to-end: accepted on `POST /v1/chat/completions`, persisted onto the originating ModelCall, propagated through the KV event stream, and surfaced on every trace and execution event tagged to the resumed Run.

### Resume from a Checkpoint

```bash
curl https://api.kovaserve.com/v1/chat/completions \
  -H "Authorization: Bearer $KOVASERVE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "qwen3-coder-30b",
    "messages": [{"role": "user", "content": "..."}],
    "kovaserve": {
      "checkpoint_id": "01J8X9Y2Z3K4M5N6P7Q8R9S0T7",
      "resume_marker": "step:7"
    }
  }'
```

The gateway resolves the Checkpoint, runs the restore plan, and binds the new Run to the same Session as the original — the resumed Run is the next Run in the Session's run history.

## What Checkpoints are not

- **Not a substitute for [Sessions](/docs/concepts/sessions).** Sessions provide multi-run continuity at the conversational level; Checkpoints provide point-in-time durability within or across Runs. A Session may host many Checkpoints; a Checkpoint is always anchored to a Run.
- **Not a memory store.** Cross-Run, cross-Session reuse of *information* belongs to the [Memory Store](/docs/concepts/memory-store) via `memory_entry` StateHandles, not raw checkpoint replay.
- **Not a transaction log.** Checkpoints are explicit safe-points, not a continuous record of every Step.
- **Not coupled to a node.** A Checkpoint is portable across replicas in the same deployment by design. KV restore is opportunistic; manifest replay is universal.
