A **Session** is a continuity container that lives above the [Run](/docs/concepts/run-hierarchy/runs). One Session hosts many sequential Runs and accumulates a turn history that persists across them. Sessions are how KovaServe represents long-lived conversational or task context without forcing the runtime to manage that state itself.

The Session is the entity to reach for when you need:

- A stable identity that survives across multiple Runs, retries, or pauses.
- A turn history that is queryable and audit-traceable, with foreign keys back to the Steps and ModelCalls that produced each turn.
- A scoped lifetime (idle reactivation, TTL expiry, explicit archival) without the runtime owning lifecycle logic.

Sessions are persisted by the **Session Registry**. They are tenant- and project-scoped; cross-tenant access is treated as not-found.

## Relationships

```
Session                 (1..N runs, 1..N turns)
├── Run                 (current_run_id points to the active one)
│   └── Step
│       └── ModelCall
└── SessionTurn         (one row per message: user, assistant, tool)
    ├── step_id         (FK to the Step that produced the turn)
    └── model_call_id   (FK to the ModelCall that produced the turn)
```

A Run has at most one Session (`runs.session_id`). That binding is set at Run creation and is **immutable** — a Run cannot be moved between Sessions.

A Session has at most one *current* Run (`sessions.current_run_id`). Earlier Runs in the Session are not erased; they remain in the run history but no longer occupy the `current_run_id` pointer.

## Lifecycle

Sessions travel a five-state FSM — `OPEN → IDLE → RESUMABLE → ARCHIVED | EXPIRED`. The full graph, drivers, and validated transitions are documented on the [Session state machine](/docs/concepts/continuity/session-state-machine) page.

Quick orientation:

| State | Meaning |
|---|---|
| `OPEN` | Active. Accepts new turns and Run bindings. The default on creation. |
| `IDLE` | No turn for longer than the idle threshold. Reactivates to `OPEN` automatically on the next turn. |
| `RESUMABLE` | Held pending a checkpoint-driven resume. Exits to `OPEN` when a new Run resumes against that [Checkpoint](/docs/concepts/recovery/checkpoints). |
| `ARCHIVED` | Explicitly closed (terminal). Readable for audit; rejects new turns. |
| `EXPIRED` | TTL elapsed (terminal). Reads return `410 Gone`; the row remains for audit. |

## Session-Run binding

`POST /v1/ks/sessions` creates an empty Session. `current_run_id` is `NULL` until a Run is bound.

A Run becomes the current Run via the `kovaserve.session_id` extension on `POST /v1/chat/completions`, or by passing `session_id` to `POST /v1/ks/runs`. Internally this calls a `bind_run` operation that enforces six invariants:

1. **Tenant scoping.** The target Run must belong to the same tenant as the Session. A cross-tenant bind returns the same `404` as a missing Session — the existence of the foreign Run is not leaked.
2. **Run ID shape.** The Run ID must be a valid 26-character Crockford ULID.
3. **Non-terminal Runs only.** Runs in `COMPLETED`, `FAILED`, or `CANCELED` cannot be bound.
4. **No cross-Session re-binding.** A Run already bound to a different Session cannot be moved.
5. **Idempotent same-pair binds.** Re-binding the Run that is already current is a no-op, not an error.
6. **`last_accessed_at` is bumped** as part of the bind, even when no new turn is recorded.

When a Run reaches a terminal state, `sessions.current_run_id` remains pinned to that Run (it is **not** cleared). The next bind replaces it.

### Multiple sequential Runs

The expected pattern is one Session, many sequential Runs. Each Run completes (or fails / is canceled), and the next Run binds in. The turn history spans every Run; turn rows carry `step_id` and `model_call_id` so each turn is traceable to the exact execution that produced it.

### Concurrent Runs

The model permits a second Run to be bound to a Session whose current Run is still active. The most recently bound Run becomes `current_run_id`; the previous Run continues independently and is still queryable via the run history. The runtime, not the Session Registry, is responsible for ordering concurrent Runs against the same Session if ordering matters.

## Turn history

Each call to a Session appends one or more `SessionTurn` rows:

```json
{
  "session_id": "ses_a7f3c2b8d4e1f0925a1b6c3d",
  "turn_number": 5,
  "role": "assistant",
  "content": "...",
  "token_count": 142,
  "timestamp": "2026-04-30T12:00:18Z",
  "step_id": "01J8X9Y2Z3K4M5N6P7Q8R9S0T5",
  "model_call_id": "01J8X9Y2Z3K4M5N6P7Q8R9S0T6"
}
```

`turn_number` is monotonic per Session. `token_count` is a real measurement from the substrate response, not a client-side estimate. `step_id` and `model_call_id` are populated for any turn produced by a Run-bound execution.

## Manifest tracking

Each Session tracks the most recently admitted [ContextManifest](/docs/concepts/context/context-manifests) via `current_manifest_id`, plus a bounded history of prior manifests. The pointer survives across Run binds — a new Run inheriting a Session inherits the current manifest reference. Full semantics (when the pointer advances, what survives across Run binds and Checkpoint restore, how the bounded history is maintained) live on the [current manifest](/docs/concepts/continuity/current-manifest) page.

## Tenant scoping

Every Session is owned by exactly one `(tenant_id, project_id)` pair, inherited from the API key that created it. The owning pair is immutable. Reads, writes, and binds from any other tenant return `404 Not Found` — the same response as a genuinely missing Session — so cross-tenant probing yields no information.

## ID format

Session IDs do not follow the bare-ULID convention used elsewhere in the surface. A Session ID is the literal prefix `ses_` followed by 24 lowercase hex characters:

```
ses_a7f3c2b8d4e1f0925a1b6c3d
```

Session IDs use this prefixed form so they are distinguishable at a glance from execution-scoped IDs. All other IDs (`run_id`, `step_id`, `model_call_id`, `checkpoint_id`, `context_manifest_id`) are bare 26-character Crockford ULIDs. See [Identity and IDs](/docs/concepts/run-hierarchy/identity-and-ids).

## API shape

### Create a Session

```http
POST /v1/ks/sessions
Authorization: Bearer ks_...
Content-Type: application/json

{
  "model": "qwen3-coder-30b",
  "expires_in_hours": 24
}
```

```json
{
  "id": "ses_a7f3c2b8d4e1f0925a1b6c3d",
  "tenant_id": "01J8X9Y2Z3K4M5N6P7Q8R9S0T0",
  "project_id": "01J8X9Y2Z3K4M5N6P7Q8R9S0T1",
  "model": "qwen3-coder-30b",
  "state": "OPEN",
  "turn_count": 0,
  "created_at": "2026-04-30T12:00:00Z",
  "last_accessed_at": "2026-04-30T12:00:00Z",
  "expires_at": "2026-05-01T12:00:00Z",
  "current_run_id": null,
  "current_manifest_id": null,
  "stable_prefix_id": null,
  "last_node_id": null
}
```

### Use a Session on inference

The most common path: pass `kovaserve.session_id` on a chat completion. The gateway resolves the Session, prepends its turn history into the context, executes, and appends the new user and assistant turns on success.

```bash
curl https://api.kovaserve.com/v1/chat/completions \
  -H "Authorization: Bearer $KOVASERVE_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "qwen3-coder-30b",
    "messages": [{"role": "user", "content": "..."}],
    "kovaserve": {
      "session_id": "ses_a7f3c2b8d4e1f0925a1b6c3d"
    }
  }'
```

If the Session is `EXPIRED` or `ARCHIVED`, the request is rejected before inference. If it is `IDLE`, it reactivates to `OPEN` atomically with the new turn.

## What Sessions are not

- **Not the unit of value.** That is the [Run](/docs/concepts/run-hierarchy/runs). A Session can host zero, one, or many Runs over its lifetime.
- **Not a runtime construct.** The Session Registry is owned by the control plane; runtimes (LangGraph, Temporal, custom) reference Sessions but do not implement them.
- **Not a memory store.** Cross-Session memory belongs to the [Memory Store](/docs/concepts/memory/memory-store) with explicit [`MemoryScope`](/docs/concepts/memory/memory-scopes) (`user`, `tenant`, `global`). Sessions hold per-conversation continuity only.
- **Not the canonical owner of every ContextManifest.** Each ModelCall carries its own `context_manifest_id`; the Session tracks `current_manifest_id` and a bounded history for resume planning, not the full per-call lineage.
