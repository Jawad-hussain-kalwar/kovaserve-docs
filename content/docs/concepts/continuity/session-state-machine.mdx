title: Session state machine
description: The Session FSM — OPEN, IDLE, RESUMABLE, ARCHIVED, EXPIRED — and how each transition is driven by turns, idle thresholds, expiry, archival, and checkpoint-driven resume.
---

The **Session state machine** is the lifecycle every [Session](/docs/concepts/continuity/sessions) travels from creation to termination. It has five states and three drivers: caller activity (turns), wall-clock thresholds (idle, expiry), and explicit transitions (archival, checkpoint-driven holds).

The FSM is enforced by the **Session Registry**. Illegal transitions are rejected at the service layer; terminal states (`ARCHIVED`, `EXPIRED`) are frozen.

## The graph

```
OPEN ─┬─▶ IDLE ─┬─▶ OPEN              (reactivation on new turn)
      │         ├─▶ RESUMABLE         (held for checkpoint-driven resume)
      │         ├─▶ ARCHIVED          (terminal)
      │         └─▶ EXPIRED           (terminal)
      ├─▶ RESUMABLE                   (held for checkpoint-driven resume)
      ├─▶ ARCHIVED                    (terminal)
      └─▶ EXPIRED                     (terminal)

RESUMABLE ─┬─▶ OPEN                   (resume Run binds against the held session)
           ├─▶ ARCHIVED               (terminal)
           └─▶ EXPIRED                (terminal)
```

`ARCHIVED` and `EXPIRED` are durable and frozen. Every other state can transition to either of them — `EXPIRED` automatically when `expires_at` elapses, `ARCHIVED` only by explicit administrative or runtime action.

## States

### `OPEN`

The Session is active. It accepts new turns, accepts new Run bindings, and reactivates from `IDLE` automatically on the next turn. This is the default on creation and the state most reads will encounter.

`current_run_id` may be set (a Run is bound) or `null` (no Run is currently bound, but the Session is ready to accept one). `last_accessed_at` advances on every turn append and on every `bind_run` call.

**Transitions out:** `IDLE` (idle threshold elapsed with no bound non-terminal Run), `RESUMABLE` (checkpoint-driven hold while a Run is bound), `ARCHIVED` (explicit close), `EXPIRED` (TTL elapsed).

### `IDLE`

The Session has been quiet — no turn appended for longer than the configured idle threshold — and no non-terminal Run is currently bound. The Session is alive and reactivatable; no work is required to bring it back, just a turn.

`IDLE` is purely a state-of-attention marker. It does not affect read access, does not gate Run lookups, and does not free resources. Its purpose is to give dashboards and routing logic a cheap signal that this Session is not actively driving work.

**Transitions out:** `OPEN` (next turn appended — atomic with the turn write), `RESUMABLE` (checkpoint-driven hold takes effect), `ARCHIVED`, `EXPIRED`.

### `RESUMABLE`

The Session is held pending a checkpoint-driven resume. Entered when the bound Run reaches a [Checkpoint](/docs/concepts/recovery/checkpoints) and the runtime suspends with intent to continue later in a new Run; exits when a new Run resumes against that Checkpoint and binds to this Session.

`RESUMABLE` is distinct from `IDLE` in that it carries an explicit "we are holding this for a known resume" semantics. Operators can identify and triage held Sessions; idle-sweep policies can leave `RESUMABLE` Sessions alone even when their idle window has elapsed.

A Session in `RESUMABLE` does **not** accept new arbitrary turns. Reads succeed; the only legal mutation is the resume bind, which transitions back to `OPEN` and begins the new Run's turns.

**Transitions out:** `OPEN` (resume Run binds in), `ARCHIVED`, `EXPIRED`.

### `ARCHIVED` (terminal)

The Session has been explicitly closed. Set by administrative or runtime action — never by a sweep — to indicate that the Session is closed for new work but should remain readable for audit. Reads return `200 OK` with the Session row and turn history; writes (turn appends, run binds) return `409 Conflict` with `error.code = "session_archived"`.

`ARCHIVED` is the right terminal status when a workflow has finished and you want the audit trail preserved without ambiguity about whether more work is coming.

### `EXPIRED` (terminal)

The Session's `expires_at` has elapsed. Reached either by a lazy check on read (`GET /v1/ks/sessions/{id}` transitions any non-terminal Session past its expiry before returning) or by the background sweep that expires Sessions in bulk.

After expiry:

- Reads return `410 Gone` with `error.code = "session_expired"`. The Session row and all turns remain in the database for audit; expiry does not purge.
- Writes (turn appends, run binds) return `410 Gone`.
- The Session cannot be reactivated. To resume the same conversational thread, create a new Session.

## Drivers and transitions

### Turn append

Appending a turn (the most common operation, executed on `/v1/chat/completions` with a `kovaserve.session_id`) drives two transitions:

- `OPEN → OPEN` — the normal case. `last_accessed_at` advances; `turn_count` increments.
- `IDLE → OPEN` — reactivation. The transition is performed atomically inside the same database transaction that records the turn, so there is no observable window where the new turn is recorded against an `IDLE` row.

A turn append against `RESUMABLE`, `ARCHIVED`, or `EXPIRED` is rejected before substrate dispatch.

### Idle threshold

The Session enters `IDLE` when `last_accessed_at` falls strictly behind the configured idle threshold and no non-terminal Run is bound. The check is strict (`<`): a Session exactly at the threshold remains `OPEN`, which avoids spurious sweeps under clock skew.

The idle threshold is a Project-level configuration with a default. Sessions with a bound non-terminal Run are never considered `IDLE` — the bound Run is itself the activity signal.

### Bind a Run

`bind_run` (called via the `kovaserve.session_id` extension on inference, or via `POST /v1/ks/runs` with `session_id`) advances `last_accessed_at` even when no turn is appended. A `bind_run` against an `IDLE` Session reactivates it to `OPEN` atomically; a bind against `RESUMABLE` (the resume case) transitions to `OPEN`; a bind against `ARCHIVED` or `EXPIRED` is rejected.

Six invariants on bind — Tenant scoping, ULID shape, non-terminal Run only, no cross-Session re-binding, idempotent same-pair binds, `last_accessed_at` bump — are detailed on [Sessions](/docs/concepts/continuity/sessions).

### Checkpoint-driven hold

The runtime sets the Session to `RESUMABLE` when the bound Run reaches a [Checkpoint](/docs/concepts/recovery/checkpoints) with intent to suspend rather than continue. The transition is `OPEN → RESUMABLE` or `IDLE → RESUMABLE`. The Run itself transitions to `PAUSED` on the [Run state machine](/docs/concepts/run-hierarchy/run-state-machine); the Session-level hold makes the resume intent observable from Session-shaped queries (e.g. "show me Sessions held for resume").

When a new Run binds in to resume the Checkpoint, the Session transitions `RESUMABLE → OPEN` and the new Run's first Step is admitted. The held Checkpoint remains addressable for further branches even after the Session reopens.

### Expiry

Each Session may carry an `expires_at`. The default is 24 hours from creation; pass `expires_in_hours` on `POST /v1/ks/sessions` to override.

Expiry is **lazy**: every read checks `expires_at` before returning. Sessions caught past their expiry transition to `EXPIRED` as part of the read. A background sweep also expires Sessions in bulk so the FSM does not lag for Sessions that are never read.

### Archival

`ARCHIVED` is set by explicit administrative or runtime action — typically a `PATCH` that closes the Session, or a runtime that finishes a workflow and wants the audit trail frozen. Archival is allowed from `OPEN`, `IDLE`, and `RESUMABLE`. It is **not** triggered by automatic sweeps.

## Terminal-state guarantees

`ARCHIVED` and `EXPIRED` are durable and frozen. Once reached:

- `state` cannot change. The Session Registry rejects any transition request.
- New turns, new Run binds, and Run transitions referencing this Session are rejected.
- Existing turn history and Run bindings remain queryable indefinitely for audit.

A Run bound to a Session that subsequently terminates — whether by archival or expiry — is **not** itself terminated. The Run continues on its own [state machine](/docs/concepts/run-hierarchy/run-state-machine); only its Session is frozen. The Run row keeps `session_id` pointing at the now-terminal Session.

## Why no `CLOSED`

`CLOSED` is intentionally not a state. The two terminal forms — `ARCHIVED` and `EXPIRED` — carry different semantics:

- `ARCHIVED` was a deliberate close. The audit story is "we ended this on purpose."
- `EXPIRED` was a wall-clock close. The audit story is "this aged out."

Collapsing them would lose information that operators and auditors care about. Reads of either return immediately, but the response codes differ — `200` for `ARCHIVED` (closed but available), `410 Gone` for `EXPIRED` (closed and aged out) — to make the distinction visible at the protocol level.

## Transitions are validated

Every transition is checked by the Session Registry before the row is updated:

| From → To | Allowed |
|---|---|
| `OPEN → IDLE` | yes (idle threshold elapsed, no bound Run) |
| `OPEN → RESUMABLE` | yes (checkpoint-driven hold) |
| `OPEN → ARCHIVED` | yes (explicit close) |
| `OPEN → EXPIRED` | yes (TTL elapsed) |
| `IDLE → OPEN` | yes (turn appended or Run bound) |
| `IDLE → RESUMABLE` | yes |
| `IDLE → ARCHIVED` | yes |
| `IDLE → EXPIRED` | yes |
| `RESUMABLE → OPEN` | yes (resume Run binds in) |
| `RESUMABLE → ARCHIVED` | yes |
| `RESUMABLE → EXPIRED` | yes |
| anything from `ARCHIVED` or `EXPIRED` | rejected |
| `OPEN → IDLE` while a non-terminal Run is bound | rejected |
| direct `OPEN → RESUMABLE` without checkpoint context | rejected |

Illegal transitions return `409 Conflict` with `error.code = "session_transition_invalid"`.
