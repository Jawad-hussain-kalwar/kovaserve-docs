KovaServe's **memory model** is the typed surface for cross-[Run](/docs/concepts/run-hierarchy/runs), cross-[Session](/docs/concepts/continuity/sessions) persistence. It is composed of three primitives plus one service:

- **[MemoryScope](/docs/concepts/memory/memory-scopes)** — the closed enum (`ephemeral` / `session` / `user` / `tenant` / `global`) that classifies retention, reuse, and policy treatment.
- **[MemoryEntry](/docs/concepts/memory/memory-entries)** — a row addressable by `(tenant, project, scope_type, scope_id, namespace, key)`, holding a JSON value, retention, optional TTL, and optional embedding.
- **[Memory Store Service](/docs/concepts/memory/memory-store)** — the Layer 2 service that hosts entries; LangGraph `Store`-compatible and the persistence backend for Anthropic's memory tool.
- **`memory_class_hints`** — the inference-envelope field through which runtimes signal which scopes and namespaces to inject into the next [ContextManifest](/docs/concepts/context/context-manifests).

The model is deliberately small. Anything that needs to persist beyond a single ModelCall and is structured (not opaque bytes) becomes a MemoryEntry under a chosen scope.

## What sits where

| Need | Surface |
|---|---|
| Per-conversation turn history | [Session](/docs/concepts/continuity/sessions) — `SessionTurn` rows |
| User preferences across Sessions | MemoryEntry, `scope_type: user` |
| Tenant-wide instructions and policies | MemoryEntry, `scope_type: tenant` |
| Deployment-baseline system prompts | MemoryEntry, `scope_type: global` (admin-controlled) |
| Bytes too large to inline (tool outputs, file diffs, snapshots) | [ContextArtifact](/docs/concepts/context/context-artifacts) |
| Compacted segment lineage | [CompactionRecord](/docs/concepts/context/compaction-records) |
| Reusable substrate state (KV prefixes, manifests) | [StateHandle](/docs/concepts/recovery/state-handles) |

Memory is the *structured* persistence layer. Bytes belong to artifacts; substrate state belongs to StateHandles; turn history belongs to Sessions.

## How memory enters context

The runtime never inlines memory content directly on the inference path. It signals selectors via `memory_class_hints` and lets GCM v2 fetch and order:

```
1. Runtime sends memory_class_hints[] on the kovaserve envelope
2. GCM v2 resolves each hint:
     selector → Memory Store query → matching entries
3. Each entry becomes a system_memory ContextSegment
4. Segments take canonical position (system breakpoint)
5. Manifest admission proceeds
```

The runtime supplies *intent*; the Memory Store supplies *content*; GCM owns *order*. This boundary is what makes memory consistent across runtimes (LangGraph `Store`, Anthropic memory tool, custom) — they all converge on the same row shape and the same injection path.

```json
{
  "kovaserve": {
    "memory_class_hints": [
      {
        "selector": { "namespace": "preferences" },
        "memory_scope": "user",
        "retention": "with_call",
        "checkpoint_relevant": true
      },
      {
        "selector": { "namespace": "policy", "key_prefix": "compliance/" },
        "memory_scope": "tenant",
        "retention": "with_call"
      }
    ]
  }
}
```

## Two integration paths, one row shape

The Memory Store is intentionally compatible with two prevailing third-party memory contracts so runtimes do not need bespoke shaping:

| Runtime contract | Memory Store mapping |
|---|---|
| LangGraph `Store` (`get` / `put` / `search` / `list`) | Direct mapping; same `(namespace, key, value)` shape. |
| Anthropic memory tool's caller-provided persistence (`create` / `update` / `view` / `delete`) | Tool actions pass through to the same row operations. |

A workload can switch between LangGraph and the Anthropic memory tool without re-shaping its memory data. The wire format is the same; only the entry point differs.

## Scope, retention, and policy as one decision

Picking a `memory_scope` picks a complete policy package. The five scopes are not interchangeable lifetime knobs — each carries distinct retention, reuse, and policy treatment:

| Scope | Lifetime | Cross-Session | Cross-User | Cross-Tenant | Writability |
|---|---|---|---|---|---|
| `ephemeral` | One ModelCall | no | no | no | runtime |
| `session` | Bound Session | no | no | no | runtime |
| `user` | Tenant retention | yes | no | no | runtime |
| `tenant` | Tenant retention | yes | yes | no | runtime |
| `global` | Tenant retention | yes | yes | yes | admin only |

The scope you pick is the entire policy. Trying to widen later requires a fresh write under the new scope; entries are not promoted in place.

## Visibility under PolicyScope

The resolved [PolicyScope](/docs/concepts/policy/policy-scopes)'s `context_share_scope` (`none` / `project` / `tenant`) bounds which scopes a Run can read:

| `context_share_scope` | Readable scopes |
|---|---|
| `none` | `ephemeral`, `session` |
| `project` | `ephemeral`, `session`, `user`, project-bounded `tenant` and `global` |
| `tenant` | All five scopes within the tenant boundary |

Foreign-scope entries return `not_found` rather than `403` — the existence of foreign-scope data is never leaked.

## Search

Optional `embedding` on each entry plus a vector backend (pgvector) enables semantic search over `(tenant, project, scope_type, scope_id, namespace)`-bounded subsets. The Memory Store remains usable without the vector backend; only the search endpoint requires it.

## What the memory model is not

- **Not Session turn history.** Turns live on `SessionTurn` rows, indexed by Session id and turn number.
- **Not opaque blob storage.** Use [ContextArtifacts](/docs/concepts/context/context-artifacts) for large bytes; reference them from entries when both are needed.
- **Not a runtime construct.** The Memory Store is a Layer 2 service; runtimes consume it.
- **Not a planner.** The runtime decides what to remember and when; the store accepts writes, never infers them.
