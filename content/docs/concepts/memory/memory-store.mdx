The **Memory Store Service** is the typed namespace-and-key store that backs cross-Run, cross-[Session](/docs/concepts/continuity/sessions) persistence. Every row is a [MemoryEntry](/docs/concepts/memory/memory-entries) rooted at a [MemoryScope](/docs/concepts/memory/memory-scopes). The service is LangGraph `Store`-compatible — runtimes that target the LangGraph store interface work against KovaServe without adapter code — and serves as the persistence backend for Anthropic's memory tool.

The store sits at Layer 2. Runtimes do not implement memory persistence themselves; they read and write through this service.

## Position in the architecture

```
Runtime (LangGraph / Temporal / custom / Layer 3)
   │
   │ writes / reads via SDK or HTTP
   ▼
Memory Store Service
   │
   ▼
PostgreSQL  (rows, indexes, retention sweeper)
   │
   ▼ optional
Vector backend (pgvector)  ◀── for semantic search
```

The store is a typed surface, not a generic key-value cache. Schema is enforced; values are JSON; scopes and retention are validated on write.

## What the store provides

| Capability | How it shows up |
|---|---|
| Scope-rooted persistence | Every entry has `scope_type` × `scope_id`. See [memory scopes](/docs/concepts/memory/memory-scopes). |
| Namespace + key addressing | Composite key `(tenant, project, scope_type, scope_id, namespace, key)`. |
| Last-write-wins versioning | New writes increment `version`; the current version is what reads return. |
| Retention and TTL | `retention` class plus optional explicit `ttl_seconds`. |
| Eventing | Every write/read/expire/delete emits a `ks.memory.entry.*` event. |
| Optional semantic search | `embedding` field plus `/v1/ks/memory/search`. |
| Scope-respecting reads | Reads filter by `(tenant, project, scope_type, scope_id)`; cross-tenant or out-of-scope reads return `not_found`. |
| Idempotent writes | Idempotency keys make duplicate writes no-ops. |

## What the store does not do

- **Not a vector database.** The optional `embedding` field plus pgvector backend supports search; the store is not a general-purpose vector engine. Embedding generation is the runtime's responsibility.
- **Not a cache.** Memory entries are durable. Caching transient computation belongs in a separate layer.
- **Not a planner.** The runtime decides what to remember and when. The store accepts writes; it does not infer them.
- **Not a UX.** There is no memory-authoring console. Operators inspect entries through standard observability surfaces.

## Integration paths

### LangGraph `Store` compatibility

LangGraph's `Store` interface (`get`, `put`, `search`, `list`) maps directly onto Memory Store endpoints. A LangGraph application configured to use the KovaServe store sees the same row shape — `(namespace, key, value)` — that LangGraph defines, with KovaServe scoping applied transparently from the request's resolved tenancy.

The mapping:

| LangGraph `Store` op | Memory Store endpoint |
|---|---|
| `Store.put(namespace, key, value)` | `PUT /v1/ks/memory/entries` |
| `Store.get(namespace, key)` | `GET /v1/ks/memory/entries?...` |
| `Store.search(namespace, query)` | `GET /v1/ks/memory/search?...` |
| `Store.list(namespace)` | `GET /v1/ks/memory/entries?namespace=...` |
| `Store.delete(namespace, key)` | `DELETE /v1/ks/memory/entries/{id}` |

### Anthropic memory tool backend

The Anthropic memory tool defines a caller-provided persistence contract — the LLM calls memory-tool actions, and the caller persists them. The Memory Store Service implements that contract directly. Runtimes wiring up the Anthropic memory tool to KovaServe pass through tool calls to memory store operations:

| Memory tool action | Memory Store operation |
|---|---|
| `create` | `PUT /v1/ks/memory/entries` (new key) |
| `update` | `PUT /v1/ks/memory/entries` (existing key, version increments) |
| `delete` | `DELETE /v1/ks/memory/entries/{id}` |
| `view` | `GET /v1/ks/memory/entries` |

Because both runtimes use the same row shape, an application can switch between LangGraph and the Anthropic memory tool without re-shaping its memory data.

## Resolution on the inference path

When `memory_class_hints` appears on a `kovaserve` envelope, GCM v2 calls the Memory Store at manifest assembly time:

```
1. GCM reads memory_class_hints[]
2. For each hint:
     selector = {namespace, optional key/prefix, scope_type}
     entries  = MemoryStore.search(selector)
3. Each entry becomes a system_memory ContextSegment
4. Segments are placed in canonical order in the manifest
5. Manifest admission proceeds
```

The runtime supplies hints; the store resolves; GCM orders. The runtime does not handle memory bytes itself on the inference path.

## Retention and the sweeper

The Memory Store runs a background sweeper that:

- Expires entries past their `ttl_seconds` or `retention` window.
- Decrements scope-extension reference counts when [CompactionRecords](/docs/concepts/context/compaction-records) referencing entries are pruned.
- Emits `ks.memory.entry.expired` events on expiry.

The sweeper is bounded — entry expiry is *lazy* on read (a read of an expired entry returns `not_found` and triggers immediate row deletion), and the background pass simply ensures rows that are never read are eventually cleaned.

## Scope and policy enforcement

Every operation validates the requesting AuthContext against the entry's scope:

- A request with `tenant_id = T` cannot read an entry owned by `tenant_id = T'`.
- A request with `policy_scope.context_share_scope = project` cannot read `tenant`-scoped entries from outside its project.
- A request with `scope_type = global` cannot **write** unless the AuthContext has deployment-admin permission. Reads are open within the deployment.

Violations return `404 Not Found`, never `403 Forbidden` — the existence of foreign-scope entries is not leaked.

## Storage backend

| Component | Backend |
|---|---|
| Row store | PostgreSQL. Indexes on `(tenant_id, project_id, scope_type, scope_id, namespace)` and `(scope_id, ttl_seconds)`. |
| Event emission | NATS JetStream (`ks.memory.*` subjects). |
| Vector search | Optional pgvector. Search endpoint returns `503` when not configured. |
| Audit projection | Standard observability pipeline. |

The wire surface is identical in managed cloud and private deployments. Migration is a row-and-event replay, not a re-shape.

## What the store is not

- **Not a substitute for [Sessions](/docs/concepts/continuity/sessions).** Sessions hold turn history; the store holds scope-rooted state.
- **Not a substitute for [ContextArtifacts](/docs/concepts/context/context-artifacts).** Large opaque payloads belong in artifacts. Entries hold structured JSON.
- **Not a runtime construct.** The store is a Layer 2 service — every runtime sees the same store.
