A **Budget** is a Run-level execution control object that bounds cost, tokens, steps, time, or tool count for a single [Run](/docs/concepts/run-hierarchy/runs). Budgets are **active** — they evaluate before every Step admission and produce explicit decisions (`ALLOW`, `WARN`, `DEGRADED_ALLOWED`, `HARD_STOP`) — not passive billing records.

Budgets are produced and consumed by the **Metering / Cost / Budget Layer**. Every non-anonymous Run carries exactly one Budget; the Budget is bound at Run creation and immutable thereafter.

## Dimensions

A Budget can constrain any subset of these dimensions. Limits left unset are unbounded.

| Dimension | Limit field | Counter |
|---|---|---|
| Cost | `cost_limit_cents` | `cost_used_cents` |
| Tokens | `token_limit` | `tokens_used` |
| Steps | `step_limit` | `steps_used` |
| Wall time | `time_limit_seconds` | `time_used_seconds` |
| Tool calls | `tool_limit` | `tools_used` |

Cost values are 4-decimal-digit strings (`"0.0420"`) per ADR-025. Token counts are real measurements from the substrate, never client-side estimates. The `tool_limit` dimension caps the count of tool invocations made during the Run; tool consumption is recorded on every `TOOL_EXECUTE` Step admission and counted against the limit alongside cost, tokens, steps, and time.

## Run-level vs context-level

Budgets are distinct from [ContextBudgets](/docs/concepts/policy/context-budgets):

| | Budget | ContextBudget |
|---|---|---|
| Governs | Cost / tokens / steps / time / tool count | Active context size |
| Scope | Whole Run | Per ModelCall |
| Decision moment | Step admission | Manifest admission, before GPU dispatch |
| Decisions | `ALLOW` / `WARN` / `DEGRADED_ALLOWED` / `HARD_STOP` | `ALLOW` / `WARN` / `OFFLOAD_REQUIRED` / `COMPACT_REQUIRED` / `DEGRADE` / `HARD_STOP` |

Both are evaluated. The Run-level Budget runs *after* ContextBudget admission, against the *admitted* manifest's token counts and cost estimate (not raw client estimates).

## State machine

```
ACTIVE ──▶ WARNING ──▶ DEGRADED_ALLOWED ──▶ HARD_STOP ──▶ CLOSED
   │           │                                 ▲
   └───────────┴─────────────────────────────────┘
```

| State | Meaning |
|---|---|
| `ACTIVE` | Consumption below the warning threshold. Decisions are `ALLOW`. |
| `WARNING` | Consumption above the warning threshold but below the limit. Emits `ks.budget.warning` events. Decisions are still `ALLOW`. |
| `DEGRADED_ALLOWED` | Consumption above limit on a non-strict dimension; the runtime is allowed to continue under reduced parameters (smaller `max_tokens`, dropped optional segments). |
| `HARD_STOP` | Consumption hit a strict limit. Further Steps are rejected with `error.code = "budget_exceeded"`. |
| `CLOSED` | Run terminal. Counters are frozen for audit. |

`HARD_STOP` is enforced on every Step admission, retry or not. A Run that exhausts its `step_limit` Budget on retries cannot continue retrying. `WARNING` may also transition directly to `HARD_STOP` when consumption jumps from below the warning threshold to a hard limit in a single admission.

## Inheritance on branches

| Operation | Budget behavior |
|---|---|
| [Branch](/docs/concepts/recovery/branches) from a Checkpoint | Limits inherited from parent (overridable downward); counters reset to zero. |
| [Resume](/docs/concepts/recovery/resume-flow) from a Checkpoint | New Run, fresh Budget — typically inheriting the original limits but with consumed counters at zero. |
| [Retry](/docs/concepts/recovery/retries) (new Step within same Run) | Counters continue advancing; the Run's Budget is not reset. |

Budgets cannot be raised after Run creation. The intent is to scope work, not to expand limits dynamically.

## Events

| Event | Emitted on |
|---|---|
| `ks.budget.warning` | Threshold crossed into `WARNING`. |
| `ks.budget.forecast_updated` | Forecast model updates (typically every Step admission). |
| `ks.budget.hard_stop` | Limit reached; subsequent admissions reject. |

See [budget events](/docs/concepts/events/budget) for the full taxonomy and envelope.

## What Budgets are not

- **Not billing records.** Budgets are control objects. Billing comes from metering against real consumed values.
- **Not raise-able.** A Run's Budget is bound at creation and immutable.
- **Not retroactive.** A Budget that crosses `HARD_STOP` does not cancel in-flight ModelCalls; it rejects subsequent Step admissions.
- **Not a substitute for [PolicyScope](/docs/concepts/policy/policy-scopes).** Budgets bound resource consumption; PolicyScope bounds *what* a Run is allowed to do.
