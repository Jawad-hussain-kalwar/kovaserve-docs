A **PolicyScope** is the normalized governance binding that attaches to every [Run](/docs/concepts/run-hierarchy/runs), [ModelCall](/docs/concepts/run-hierarchy/model-calls), and [Checkpoint](/docs/concepts/recovery/checkpoints). It carries model ACLs, rate limits, share-scope rules, data boundary constraints, environment/deployment restrictions, and tool policy enrichment fields.

Every request resolves to exactly one PolicyScope through the tenant hierarchy. Allow, deny, and degrade decisions are explainable and auditable.

## What a PolicyScope binds

| Field | Purpose |
|---|---|
| `model_acl[]` | Allowed model ids and families. Requests for non-listed models reject. |
| `rate_limits` | Per-tenant / project / environment rate limits in requests per second and tokens per second. |
| `context_share_scope` | Cache and manifest reuse boundary: `none` / `project` / `tenant`. Controls cross-Run KV cache and manifest-tier reuse. The `tenant` enum value is the wire token for tenant-wide reuse. |
| `data_boundary` | Artifact and context residency constraint: `tenant_local` / `region_local` / `unrestricted`. The `tenant_local` enum value pins residency to tenant-allocated storage. |
| `tool_policy_ref` | Reference to a tool policy applied at admission. The tool policy gates which tools the resolved scope may invoke and is evaluated alongside `model_acl`. |
| `deployment_restrictions[]` | Deployment ids the scope may execute against; empty = all deployments in scope. |
| `environment_restrictions[]` | Environment ids — typically `prod` / `staging` / `dev`. |
| `degraded_mode_policy` | Whether `DEGRADE` admission decisions are accepted, rejected, or warn-only. |

PolicyScopes are tenant- and project-resolvable but not project-immutable: a Run created under a project picks up that project's resolved PolicyScope at creation time, and the scope is then *frozen* on the Run for the Run's lifetime — later changes to the project's PolicyScope do not retroactively apply.

## Resolution

Every request's PolicyScope is resolved through the tenant hierarchy at admission time:

```
Tenant policy
  ▼ overlay
Project policy
  ▼ overlay
Environment policy
  ▼ overlay
Deployment policy
  ▼ overlay
API key scope
  ▼
Resolved PolicyScope (frozen on the Run)
```

Each layer overlays the previous one — a more-specific layer can narrow but not widen. A Project policy that adds a model to the ACL when the Tenant policy excludes it has no effect; the Tenant exclusion wins.

The resolved scope is recorded on the Run so subsequent Steps and ModelCalls do not re-resolve. API keys are issued bound to a PolicyScope at the project or environment layer — the key carries bearer identity, while the resolved scope carries the policy itself.

## Decision shape

Policy decisions are explicit and structured:

| Decision | When |
|---|---|
| `ALLOW` | Request matches scope; proceed. |
| `DENY` | Hard mismatch (model not in ACL, rate limit exceeded, foreign deployment). Returns `403 Forbidden` with explanation. |
| `DEGRADED_ALLOW` | Scope permits a degraded variant of the request (e.g. a smaller model, a reduced context window). Used in conjunction with admission-side `DEGRADE`. |

Every decision is emitted as a `ks.policy.evaluated` event with the matched rule and rationale. Denials emit `ks.policy.denied`; degraded paths emit `ks.policy.degraded_mode_applied`.

## `context_share_scope`

The reuse boundary for cache and manifest-tier reuse:

| Value | Reuse boundary |
|---|---|
| `none` | Each Run gets a private cache namespace. No cross-Run reuse, even within the same Session. |
| `project` | Reuse across Runs in the same project. The default for managed cloud workloads. |
| `tenant` | Reuse across Runs in the same tenant, across projects. Used for shared-tenant deployments. |

`context_share_scope` is a hard scope filter on routing. The [manifest-aware router](/docs/concepts/routing/manifest-aware-routing) only considers replicas whose cache origin agrees with the resolved `context_share_scope`. A request from a `project`-scoped Run cannot route to a replica warmed by a different project even if the cache content would otherwise match.

## `data_boundary`

The residency constraint for [ContextArtifacts](/docs/concepts/context/context-artifacts) and KV state:

| Value | Constraint |
|---|---|
| `tenant_local` | All artifacts and KV exports stay in storage allocated to the tenant. |
| `region_local` | All artifacts and KV exports stay in the deployment's region. |
| `unrestricted` | No residency constraint beyond tenant scoping. |

`data_boundary` is enforced at write time on the Artifact Store and at restore time on the [Checkpoint Manager](/docs/concepts/recovery/checkpoints). A Checkpoint cannot restore in a deployment that violates its `data_boundary`.

## Inheritance on branches

| Operation | PolicyScope behavior |
|---|---|
| [Branch](/docs/concepts/recovery/branches) | Inherited; overridable narrower (never wider). |
| [Resume](/docs/concepts/recovery/resume-flow) | The resumed Run must satisfy the Checkpoint's recorded scope. |

A scope is monotonically narrowing across lineage. There is no path that widens scope without an out-of-band administrative grant.
