title: Run state machine
description: The 12-state Run FSM — every status, every legal transition, and what each state means for the substrate, the runtime, and the control plane.
---

The **Run state machine** is the lifecycle every [Run](/docs/concepts/run-hierarchy/runs) travels from creation to termination. It has twelve states and a strict, validated transition graph: the Canonical Execution Service rejects illegal transitions at the service layer, and terminal states are frozen.

The FSM is the contract that lets long-running execution be recoverable, resumable, and auditable. Every state has a clear meaning for what the substrate, the runtime, and the control plane are doing — and what they must not do.

## The graph

```
CREATED
  └─▶ READY
        └─▶ RUNNING ──┬─▶ WAITING ─────▶ RUNNING
                      ├─▶ PAUSED ──────▶ RESUMING ─▶ RUNNING
                      ├─▶ CHECKPOINTING ┬─▶ RUNNING
                      │                 └─▶ PAUSED
                      ├─▶ RECOVERING ──▶ RESUMING ─▶ RUNNING
                      ├─▶ BRANCHING ───▶ RUNNING
                      ├─▶ COMPLETED                    (terminal)
                      ├─▶ FAILED                       (terminal)
                      └─▶ CANCELED                     (terminal)

CREATED ─▶ CANCELED                                    (terminal)
READY   ─▶ CANCELED                                    (terminal)
```

Every non-terminal state can transition to `CANCELED` if cancellation is requested while the Run is in that state. `FAILED` is reachable from any non-terminal state when an error condition makes continued execution impossible. The diagram above shows the canonical paths; the cancellation and failure escape hatches apply uniformly.

## States

### `CREATED`

The Run row exists; setup is in progress. PolicyScope, Budget, and ContextBudget are being resolved and bound. No Steps exist yet. The runtime has not yet been notified that the Run is ready to execute.

**Transitions out:** `READY` (setup complete), `CANCELED` (cancellation requested before readiness), `FAILED` (setup failed — e.g. PolicyScope resolution rejected the Run).

### `READY`

Setup is complete. PolicyScope, Budget, and ContextBudget are bound. The Run is admissible and the runtime can begin work, but no Step has yet been admitted.

**Transitions out:** `RUNNING` (first Step admitted), `CANCELED`, `FAILED`.

### `RUNNING`

The Run is actively executing. Steps are being admitted and ModelCalls are landing on the substrate. This is the central hub — every operational state below is reached from `RUNNING` and most return to `RUNNING` when their work completes.

**Transitions out:** `WAITING`, `PAUSED`, `CHECKPOINTING`, `RECOVERING`, `BRANCHING`, `COMPLETED`, `FAILED`, `CANCELED`.

### `WAITING`

The Run is alive but blocked on an external dependency the runtime declared — a tool result, a long-running process, an external system response, an asynchronous callback. The Run holds its bindings and its current ContextManifest position; it is not consuming GPU dispatch.

**Transitions out:** `RUNNING` (the wait resolved), `FAILED`, `CANCELED`.

### `PAUSED`

The Run is paused — by an explicit `PAUSE` Step from the runtime, by a budget warning that triggered a pause-and-checkpoint policy, or by an operator action. Distinct from `WAITING` in that the pause is a deliberate hold, not a blocked external dependency.

**Transitions out:** `RESUMING` (resume requested; goes through the resume sub-state), `CANCELED`, `FAILED`.

### `CHECKPOINTING`

The Checkpoint Manager is capturing a [Checkpoint](/docs/concepts/recovery/checkpoints) — freezing the current ContextManifest snapshot, gathering segment / artifact / compaction references, exporting recoverable KV state if available, writing the checkpoint manifest to object storage, and validating compatibility.

**Transitions out:** `RUNNING` (capture complete, Run continues), `PAUSED` (capture complete, Run held), `FAILED` (capture failed unrecoverably).

### `RECOVERING`

The first half of the resume sub-state. The Checkpoint Manager has loaded a checkpoint manifest, is validating the ContextManifest and referenced ContextArtifacts, and is checking model and tokenizer compatibility against the target deployment.

This state is entered by:
- A new Run that was created to resume from a Checkpoint (`CREATED → READY → RECOVERING`).
- An existing Run that hit a recoverable failure and is being driven through restore (`RUNNING → RECOVERING`).

**Transitions out:** `RESUMING` (validation succeeded), `FAILED` (validation failed — the manifest, artifacts, or compatibility vector did not check out).

### `RESUMING`

The second half of the resume sub-state. KV restore is being attempted if a `kv_prefix` StateHandle is present and compatibility holds; otherwise the active context is being reconstructed from the manifest and artifacts. Routing has selected a target node based on `resume_feasible` and restore cost.

**Transitions out:** `RUNNING` (resume complete), `FAILED`.

The `RECOVERING → RESUMING → RUNNING` path is the canonical resume sequence. A resumed Run is functionally indistinguishable from one that ran end-to-end on a single node — only the latency profile differs based on whether KV restore was available.

### `BRANCHING`

A new Run lineage is being created from the current Run's stable point or Checkpoint. The branch inherits the parent's PolicyScope and a bounded copy of its Budget. The original Run continues at `RUNNING` once the branch is dispatched; the branch itself is a separate Run that walks its own state machine from `CREATED`.

**Transitions out:** `RUNNING` (branch dispatched), `FAILED`, `CANCELED`.

### `COMPLETED` (terminal)

The Run finished its work successfully. `completed_at` is set, the cost ledger is closed, the bound Budget is released, and `ks.run.completed` is published. The Run row is durable and read-only.

A `COMPLETED` Run cannot be re-opened. If more work is needed, a new Run is created — typically dispatched as a sub-run from the same parent, or chained in the same Session as a follow-up turn.

### `FAILED` (terminal)

The Run failed in a way the system classifies as terminal — an unrecoverable substrate error, a resume that exhausted all restore options, a PolicyScope resolution that rejected the Run at setup, an explicit `FAIL` Step from the runtime. `completed_at` is set, the Budget is closed, and `ks.run.failed` is published.

`FAILED` is distinct from a `FAILED`-status Step or ModelCall: the Run terminates only when failure is unrecoverable. A single ModelCall failure that the runtime retries does not move the Run to `FAILED`.

### `CANCELED` (terminal)

The Run was canceled — by `POST /v1/ks/runs/{id}/cancel`, by a budget hard-stop with a cancel policy, or by an operator action. In-flight ModelCalls are allowed to settle so cost attribution stays accurate; further admission stops immediately. `completed_at` is set, the Budget is closed, and `ks.run.canceled` is published.

## Terminal-state guarantees

`COMPLETED`, `FAILED`, and `CANCELED` are durable and frozen. Once the Run reaches a terminal state:

- `status` cannot change. The Canonical Execution Service rejects any transition request.
- `completed_at` is set and is immutable.
- New Steps and ModelCalls cannot be inserted under the Run.
- The bound Budget is closed; further metering attempts targeting this Run are rejected.
- The Run row remains queryable indefinitely for audit. Terminality is a guarantee about *future* writes, not a signal to delete the row.

A bound [Session](/docs/concepts/continuity/sessions)'s `current_run_id` remains pinned to the terminal Run until a new Run binds in. This is intentional — it preserves a "last run" pointer that is useful for resume planning.

## Cancellation can arrive in any non-terminal state

`CANCELED` is reachable from `CREATED`, `READY`, `RUNNING`, `WAITING`, `PAUSED`, `CHECKPOINTING`, `RECOVERING`, `RESUMING`, and `BRANCHING`. The Canonical Execution Service treats cancellation as a high-priority transition: it is admitted ahead of normal work, and any in-flight operation that has not yet committed is allowed to settle without producing further side effects.

Cancellation while `CHECKPOINTING` is the trickiest case. If the capture has not yet validated, the Checkpoint is left in `FAILED` (it never reached `VALIDATED`); the Run transitions to `CANCELED` cleanly. If the capture validated before cancellation arrived, the Checkpoint is `VALIDATED` and remains usable — the Run is canceled, but a future Run can still resume from the validated Checkpoint.

## Failure can arrive in any non-terminal state

`FAILED` is similarly reachable from any non-terminal state, but only when the system classifies the failure as unrecoverable. Recoverable failures stay inside `RUNNING` (or move into `RECOVERING` if a Checkpoint-driven retry is initiated). The runtime decides recoverability for runtime-level errors; the Canonical Execution Service decides it for setup, admission, and resume errors.

## Transitions are validated

Every transition is checked against this graph by the Canonical Execution Service before the Run row is updated. Illegal transitions — `COMPLETED → RUNNING`, `RECOVERING → COMPLETED` without going through `RESUMING`, `READY → CHECKPOINTING` — are rejected. The service is the only writer of `runs.status`; runtimes and operators request transitions, they do not perform them directly.

## Events emitted on each transition

Every transition emits an event on the execution stream so dashboards, audit consumers, and the [Run timeline](/docs/concepts/run-hierarchy/runs) can stitch the lifecycle without polling:

| Transition | Event |
|---|---|
| → `CREATED` | `ks.run.created` |
| → `RUNNING` from `READY` | `ks.run.started` |
| → `PAUSED` | `ks.run.paused` |
| → `RECOVERING` | `ks.run.recovering` |
| → `RUNNING` from `RESUMING` | `ks.run.resumed` |
| → `COMPLETED` | `ks.run.completed` |
| → `FAILED` | `ks.run.failed` |
| → `CANCELED` | `ks.run.canceled` |

Intermediate transitions (`WAITING`, `CHECKPOINTING`, `BRANCHING`) emit corresponding events on the relevant subsystem stream — `ks.checkpoint.*` for the checkpoint flow, `ks.branch.created` for branching, runtime-level events for waits.
