title: Step state machine
description: The Step FSM — six statuses, one linear path from PENDING to a terminal state, and how each terminal status is interpreted.
---

The **Step state machine** is the lifecycle every [Step](/docs/concepts/run-hierarchy/steps) travels from creation to termination. It has six statuses and a strict, linear shape: a Step starts in `PENDING`, moves to `STARTED` when work begins, and ends in exactly one of four terminal statuses — `SUCCEEDED`, `FAILED`, `SKIPPED`, or `CANCELED`.

The FSM is intentionally simple. Steps are not transactions and not synchronization primitives; they are typed progression markers. Their state machine records *whether the marker fired* and *how it resolved*, nothing more. The richer lifecycle of long-running execution — pauses, resumes, checkpoints, branches — lives on the [Run state machine](/docs/concepts/run-hierarchy/run-state-machine).

## The graph

```
PENDING ─▶ STARTED ──┬─▶ SUCCEEDED   (terminal)
                     ├─▶ FAILED      (terminal)
                     ├─▶ SKIPPED     (terminal)
                     └─▶ CANCELED    (terminal)

PENDING ─────────────┬─▶ SKIPPED     (terminal — never started)
                     └─▶ CANCELED    (terminal — never started)
```

Two non-substrate paths are legal:

- A `PENDING` Step can transition directly to `SKIPPED` if the runtime decides the Step should not execute (e.g. a `TOOL_PLAN` whose plan resolves to "no tool needed").
- A `PENDING` Step can transition directly to `CANCELED` if the owning Run is canceled before the Step is admitted to execution.

A `STARTED` Step always reaches one of the four terminal statuses. There is no path back to `PENDING`, and there is no `RESUMED` status — a Step is a single-shot record. If work resumes, a new Step is emitted.

## Statuses

### `PENDING`

The Step row exists but no work has begun. The Step has been admitted (admission gates run *before* Step creation — see [Steps](/docs/concepts/run-hierarchy/steps#when-a-step-is-created)), so a Step in `PENDING` is committed to executing unless explicitly skipped or canceled.

**Transitions out:** `STARTED` (work begins), `SKIPPED` (runtime decided not to execute), `CANCELED` (Run canceled before start).

### `STARTED`

Work is in progress. For substrate-bound Steps (`MODEL_CALL`, `TOOL_PLAN`), this means the Cache-Aware Router has selected a node and dispatch is underway. For runtime-emitted Steps (`OBSERVE`, `THINK`, `CHECKPOINT`, `PAUSE`, `RESUME`, `BRANCH`, `COMPLETE`, `FAIL`), `STARTED` means the runtime has begun the corresponding action.

`started_at` is set on this transition. The Canonical Execution Service publishes `ks.step.started`.

**Transitions out:** `SUCCEEDED`, `FAILED`, `CANCELED`. (A `STARTED` Step never moves to `SKIPPED` — skipping is a pre-execution decision.)

### `SUCCEEDED` (terminal)

The Step's work completed successfully. For substrate-bound Steps, all owned ModelCalls landed and recorded usage. For runtime Steps, the action completed without error. `completed_at` is set; `ks.step.completed` is published with the terminal status.

A `SUCCEEDED` `MODEL_CALL` Step does **not** imply the model produced a useful response — only that the substrate executed and the [ModelCall](/docs/concepts/run-hierarchy/model-calls) row was recorded. The runtime decides whether the response was useful and emits subsequent Steps accordingly.

### `FAILED` (terminal)

The Step's work failed. For substrate-bound Steps, this covers substrate errors, admission-time rejections that escaped the pre-Step gates, and routing failures with no viable fallback. For runtime Steps, this covers the runtime's own classification of failure — a tool that errored, a `FILE_WRITE` blocked by workspace policy, a process that violated its sandbox.

A `FAILED` Step does **not** automatically fail the Run. The runtime decides whether to retry (a new Step), branch off a Checkpoint, or surface the failure as terminal via a `FAIL` Step. The [Run state machine](/docs/concepts/run-hierarchy/run-state-machine) only moves to `FAILED` when failure is unrecoverable at the Run level.

### `SKIPPED` (terminal)

The Step was decided not to execute, with no failure semantics. Reached only from `PENDING`. Use cases:

- A `TOOL_PLAN` Step whose plan resolves to "no action needed."
- A speculative Step the runtime emitted optimistically and then declined to execute.
- A `FILE_READ` Step where the runtime's cache already covers the requested paths.

`SKIPPED` is distinct from `FAILED` in that it carries no error metadata and counts as a successful no-op for retry and recovery logic. The Step's `ordinal` is consumed and not reused.

### `CANCELED` (terminal)

The Step was canceled before completion. Reachable from both `PENDING` (Run canceled before Step started) and `STARTED` (Run canceled mid-Step). When canceled from `STARTED`, in-flight ModelCalls are allowed to settle so cost attribution stays accurate; the resulting ModelCall rows are recorded under the canceled Step.

Cancellation propagates from the Run, never from another Step. There is no inter-Step cancellation primitive.

## Step status vs. Run status

Step statuses and Run statuses are **independent**. A Run in `RUNNING` will have Steps continuously moving through their own state machines, and the Run's status reflects the Run-level lifecycle, not any individual Step's:

| Run status | Step status implications |
|---|---|
| `RUNNING` | Any number of Steps in any non-terminal status. |
| `WAITING` / `PAUSED` | Existing Steps are typically in a terminal status; new Steps are not being admitted. |
| `CHECKPOINTING` | The Checkpoint Step itself is `STARTED`; other Steps may be in any status. |
| `RECOVERING` / `RESUMING` | A `RESUME` Step is `STARTED`; the rest of the Run's Steps are in their previous terminal states. |
| `COMPLETED` / `FAILED` / `CANCELED` | All Steps must be in a terminal status. The Run does not reach a terminal state with non-terminal Steps outstanding. |

The terminal-state guarantee on the Run implies the same on its Steps: when the Run is terminal, every Step has reached a terminal status. The Canonical Execution Service enforces this on Run termination — a Run cannot complete with a `PENDING` or `STARTED` Step, and any such Step is settled (typically to `CANCELED`) as part of the Run's terminal transition.

## Failure recovery is a new Step

The Step state machine has no retry transition by design. A `FAILED` Step is the durable record of what happened; a retry is a new Step. This keeps the timeline honest:

- The original failure is queryable forever.
- The retry has its own `ordinal`, its own `created_at`, and its own ModelCalls.
- Cost attribution is per-attempt — the failed Step's ModelCall (if any) is recorded against the Run's Budget; the retry's ModelCall is recorded separately.
- Audit shows the full history, not just the last successful attempt.

The retry Step may have a different `step_type` than the original. A `FAILED` `MODEL_CALL` may be retried as a `MODEL_CALL` against a fallback model, or as a `TOOL_PLAN` if the runtime decided to switch strategies. Lineage between original and retry, when meaningful, is recorded in the retry Step's `metadata`.

## Skipping is a deliberate, recorded decision

`SKIPPED` exists so the timeline records *that the runtime considered this Step and chose not to execute it*, distinct from a Step that was never created. This matters for:

- Audit — explainability of the Run's decisions.
- Metering — `SKIPPED` Steps consume an `ordinal` and an event but no cost.
- Branch reasoning — a Branch from a Step boundary that includes a `SKIPPED` Step preserves the decision context.

Runtimes are not required to use `SKIPPED`. A runtime that simply does not emit a Step it considered is also valid. `SKIPPED` is the option for runtimes that want the consideration on the record.

## Events on transitions

Three events on the execution stream make Step transitions observable:

| Transition | Event |
|---|---|
| (insert) `→ PENDING` | `ks.step.created` |
| `PENDING → STARTED` | `ks.step.started` |
| `STARTED | PENDING → SUCCEEDED | FAILED | SKIPPED | CANCELED` | `ks.step.completed` |

`ks.step.completed` carries the terminal status in its payload. There is intentionally no per-terminal-status event — consumers filter on the payload, not on the event subject.

Each event carries `run_id`, `step_id`, `ordinal`, `step_type`, the transition's previous and new statuses, and `tenant_id` / `project_id` for tenant-safe filtering.

## Transitions are validated

Every transition is checked by the Canonical Execution Service before the Step row is updated:

- `PENDING → STARTED | SKIPPED | CANCELED` — legal.
- `STARTED → SUCCEEDED | FAILED | CANCELED` — legal.
- Anything else, including any transition out of a terminal status, is rejected with `409 Conflict` and `error.code = "step_transition_invalid"`.

The service is the only writer of `steps.status`. Runtimes request transitions through the Step API or implicitly through substrate dispatch; they do not write the column directly.
