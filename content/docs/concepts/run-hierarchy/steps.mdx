A **Step** is one typed unit of progression inside a [Run](/docs/concepts/run-hierarchy/runs). Steps are how you reason about *what the Run is doing*; [ModelCalls](/docs/concepts/run-hierarchy/model-calls) inside a Step are how you reason about *what it cost and how it was placed*. Together they give a long-running execution a discrete, queryable timeline without forcing the runtime to standardize on a particular orchestration framework.

Steps are owned by the **Canonical Execution Service**. The service creates a Step row only after admission has succeeded, transitions the Step through its lifecycle, and embeds Steps into the Run timeline payload returned by `GET /v1/ks/runs/{id}`.

## What a Step records

A Step is a thin row by design — it points at the things that did the work, rather than carrying the work itself.

| Field | Purpose |
|---|---|
| `id` | The Step identifier (bare 26-character Crockford ULID). |
| `run_id` | The owning [Run](/docs/concepts/run-hierarchy/runs). Immutable. |
| `step_type` | The typed progression marker. See [Step types](/docs/concepts/run-hierarchy/step-types) for the canonical enum. |
| `status` | The current state in the [Step state machine](/docs/concepts/run-hierarchy/step-state-machine). |
| `ordinal` | Monotonic position within the Run. Assigned at creation and never reused. |
| `parent_step_id` | The dispatching Step when this Step nests inside another (e.g. a `TOOL_EXECUTE` under a `TOOL_PLAN`). `null` for top-level Steps. |
| `created_at` / `started_at` / `completed_at` | Lifecycle timestamps. `completed_at` is set when the Step reaches a terminal state. |
| `metadata` | Open key-value bag for runtime-supplied annotations the Canonical Execution Service does not interpret. |

Prompts, tool outputs, model responses, and accumulated context live on the [ContextManifest](/docs/concepts/context/context-manifests) (per ModelCall), on [ContextArtifacts](/docs/concepts/context/context-artifacts) (for offloaded payloads), and on tool result records. The Step row points at this material rather than inlining it.

## When a Step is created

Step creation is **deferred until after admission succeeds**. This is a load-bearing rule, not an implementation detail:

```
1. Gateway authenticates and resolves PolicyScope.
2. GCM assembles the ContextManifest.
3. ContextBudget admission evaluates the manifest.
   ├─ HARD_STOP  → request rejected. NO STEP ROW IS CREATED.
   ├─ DEGRADE / OFFLOAD_REQUIRED / COMPACT_REQUIRED → manifest re-shaped, re-admitted.
   └─ ALLOW / WARN → admission succeeds.
4. Run-level Budget pre-check against admitted manifest.
   ├─ Reject → NO STEP ROW IS CREATED.
   └─ Allow → continue.
5. STEP ROW IS CREATED, bound to the admitted manifest.
6. Router plans placement; Layer 1 executes; ModelCall is recorded.
```

The consequence: a Run's `step_count` reflects only Steps the system actually committed to executing. A request that bounced off [ContextBudget](/docs/concepts/policy/context-budgets) admission or run-level [Budget](/docs/concepts/policy/budgets) pre-check leaves no Step trace on the Run row. Audit and metering of *rejected* requests live on the policy and budget event streams instead — `ks.policy.denied`, `ks.context.budget.hard_stop`, `ks.budget.hard_stop` — keyed to the Run but not consuming a Step ordinal.

This rule applies to every Step type that is admission-gated. Runtime-emitted markers like `OBSERVE`, `THINK`, `CHECKPOINT`, `PAUSE`, `RESUME`, and `BRANCH` are not gated by ContextBudget admission — they record state transitions, not GPU dispatches — and are created when the runtime emits them.

## Binding to the ContextManifest

When a Step is created in step 6 above, it is bound to the admitted [ContextManifest](/docs/concepts/context/context-manifests). The binding is recorded on the Step's [ModelCall](/docs/concepts/run-hierarchy/model-calls) row via `context_manifest_id`, so every ModelCall inside the Step traces back to the exact manifest the substrate consumed.

A Step typically owns one ModelCall, but Steps that issue multiple parallel substrate calls — a `TOOL_PLAN` that fans out across several models, a `MODEL_CALL` Step that triggers structured output paired with reasoning — may own several. Each ModelCall inside the Step carries its own `context_manifest_id`; the Canonical Execution Service does not coalesce them.

Steps that do not invoke the substrate (`OBSERVE`, `THINK`, `CHECKPOINT`, `PAUSE`, `RESUME`, `BRANCH`, `COMPLETE`, `FAIL`) own zero ModelCalls. Their effect is recorded on the Step row, on the relevant subsystem (Checkpoint Manager, branch lineage), and on the corresponding event stream.

## Ordering inside a Run

`ordinal` is the canonical ordering within a Run. It is monotonic, gap-free, and assigned at Step creation. Two Steps cannot share an ordinal; ordinals are not reused if a Step transitions to `FAILED`, `SKIPPED`, or `CANCELED`.

`created_at` is also recorded; the ordering source is `ordinal`. Clock skew across a distributed control plane can produce timestamp collisions or near-collisions, while `ordinal` is allocated by the Canonical Execution Service inside a single transaction with the Step insert. Clients that reconstruct timelines sort by `ordinal`.

Nested Steps reference their parent through `parent_step_id`. The runtime emits the parent Step first (e.g. a `TOOL_PLAN`), receives its ID, and then dispatches child Steps that point at it (`TOOL_EXECUTE` × N). Children appear after their parent in the ordinal sequence and may be interleaved with other top-level Steps as the runtime sees fit.

## Lifecycle in summary

A Step moves through six possible statuses: `PENDING`, `STARTED`, `SUCCEEDED`, `FAILED`, `SKIPPED`, `CANCELED`. Terminal statuses (`SUCCEEDED`, `FAILED`, `SKIPPED`, `CANCELED`) are durable; the Step row is never re-opened once it reaches one. The full transition graph and its semantics live on the [Step state machine](/docs/concepts/run-hierarchy/step-state-machine) page.

Three events on the execution stream make the lifecycle observable:

| Event | When |
|---|---|
| `ks.step.created` | The Step row is inserted — admission has already succeeded. |
| `ks.step.started` | The Step transitions from `PENDING` to `STARTED`. |
| `ks.step.completed` | The Step reaches a terminal status. |

Each event carries `run_id`, `step_id`, `ordinal`, and `step_type` in its envelope, alongside `tenant_id` and `project_id` for tenant-safe filtering.

## API shape

Steps are most often created implicitly by the gateway as part of a `/v1/chat/completions` or runtime call. Layer 3 runtimes also create Steps directly when emitting non-substrate markers.

### Create a Step explicitly

```http
POST /v1/ks/runs/01J8X9Y2Z3K4M5N6P7Q8R9S0T1/steps
Authorization: Bearer ks_...
Content-Type: application/json

{
  "step_type": "TOOL_EXECUTE",
  "parent_step_id": "01J8X9Y2Z3K4M5N6P7Q8R9S0T5",
  "metadata": {
    "tool_name": "shell.exec",
    "tool_invocation_id": "tinv_01J8X9..."
  }
}
```

```json
{
  "id": "01J8X9Y2Z3K4M5N6P7Q8R9S0T6",
  "run_id": "01J8X9Y2Z3K4M5N6P7Q8R9S0T1",
  "step_type": "TOOL_EXECUTE",
  "status": "PENDING",
  "ordinal": 7,
  "parent_step_id": "01J8X9Y2Z3K4M5N6P7Q8R9S0T5",
  "created_at": "2026-04-30T12:00:18Z",
  "started_at": null,
  "completed_at": null,
  "metadata": {
    "tool_name": "shell.exec",
    "tool_invocation_id": "tinv_01J8X9..."
  },
  "model_calls": []
}
```

### List Steps for a Run

```http
GET /v1/ks/runs/01J8X9Y2Z3K4M5N6P7Q8R9S0T1/steps
Authorization: Bearer ks_...
```

Returns Steps sorted by `ordinal` with each Step's ModelCalls embedded. The same payload appears inline in `GET /v1/ks/runs/{id}` — clients reconstructing a Run's full shape should prefer the embedded form to avoid an extra round trip.

### Read a single Step

```http
GET /v1/ks/runs/01J8X9Y2Z3K4M5N6P7Q8R9S0T1/steps/01J8X9Y2Z3K4M5N6P7Q8R9S0T6
Authorization: Bearer ks_...
```

## Step types are an open enum

Step types are validated at the application layer, not via a database `CHECK` constraint. The canonical set is large enough to cover the documented runtime semantics — the standard agent loop, workspace and process orchestration, reflection, durability and lineage, terminal markers, and context engineering — and new types can be introduced without a DDL change. See [Step types](/docs/concepts/run-hierarchy/step-types) for the full canonical list and what each type represents.

A runtime that does not need a particular type can ignore it. A runtime that wants a custom type beyond the canonical set should propose it; KovaServe extends the enum rather than letting the surface fragment.
