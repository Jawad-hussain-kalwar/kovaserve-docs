A **Run** is one durable execution objective. Inside it, **Steps** mark typed units of progression, and each Step contains the **ModelCalls** that actually hit the inference substrate. This is the smallest hierarchy that makes long-running work auditable without imposing an orchestration framework.

KovaServe's canonical execution service owns these records. The Run is the unit of value; Steps and ModelCalls exist so you can reconstruct what happened and where the time and cost went.

## Run

A Run represents one durable execution objective вЂ” a coding task, a document review, a multi-step tool loop. It is created by `POST /v1/ks/runs` (or implicitly by the first `/v1/chat/completions` call carrying a `kovaserve.run_id`) and persists until terminal.

A Run owns:

- A `PolicyScope` (resolved from tenant / project / environment).
- A `Budget` (cost / tokens / steps / time / tools).
- An optional `ContextBudget` (active-context admission control).
- An optional `parent_run_id` (sub-run lineage) and `summary_artifact_id` (compact handoff to the parent).
- A `Session` binding when continuity across turns is needed.

## Step

A Step is one typed unit of progression inside a Run. Steps are how you reason about *what the run is doing*; ModelCalls are how you reason about *what it cost and how it was placed*.

Step types are an open enum validated at the application layer. The current canonical set:

### Step types

| Group | Types |
|---|---|
| Standard agent loop | `MODEL_CALL`, `TOOL_PLAN`, `TOOL_EXECUTE` |
| Workspace and process | `FILE_READ`, `FILE_WRITE`, `PROCESS_START` |
| Reflection and reasoning | `OBSERVE`, `THINK` |
| Durability and lineage | `CHECKPOINT`, `PAUSE`, `RESUME`, `BRANCH` |
| Terminal markers | `COMPLETE`, `FAIL` |
| Context engineering | `COMPACT`, `MEMORY_READ`, `MEMORY_WRITE` |
| Sub-run | `SUB_RUN_DISPATCH`, `SUB_RUN_JOIN` |

New types extend the list without DDL changes. Runtimes that don't need a type can ignore it.

A Step row is created **after** admission succeeds. A request rejected by Budget or ContextBudget admission does not insert a Step.

## ModelCall

A ModelCall is one substrate-bound model invocation belonging to a Step. It records:

- The model (`model_id`) and node (`node_id`) where the call ran.
- A foreign key to the `ContextManifest` consumed by the call.
- Token usage, including `cached_tokens` (OpenAI-convergent) for prompt-cache hits.
- Cost broken into input / output / cache savings.
- The `traceparent` for OTel correlation.

A Step typically has one ModelCall, but tool-planning Steps may have several when the runtime issues parallel calls.

## How they appear in the API

`GET /v1/ks/runs/{id}` returns the Run with its Steps embedded, and each Step with its ModelCalls embedded. This embedding is the canonical timeline payload вЂ” clients do not need to issue N+1 calls to reconstruct the run shape.

## ID format

All IDs are bare 26-character Crockford ULIDs, no prefixes. The format is the same for runs, sessions, steps, model calls, manifests, checkpoints, and every other entity. Prefixing is intentionally not used вЂ” clients should not attempt to infer entity type from an ID.
