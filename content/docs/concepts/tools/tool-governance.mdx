**Tool governance** is the policy enforcement model that decides whether tool definitions and tool outputs may enter context, and how. It is owned by the **Tool Context Governance** subsystem (a member of the Context Engineering subsystem alongside GCM v2 and the Context Admission Service) and produces [ToolContextDispositions](/docs/concepts/tools/tool-context-disposition) for every tool surface that touches a [ContextManifest](/docs/concepts/context/context-manifests).

Governance is fast, deterministic, and auditable. Every decision is recorded; every rule that fired is preserved.

## What governance gates

Two surfaces flow through governance on every Run:

1. **Tool definitions.** When a [ToolCatalog](/docs/concepts/tools/tool-catalogs) is referenced, every definition is evaluated against the resolved [PolicyScope](/docs/concepts/policy/policy-scopes)'s `tool_policy_ref`. Definitions failing policy are dropped from the manifest's `tool_definitions` segment or the entire request is denied.
2. **Tool outputs.** When the runtime registers a tool result via `POST /v1/ks/tools/results` or `POST /v1/ks/tools/report`, the output is evaluated and a disposition is returned. The runtime acts on the disposition — inlines, summarizes, offloads, drops, or aborts.

In both cases, the same policy file produces the decision. Definitions and outputs are not governed by separate policies.

## Decision shape

Every governance evaluation produces a structured record:

| Field | Purpose |
|---|---|
| `decision` | `ALLOW` / `DENY` / `DEGRADED_ALLOW`. Hard gating. |
| `disposition` | The [ToolContextDisposition](/docs/concepts/tools/tool-context-disposition) value: `inline_full` / `inline_preview` / `summary_required` / `artifact_ref` / `drop` / `blocked`. |
| `risk_level` | Carried from the definition or escalated by the policy. |
| `checkpoint_recommended` | Advisory: should the runtime checkpoint before this invocation? |
| `context_disposition` | Structured carrier for `max_inline_tokens`, `artifact_required`, `redaction_policy`. |
| `rule_fired` | The first matching rule's id. |
| `reason` | Tag explaining the decision. |

`decision` is the hard gate. `disposition` is the shape gate. A `decision: ALLOW` with `disposition: blocked` is contradictory and rejected at policy compile time.

## Where governance fits in the flow

```
Tool catalog reference                    Tool result registration
       │                                          │
       ▼                                          ▼
   GCM resolves catalog              Tool gateway captures result
       │                                          │
       ▼                                          ▼
       └────▶ Tool Context Governance ◀──────────┘
                       │
                       │  ToolContextDisposition
                       ▼
       ┌──── manifest assembly ─────┐
       │                            │
   tool_definitions          tool_output / tool_output_summary
       segment                       segment
       │                            │
       └────▶ Context Admission ◀───┘
```

Governance runs *before* manifest admission. The disposition is the input to admission's transformation step — `summary_required` ensures admission expects a `tool_output_summary` segment; `artifact_ref` ensures admission expects a small reference segment, not a full inline tool output.

## Policy file shape

A tool policy is a structured rule list bound to a [PolicyScope](/docs/concepts/policy/policy-scopes) via `tool_policy_ref`:

```yaml
id: 01J8X9Y2Z3K4M5N6P7Q8R9TP01
name: production-default
default:
  decision: ALLOW
  disposition: inline_preview
  context_disposition:
    max_inline_tokens: 512
rules:
  - id: deny-secret-tools
    match:
      tool_name_in: [read_secrets, dump_credentials]
    decision: DENY
  - id: shell-high-risk
    match:
      tool_name: shell_exec
      risk_level: high
    decision: ALLOW
    disposition: summary_required
    checkpoint_recommended: true
  - id: oversize-outputs
    match:
      output_token_count_gt: 4000
    decision: ALLOW
    disposition: summary_required
  - id: image-outputs
    match:
      output_mime_type_glob: image/*
    decision: ALLOW
    disposition: artifact_ref
  - id: production-block-experimental
    match:
      tool_label: experimental
      environment: prod
    decision: DENY
```

Rules evaluate in declaration order; first match wins. The default applies when no rule matches. Compile-time validation catches contradictions (e.g., `decision: ALLOW` with `disposition: blocked`) and unreachable rules (a rule shadowed by an earlier broader rule).

## Risk levels

`risk_level` is set on every tool definition and may be escalated by policy. The default semantics:

| Risk | Default disposition | Default checkpoint advisory |
|---|---|---|
| `low` | `inline_preview` | none |
| `medium` | `inline_preview` (or `summary_required` for large outputs) | first-invocation-of-Run |
| `high` | `summary_required` | always |

A tool's intrinsic risk is the catalog's `risk_level`. Policy can lift a `low` tool to `high` for sensitive deployments, but cannot lower a `high` tool below `medium` — the floor is a property of the definition, not the policy.

## Checkpoint advisories

`checkpoint_recommended: true` is an advisory the runtime reads before invoking the tool. The runtime's response is its choice; governance does not enforce checkpoint capture.

Typical runtime behavior:

- `checkpoint_recommended: true` and the Run has not checkpointed in the last `n` Steps → take a [Checkpoint](/docs/concepts/recovery/checkpoints) before invoking.
- `checkpoint_recommended: true` and the Run is configured for `auto_checkpoint: false` → log the advisory, proceed.
- `checkpoint_recommended: false` → no advisory action.

Operators can audit "advised but not taken" through the `ks.tool.invocation.started` event payload.

## Audit trail

Every governance decision emits one event:

| Event | Emitted on |
|---|---|
| `ks.policy.evaluated` | Every decision. Includes `rule_fired`, `decision`, `disposition`. |
| `ks.policy.denied` | `decision: DENY`. Tool call rejected. |
| `ks.policy.degraded_mode_applied` | `decision: DEGRADED_ALLOW`. |

The events carry enough information to reconstruct, after the fact, why a tool was allowed, denied, summarized, or dropped — without re-evaluating the policy.

## Operator override

Deployment operators can override a denial through an explicit grant scoped to a single Run:

```http
POST /v1/ks/policies/overrides
Authorization: Bearer ks_admin_...
Content-Type: application/json

{
  "run_id": "01J8X9Y2Z3K4M5N6P7Q8R9S0T1",
  "tool_name": "read_secrets",
  "decision": "ALLOW",
  "disposition": "artifact_ref",
  "reason": "incident debugging — IR-2026-04-30",
  "expires_at": "2026-04-30T13:00:00Z"
}
```

Overrides are time-bound, audit-logged, and visible on the Run's timeline. They never apply tenant-wide; they apply to a specific Run.

## What governance is not

- **Not the runtime's policy engine.** Runtimes can carry their own checks, but governance at Layer 2 is the authoritative gate.
- **Not free-form.** Decisions and dispositions are closed enums.
- **Not silent.** Every decision is an event; no decision is implicit.
- **Not the tool gateway.** The tool gateway is the surface that *executes* tool calls. Governance decides whether the call may participate in context. The two are independent — a `decision: ALLOW` tool may still fail at execution.
