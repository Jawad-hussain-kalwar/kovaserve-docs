title: Private deployment

---

**Private deployment** is the KovaServe deployment topology where the entire control plane runs inside the customer's own infrastructure. It is the path for sovereign AI platforms, regulated enterprises, and neoclouds that need data residency, jurisdictional control, or air-gap operation while keeping the same Run / Session / Step / ModelCall / ContextManifest / Budget / Checkpoint semantics that [managed cloud](/docs/concepts/deployment/managed-cloud) provides.

```
┌─ Customer-controlled Kubernetes (k3s + Helm) ───────────┐
│                                                         │
│  Control plane (KovaServe Helm chart)                   │
│   ├── API Gateway                                       │
│   ├── Canonical Execution Service                       │
│   ├── Context engineering subsystem                     │
│   ├── Cache-Aware Router                                │
│   ├── Checkpoint Manager                                │
│   ├── Metering / Budgets                                │
│   └── Dashboard                                         │
│                                                         │
│  Customer-owned datastores                              │
│   ├── PostgreSQL                                        │
│   ├── Redis                                             │
│   ├── NATS JetStream                                    │
│   └── S3-compatible object storage                      │
│                                                         │
│  GPU nodes (vLLM + Node Agent + Engine Adapter)         │
│   ├── Node 1                                            │
│   └── Node N                                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

The deployment is single-tenant by topology — the entire control plane runs inside the customer's network. The same APIs, the same wire shapes, the same event taxonomy, and the same dashboard are served from that network. A workload that originated on managed cloud and migrates here keeps its IDs, its manifests, its checkpoints, and its [cost-per-completed-run](/docs/concepts/cost/cost-per-completed-run) accounting.

## Topology

KovaServe ships a Helm chart that targets [k3s or any conformant Kubernetes](/docs/guides/self-hosting-kubernetes) cluster. The reference 3-VM split:

| VM role | Components |
|---|---|
| Control node | Kubernetes control plane, API Gateway, Canonical Execution Service, Context engineering subsystem, Router, Checkpoint Manager, Metering, Dashboard, Postgres, Redis, NATS, MinIO. |
| GPU node 1 | vLLM, Node Agent, Engine Adapter, Transfer/Restore Worker, local Event Publisher. |
| GPU node N | Same as GPU node 1. Scales horizontally. |

Larger deployments split the control-plane datastores onto dedicated nodes and run more GPU nodes; smaller deployments compress everything onto fewer nodes. The chart parameterizes the layout — semantics do not change with topology.

## Customer-owned datastores

The contract is that the customer supplies them. The chart accepts connection strings and credentials at install time:

| Store | Purpose |
|---|---|
| PostgreSQL | Canonical entity persistence — Runs, Sessions, Steps, ModelCalls, ContextManifests, Budgets, Checkpoints, PolicyScopes, usage records. Migrations are managed by Alembic and apply on startup. |
| Redis | Rate-limit counters, idempotency cache, hot-path lookups. |
| NATS JetStream | `ks.>` event subjects with configurable retention; `kv_events` substrate stream; durable consumers for the dashboard, metering, and customer event consumers. |
| S3-compatible object storage | [Context Artifact Store](/docs/concepts/context/context-artifacts), [Checkpoint](/docs/concepts/recovery/checkpoints) blobs, KV export bundles, [WorkspaceArtifact](/docs/concepts/runtime/workspace-artifacts) bytes. MinIO, AWS S3, or any compatible service. |

A customer running on a sovereign cloud points the chart at the cloud's managed Postgres, managed Redis, managed NATS, and managed object store. A customer running fully air-gapped points it at locally operated equivalents.

## Layer 1 in private mode

The substrate layer also runs inside the customer's network. Each GPU node hosts:

- a **vLLM engine** with the KovaServe execution-metadata carrier extension
- a **Node Agent** — gRPC server that exposes node telemetry, KV state lookups, and lifecycle hooks
- an **Engine Adapter** — translates the canonical execution envelope into vLLM request shape
- a **Transfer/Restore Worker** — moves KV state across HBM, host RAM, NVMe, and object tiers while preserving stable state identity
- a **local Event Publisher** — emits `kv_events` into NATS

These are the same components that run on managed cloud. The boundary between Layer 1 and Layer 2 is identical.

## Identity, isolation, and tenancy

Private deployment is single-tenant by topology, but it still uses the four-level [tenancy](/docs/concepts/tenancy) hierarchy — Tenant → Project → Environment → Deployment — to scope resources, policies, budgets, and cache reuse internally:

| Surface | Behavior in private mode |
|---|---|
| Tenant | Typically one — the deploying entity. Multi-tenant deployments are supported when an enterprise wants strong logical separation between business units. |
| Project | Per application, product line, or customer-of-customer. Same semantics as managed cloud. |
| Environment | `production` / `staging` / `development` — independent traffic, deployments, and cache namespaces. |
| Key | Bearer credentials issued from the in-cluster Tenant Management surface. |
| `context_share_scope` | Defaults to `project`. Adjustable per [PolicyScope](/docs/concepts/policy/policy-scopes) — sovereign deployments often pin it to `project` or `environment` to forbid cross-application reuse. |
| `data_boundary` | Configurable per Project to constrain artifact and KV storage to specific buckets, regions, or volumes. |

Cross-tenant scoping is contained within the deployment because the deployment is the entire surface.

## Endpoints

The customer chooses the base URL. Typical conventions:

| Surface | Example URL |
|---|---|
| Inference (OpenAI-compatible) | `https://kovaserve.acme.internal/v1/chat/completions` |
| Native control plane | `https://kovaserve.acme.internal/v1/ks/...` |
| Dashboard | `https://kovaserve.acme.internal` (or a separate hostname) |
| Events | Pulled through the `v1/ks/events/...` API or consumed directly from NATS by trusted in-cluster consumers. |

The wire shapes, error envelope, pagination, idempotency, cost headers, and SSE streaming conventions are identical to managed cloud — see [API overview](/docs/api/index).

## Operating the deployment

The customer operates the cluster. Day-to-day operations:

| Operation | Path |
|---|---|
| Install | `helm install kovaserve oci://ghcr.io/kovaserve/charts/kovaserve --values customer-values.yaml`. |
| Upgrade | `helm upgrade ...` with the new chart version; Alembic applies any new migrations on pod start. |
| Scale GPU fleet | Add a node to the cluster; the Node Agent and Engine Adapter join automatically. |
| Rotate keys | Through the in-cluster Tenant Management surface or the dashboard. |
| Backup | Standard Postgres / object-storage backup. ContextManifest, Checkpoint, and ContextArtifact records are designed to round-trip through standard backup tooling. |
| Disaster recovery | Restore Postgres + the configured object bucket; replay NATS streams from their durable backing. |

Helm parameters and the full install procedure are documented in the [self-hosting Kubernetes guide](/docs/guides/self-hosting-kubernetes) and the [sovereign deployment guide](/docs/guides/sovereign-deployment).

## Air-gap operation

The chart and the Node Agent images are pullable from a private registry. Once the images and the Helm chart are mirrored into the customer's network, no further outbound connectivity is required for operation. Telemetry export, model downloads, and license validation are configurable and disabled by default in air-gap mode.

## Storage residency

Three categories of bytes leave the GPU and enter durable storage:

| Category | Stored at |
|---|---|
| ContextManifest rows | Customer Postgres. |
| ContextArtifact bytes | Customer object storage, under the bucket configured at the [PolicyScope](/docs/concepts/policy/policy-scopes) `data_boundary`. |
| Checkpoint blobs (logical context + optional KV exports) | Customer object storage. |
| WorkspaceArtifact bytes (managed runtime) | Customer object storage. |
| Events | Customer NATS JetStream. |
| Usage records | Customer Postgres. |

No data leaves the deployment. The dashboard, the API surface, and the event streams all read from the customer's stores.

## Semantic parity with managed cloud

The product principle is that managed and private differ in *operations*, not in *semantic meaning*. A Run, a Checkpoint, a Budget, a PolicyScope, and a [StateHandle](/docs/concepts/recovery/state-handles) mean the same thing across both topologies.

Concretely:

| Surface | Managed cloud | Private deployment |
|---|---|---|
| Run / Session / Step / ModelCall shape | Identical | Identical |
| ContextManifest contract | Identical | Identical |
| Checkpoint resume order | Identical (logical-context-first) | Identical (logical-context-first) |
| Event taxonomy | Identical (`ks.>`, `kv_events`) | Identical |
| Cost per completed run formula | Identical | Identical |
| Tenancy resolution | Identical | Identical |
| Dashboard | Hosted at `app.kovaserve.com` | Hosted in-cluster |
| Datastores | Hosted by KovaServe | Operated by customer |
| GPU fleet | Shared or partner | Customer-operated |

Migration between modes — managed cloud to private, or partner cloud to fully private — preserves IDs, manifests, and checkpoint records. There is no semantic re-mapping step.

## When to choose private deployment

| Choose private deployment when | Choose [managed cloud](/docs/concepts/deployment/managed-cloud) when |
|---|---|
| Data residency or sovereign jurisdiction is required. | Hosted operation is acceptable. |
| Air-gap operation is required. | Internet-reachable APIs are acceptable. |
| Customer-owned Postgres / object storage is required. | Hosted datastores are acceptable. |
| GPU fleets must be operated under customer control. | Shared or partner fleets cover the workload. |
| The deployment is the single boundary — no multi-tenant control plane is wanted. | Multi-tenant logical isolation is sufficient. |
