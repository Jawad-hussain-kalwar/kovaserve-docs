The **KV cache** is the substrate-resident store of attention key/value tensors produced during prompt processing. When an input prefix matches material the model has already attended to, the substrate skips re-prefill on that prefix and starts generation faster. KV cache is the single largest performance lever in long-running AI execution, and the reason KovaServe routes on continuity rather than blind load.

This page covers the substrate-level fundamentals. The control-plane projection — how the [KV Index](/docs/concepts/kv-index) tracks cache state and how [manifest-aware routing](/docs/concepts/manifest-aware-routing) consumes it — lives on dedicated pages.

## What the substrate stores

For each token the model has processed, the substrate retains a per-layer key/value pair. A processed prefix of `n` tokens occupies roughly:

```
bytes ≈ 2 × n_layers × n_kv_heads × head_dim × bytes_per_element × n_tokens
```

The exact size depends on the model; a typical mid-sized model produces tens of megabytes per thousand tokens of cache. Cache lives wherever the substrate places it — HBM by default, demoted to lower tiers under pressure.

## Tiers and warmness

The substrate moves cache across a tier hierarchy as memory pressure dictates:

| Tier | Where | Latency to reuse |
|---|---|---|
| HBM | GPU memory | sub-millisecond |
| RAM | host memory | low milliseconds |
| NVMe | local disk | tens of milliseconds |
| object | S3 / MinIO (export) | hundreds of milliseconds, plus network |

Warmness is the control-plane summary of where a given prefix lives, exposed by Layer 1 to higher layers:

| Warmness | Meaning |
|---|---|
| `hot` | KV resident in HBM. |
| `warm` | KV resident in host RAM. |
| `cool` | KV resident in NVMe. |
| `cold` | KV not resident — would need full prefill. |
| `partial` | Some breakpoints match; later breakpoints do not. |

Warmness is a soft signal — it reflects substrate state at the moment the latest event was emitted. Reuse cost depends on tier and on the substrate's restore latency for that tier.

## Prefix matching

KV cache is only reusable when the *bytes* the model sees match a prior call exactly. Two requests with identical user-visible prompts but different rendered bytes — different role tags, different whitespace normalization, different tokenizer fingerprints — produce different KV state and cannot share cache.

KovaServe's [renderer versions and tokenizer fingerprints](/docs/concepts/renderer-versions) are stamped on every [ContextManifest](/docs/concepts/context-manifests) precisely because they decide whether two manifests have a chance at sharing cache. The router uses them as hard scope filters; mismatched replicas are not scored.

The unit of reuse is a *prefix*. A request whose prompt diverges at token `k` reuses cache up to `k − 1` and pays full prefill from `k` onward. This is why KovaServe partitions manifests at canonical [breakpoints](/docs/concepts/segment-types) (tools / system / history-prefix / summary / active) — earlier breakpoints change less often than later ones, so prefix reuse is highest at the tools and system tiers.

## Lifecycle events

The substrate emits real lifecycle events for every cache mutation. The [`kv_events`](/docs/concepts/events/state) JetStream stream carries them on subjects under `kv.state.*`:

| Event | Meaning |
|---|---|
| `kv.state.stored` | A new prefix was stored at a tier. |
| `kv.state.accessed` | An existing prefix was reused. |
| `kv.state.partial_match` | A request matched a prefix only up to some breakpoint. |
| `kv.state.demoted` | A prefix moved down a tier (HBM → RAM, RAM → NVMe). |
| `kv.state.promoted` | A prefix moved up a tier. |
| `kv.state.transferred` | A prefix was exported to or imported from another node. |
| `kv.state.restored` | A prefix was re-imported during checkpoint restore. |
| `kv.state.evicted` | A prefix was removed from a tier. |
| `kv.state.invalidated` | A prefix was marked unusable (compatibility violation, model reload). |

These events are the **only** input to the control plane's view of cache state. KovaServe does not poll, does not estimate, does not reconstruct cache state from heuristics. The discipline is non-negotiable: routing, tiering, and resume planning rely on real substrate truth or they don't operate.

## Reuse classes

Three classes of reuse drive KovaServe's economics, in increasing depth:

| Class | Identity | Reuse boundary |
|---|---|---|
| Prefix-tier | `stable_prefix_id` (legacy) | A canonical prefix break (typically the system breakpoint). |
| Manifest-tier | `stable_context_id` | The full manifest under canonical ordering — including renderer, tokenizer, scope. |
| Session-affinity | Most-recent-replica from the bound Session | Continuity across a Session's turns. |

Prefix-tier reuse benefits any request that shares the leftmost breakpoints with a prior call. Manifest-tier reuse benefits requests with deeper alignment. Session-affinity reuse layers on top, weighting the Session's most-recently-used replica.

## Multi-tier breakpoints

The five canonical breakpoints in a manifest each produce their own digest. The substrate's KV cache reuses up to the deepest matching breakpoint:

```
[ tools ][ system ][ history-prefix ][ summary ][ active ]
   ▲         ▲           ▲              ▲          ▲
   │         │           │              │          │
   tools-tier digest matches more often than active-tier digest;
   reuse value is highest at the leftmost breakpoints.
```

A request matching `tools` and `system` but diverging at `history-prefix` reuses the first two and re-prefills the rest. The router scores each match independently to pick the candidate with the deepest reuse for the candidate's tier latency.

## Tier movement

The Tiering / State Manager — a Layer 2 controller — observes pressure signals from substrate events and instructs Layer 1 to move state. Movement preserves stable identity: a prefix demoted from HBM to NVMe is the same `stable_prefix_id`; only its tier and warmness change.

Movement is bounded by:

- **Compatibility.** A prefix on a node loaded with an incompatible model cannot be promoted there.
- **Scope.** A prefix warmed under one `policy_scope.context_share_scope` is not transferable to a foreign-scope replica.
- **Capacity.** Tier write quotas are honored to avoid thrashing.

The transfer/restore work itself is owned by the Layer 1 sidecar (Transfer/Restore Worker), not by the control plane.

## Eviction and invalidation

| Cause | Outcome |
|---|---|
| Tier capacity pressure | Eviction from the tier (often demotion to a lower tier first). |
| Model reload | Invalidation — the cache no longer matches the loaded weights. |
| Renderer or tokenizer roll-forward | Invalidation under the old fingerprint; new prefixes accumulate under the new fingerprint. |
| Tenant retention rule | Eviction across tiers as part of standard cleanup. |
| Operator command | Targeted invalidation (e.g. for an incident). |

Each cause emits the corresponding `kv.state.evicted` or `kv.state.invalidated` event so the [KV Index](/docs/concepts/kv-index) can mirror the change.

## Why KV cache is an optimization, not a correctness concern

KovaServe's [resume flow](/docs/concepts/resume-flow) treats KV restore as a latency optimization layered on top of logical-context restore. A resumed Run that fails to restore KV simply re-prefills under the manifest replay path; correctness is identical. This is the practical meaning of the architecture's "logical context first" rule — every system that needs cache (routing, resume, tiering) must function when the cache is missing.

## What the KV cache is not

- **Not the source of truth.** [ContextManifests](/docs/concepts/context-manifests), [ContextSegments](/docs/concepts/context-segments), and [ContextArtifacts](/docs/concepts/context-artifacts) are the source of truth for what the model is thinking about. KV cache is an accelerator.
- **Not portable across models.** A KV prefix is bound to a specific model and tokenizer.
- **Not per-Run.** Cache state is per-replica, per-prefix. Many Runs across many Sessions can share a single hot prefix.
- **Not synthetic.** The control plane never estimates cache state; only real substrate events populate the [KV Index](/docs/concepts/kv-index).
