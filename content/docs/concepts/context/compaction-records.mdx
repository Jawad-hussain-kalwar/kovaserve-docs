A **CompactionRecord** is the lineage object that describes how a set of source [ContextSegments](/docs/concepts/context/context-segments) was reduced into one or more compacted output segments. It records the source manifest, the source segment ids, the output segment ids, the method, the model used (if any), token counts before and after, and quality flags.

CompactionRecords are produced by the **Compaction Lineage Service**. They make compaction reconstructible: every compacted segment in a [ContextManifest](/docs/concepts/context/context-manifests) carries a `compaction_record_id` that resolves to a complete description of how that segment came to exist.

## Why a separate lineage entity

Compaction is a lossy transformation â€” a 10,000-token tool transcript becomes a 400-token summary. Without a lineage record, that loss is irrecoverable: the manifest contains the summary but no trace of what it was summarized from, no way to verify quality, and no way to replay against the original source.

The CompactionRecord is the lineage anchor that makes compaction safe to use as an automatic operation:

- Resume can verify that compaction was applied correctly by checking source-to-output token deltas against recorded values.
- Operators can audit which calls invoked compaction, with what method, against what input.
- Re-compaction with a different method (or a managed summarizer roll-forward) can use the original sources, not the prior summary.
- Quality regressions are debuggable â€” the record carries enough metadata to reproduce the prior compaction pass.

## The two methods

The Compaction Lineage Service supports two production paths under one identical-shape contract. Both are first-class; the lineage record looks the same in both cases. The `method` field on the record names which path produced it:

| `method` | Where the summary text comes from |
|---|---|
| `runtime_reported` | The runtime calls a model itself, produces the summary, and reports the record via `POST /v1/ks/context/compact`. KovaServe records the lineage; the runtime owns the summary content. |
| `kovaserve_managed` | The Compaction Lineage Service runs a managed summarizer (model-specific policies, optional quality scoring) and writes both the output segments and the record in one transaction. |

The lineage contract is identical across both methods. The same downstream consumers â€” resume, observability, re-compaction â€” operate on records produced by either path without distinguishing them. Lineage is always KovaServe's responsibility; summary quality is the runtime's responsibility under `runtime_reported` and a managed enhancement under `kovaserve_managed`.

## Fields

| Field | Purpose |
|---|---|
| `id` | Bare 26-char ULID. |
| `tenant_id`, `project_id` | Scope ownership. Cross-scope reads return 404. |
| `run_id`, `session_id` | The Run and Session the compaction belongs to. |
| `source_manifest_id` | The ContextManifest the source segments belonged to. |
| `source_segment_ids[]` | Ordered ids of the segments that were compacted. JSONB column. |
| `output_segment_ids[]` | Ids of the resulting segments. Usually one; may be more when the source spans distinct topics. JSONB column. |
| `method` | `runtime_reported` or `kovaserve_managed`. The closed enum maintained by the Compaction Lineage Service. |
| `model_id` | The model used to produce the summary. Required for `kovaserve_managed`; optional and informational for `runtime_reported`. |
| `token_count_before` | Sum of source segment `token_count`s under the source manifest's tokenizer fingerprint. |
| `token_count_after` | Sum of output segment `token_count`s. The savings is the difference; it is recorded implicitly. |
| `quality_flags` | JSONB. Method- and runtime-specific structured flags asserting properties the summarizer guaranteed. |
| `metadata` | JSONB. Free-form metadata; conventionally carries `source_artifact_ids` and `output_artifact_id` when source or output segments are artifact-backed, byte counts, and policy refs. |
| `created_at` | Timestamp. |

```http
    "01J8X9Y2Z3K4M5N6P7Q8R9SG10",
    "01J8X9Y2Z3K4M5N6P7Q8R9SG11",
    "01J8X9Y2Z3K4M5N6P7Q8R9SG12"

  "method": "runtime_reported",
  "model_id": "qwen3-coder-30b",
  "output_segments": [
    {
      "context_type": "summary",
      "memory_class": "session",
      "text": "<the summary>",
      "token_count": 412
    }

  "quality_flags": {
    "verbatim_quotes_preserved": true,
    "tool_call_records_preserved": true,
    "decisions_preserved": true
  }

```

The Compaction Lineage Service writes the output segment rows, the CompactionRecord row, and the `compaction_record_id` back-reference in a single transaction. The next manifest assembled by GCM v2 may then drop the source segments and include the output segment instead â€” admission re-runs against the reduced shape.

For `method: kovaserve_managed` the call is internal â€” the managed summarizer invokes the same atomic write path with `method: kovaserve_managed`, the model_id of the summarizer it ran, and `quality_flags` produced by its policy. Runtimes do not call `kovaserve_managed` from the wire; they observe the resulting record like any other.

## Triggers

CompactionRecords are produced when:

| Trigger | `method` | Source |
|---|---|---|
| `ContextBudget` admission returned `COMPACT_REQUIRED` | `runtime_reported` or `kovaserve_managed` | The runtime (or managed summarizer) compacts to fit; the record is registered before re-admission. |
| Runtime-driven proactive summarization | `runtime_reported` | The runtime decides to compact ahead of an admission failure. |
| Managed summarizer policy | `kovaserve_managed` | A scheduled policy compacts long Sessions or large tool outputs without runtime involvement. |

The `COMPACT` [step type](/docs/concepts/run-hierarchy/step-types) is the canonical step under which the runtime performs `runtime_reported` compaction. `kovaserve_managed` compactions are not Steps on the runtime's Run; they are control-plane-internal operations that produce records but do not occupy step slots.

## Retention

CompactionRecords carry their own retention horizon, which dominates their source artifacts and source segments:

- A `with_compaction_lineage`-retention source segment lives until **every** record listing it as a source has been pruned.
- A `with_compaction_lineage`-retention source artifact lives under the same rule.
- The CompactionRecord itself follows tenant retention, typically longer than per-call retention.

This makes lineage replayable across the full retention window: a record from three months ago can still resolve its source segments and source artifacts to verify the compaction or to re-compact under a newer method.

## Quality flags

`quality_flags` is a method- and summarizer-specific JSON object. The Compaction Lineage Service does not enforce semantics on the flag values â€” it stores them. The runtime (under `runtime_reported`) or the managed summarizer policy (under `kovaserve_managed`) decides which flags to assert. KovaServe does not score or guarantee summary quality; it stores what the producer asserted and exposes it for downstream replay, observability, and re-compaction policies.

Common conventions:

| Producer pattern | Typical flags |
|---|---|
| History summary | `verbatim_quotes_preserved`, `tool_call_records_preserved`, `decisions_preserved`, `latest_user_request_preserved`. |
| Tool-output summary | `error_messages_preserved`, `result_schema_preserved`, `numeric_values_preserved`. |
| Truncation with marker | `truncation_marker_inserted`, `head_kept_tokens`, `tail_kept_tokens`. |
| Decision extraction | `decision_count`, `bullet_form`. |

The flags are read by observability, by re-compaction policies, and by the `compaction-lineage` view in dashboards.

## What CompactionRecords are not

- **Not a summarizer.** The Compaction Lineage Service stores lineage. Summary generation is either the runtime's job (`runtime_reported`) or the managed summarizer's job (`kovaserve_managed`); the record is the durable trace of the operation, not the operation itself.
- **Not optional metadata.** Any compacted segment **must** carry `compaction_record_id`. A compacted segment without lineage is rejected at admission.
- **Not chained automatically.** Re-compaction (compacting an already-compacted segment further) produces a fresh record whose sources include the prior output. The chain is explicit; there is no implicit transitive lineage.
- **Not a quality contract.** KovaServe stores `quality_flags` as the producer asserted them. KovaServe does not score summaries, does not enforce flag semantics, and does not guarantee semantic fidelity â€” that responsibility lives with whoever produced the summary.
- **Not free.** Source-segment retention extension under `with_compaction_lineage` increases storage; tenants pay for it through standard metering.
