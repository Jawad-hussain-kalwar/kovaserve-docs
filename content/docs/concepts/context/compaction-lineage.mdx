title: Compaction lineage
description: How source and output segments chain across one or more CompactionRecords — the traversal rules, multi-hop lineage, replay guarantees, and what survives across retention windows.
---

**Compaction lineage** is the graph formed by chaining [CompactionRecords](/docs/concepts/context/compaction-records) through their source and output [ContextSegments](/docs/concepts/context/context-segments). Where a single record describes one compaction event, lineage describes the full ancestry of any compacted segment back to the original raw context — across one hop, ten hops, or as many compactions as a long-running Run accumulates.

Lineage is owned by the **Compaction Lineage Service**. It is a traversal pattern over the record-and-segment graph, with explicit rules about what survives across retention windows and what consumers can reconstruct from the graph.

## The graph

Every compacted segment points back at one CompactionRecord. Every record points at the segments it consumed and produced. Re-compaction of an already-compacted segment forms a chain:

```
Raw segments (T_0):           SG10  SG11  SG12  SG13  SG14
                                │     │     │     │     │
                                ▼     ▼     ▼     ▼     ▼
CompactionRecord CR_a (T_1):  ┌───────────────────────────┐
                              │ sources: SG10..SG14       │
                              │ outputs: SG20             │
                              │ method:  runtime_reported │
                              └─────────────┬─────────────┘
                                            ▼
Compacted (T_1):                          SG20  SG21  SG22
                                            │     │     │
                                            ▼     ▼     ▼
CompactionRecord CR_b (T_2):  ┌───────────────────────────┐
                              │ sources: SG20..SG22       │
                              │ outputs: SG30             │
                              │ method:  kovaserve_managed│
                              └─────────────┬─────────────┘
                                            ▼
Compacted (T_2):                          SG30
```

`SG30` has direct lineage `CR_b` and transitive lineage `CR_b → SG20..22 → CR_a → SG10..14`. The graph terminates at segments whose `compaction_record_id` is null — the original raw segments.

## Traversal

Given any segment, traversal walks `compaction_record_id` references upward until null:

```
walk_up(seg):
  result = []
  while seg.compaction_record_id is not null:
    record = fetch(seg.compaction_record_id)
    result.append(record)
    for src_id in record.source_segment_ids:
      walk_up(fetch(src_id))   # fan-out at each compaction
  return result
```

Branches in the graph are common: a single CompactionRecord typically has multiple source segments, each of which may have its own ancestry. The full lineage of a leaf compacted segment is therefore a tree, not a chain.

The Compaction Lineage Service exposes the traversal directly so consumers do not need to walk the graph segment by segment:

```http
GET /v1/ks/context/compaction-records/01J8X9Y2Z3K4M5N6P7Q8R9CR04?include=lineage
Authorization: Bearer ks_...
```

```json
{
  "id": "01J8X9Y2Z3K4M5N6P7Q8R9CR04",
  "lineage": {
    "depth": 2,
    "ancestors": [
      {
        "id": "01J8X9Y2Z3K4M5N6P7Q8R9CR03",
        "method": "runtime_reported",
        "source_segment_ids": ["..."],
        "output_segment_ids": ["..."]
      }
    ],
    "raw_source_segment_ids": [
      "01J8X9Y2Z3K4M5N6P7Q8R9SG10",
      "01J8X9Y2Z3K4M5N6P7Q8R9SG11",
      "01J8X9Y2Z3K4M5N6P7Q8R9SG12"
    ]
  }
}
```

`depth` is the longest chain to a raw segment. `raw_source_segment_ids` is the set of leaves — segments with no `compaction_record_id`.

## What survives across retention

Lineage replay correctness depends on source segments and source artifacts surviving as long as any record that references them. The retention rules pin this:

| Element | Lives until |
|---|---|
| CompactionRecord row | Tenant retention. |
| Source segment marked `with_compaction_lineage` | Every record listing it as a source has been pruned. |
| Source artifact marked `with_compaction_lineage` | Every record listing it as a source has been pruned. |
| Output segment | Its own retention (`with_call`, `with_session`, etc.). Usually outlives the source segments because output segments are referenced from active manifests. |

A source segment that was originally `with_call` retention but became referenced by a CompactionRecord is **automatically promoted** to `with_compaction_lineage` retention at record creation time. The lineage service mutates the segment's retention field as part of the record-write transaction. This is the only path that mutates a segment row's metadata after creation.

When a record is finally pruned (its tenant retention has elapsed and no still-live record depends on its outputs), the lineage service unsets the back-reference from any source segments whose only retention extension came from this record. Segments whose retention is satisfied by another path retain their rows; segments that lose their last reason are demoted to their original retention class and may be cleaned up by the next sweep.

## Replay

Lineage is replayable when, for some target depth `d`, every record at depth ≤ `d` resolves and every source segment / artifact at depth ≤ `d` resolves. The Compaction Lineage Service exposes this as a precondition check:

```http
POST /v1/ks/context/compaction-records/{id}/replay-check
```

```json
{
  "id": "01J8X9Y2Z3K4M5N6P7Q8R9CR04",
  "replayable": true,
  "max_replayable_depth": 2,
  "missing": []
}
```

When `max_replayable_depth` is less than `depth`, the response lists the missing source ids. This is the same mechanism resume relies on to validate that compacted segments in a frozen [ContextManifest](/docs/concepts/context/context-manifests) snapshot can be re-derived if needed.

## Truncated lineage

When a source segment past `with_compaction_lineage` retention is finally cleaned up, the chain is truncated at that point. Downstream records and segments still resolve as opaque — the output segment is intact, the record row is intact — but `walk_up` terminates at the missing source with `truncated: true` rather than reaching a raw segment.

Truncation is a normal end state for old lineage. It does not break the manifests that reference the leaf compacted segments; those segments are still readable, still tokenizable, still admissible. It only means re-compaction from raw sources is no longer possible past the truncation point.

## Cross-Run, cross-Session lineage

Lineage crosses Run and Session boundaries naturally. A CompactionRecord written under Run R_a may be referenced by an output segment that ends up in a manifest belonging to Run R_b, R_c, or a Run in a different Session — for example, a `tenant`-scoped summary survives across every Session in the tenant.

The graph is scoped at the tenant/project level: a record under one `(tenant, project)` is invisible to others. Within scope, lineage is universally addressable.

## Why lineage matters for resume

The [resume flow](/docs/concepts/recovery/resume-flow) validates that every `compaction_record_id` referenced from a frozen manifest snapshot resolves. If lineage is intact at depth 1, the resumed Run sees the same compacted segments the captured Run saw — full byte equivalence. If lineage is truncated past depth 1, the manifest is still resumable (the leaf compacted segments are intact), but the resumed Run cannot re-compact further from the original sources without first re-creating them.

Resume reconstructs logical context from the manifest's segments and the artifact store; KV cache restore is layered on top when available but is not required for the resumed Run to proceed. Compaction lineage is the addressable history that lets resume re-derive a compacted segment from its sources whenever the chain remains intact.

## Properties of the lineage graph

Every edge in the graph is an explicit `compaction_record_id` or `source_segment_ids` reference, written at the time the compaction was registered. The graph is scoped to `(tenant, project)`, so traversal terminates at scope boundaries with `404`. The `with_compaction_lineage` retention extension keeps source segments and source artifacts alive while any record references them, and that storage is billed through standard metering.

Compaction lineage tracks the relationship between *segments*. [ContextArtifacts](/docs/concepts/context/context-artifacts) carry their own `name + version` history, which is a separate axis: a record may reference an artifact at a specific version, and that version remains addressable for the lifetime of the record.
