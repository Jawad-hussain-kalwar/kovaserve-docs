A **ContextManifest** is the durable, per-[ModelCall](/docs/concepts/run-hierarchy/model-calls) record of the exact context that was dispatched to the model. It captures the segment list, token accounting, stable identities, renderer version, model id, tokenizer fingerprint, scope, and admission decision — everything needed to answer "what was this model call thinking about?" without replaying the original request.

Every successful ModelCall is bound to exactly one ContextManifest via `context_manifest_id` FK. Every [Checkpoint](/docs/concepts/recovery/checkpoints) freezes a manifest snapshot. The [Cache-Aware Router](/docs/concepts/routing/manifest-aware-routing) consults manifest identity when scoring placements. Every [Budget](/docs/concepts/policy/budgets) decision and [ContextBudget](/docs/concepts/policy/context-budgets) admission runs against an assembled manifest. The manifest is the **audit anchor** for the entire context-engineering subsystem.

ContextManifests are produced by the **Global Context Manager (GCM)** and stored in the **Context Manifest Registry**.

## Why a per-call durable record

KovaServe ships the manifest as a first-class entity because every other context-engineering capability depends on having a stable answer to "what was the context?" that survives failures, redeployments, retention windows, and tokenizer migrations:

- Budgets and admission can only be enforced against a concrete shape.
- Resume needs the exact prompt to reconstruct, not a regenerated approximation.
- Routing decisions are only meaningful against an identity that the cache layer can recognize.
- Compaction lineage needs source manifests that do not disappear when memory is recompacted.
- Audit trails need a record that survives long after the inference engine has evicted its working state.

The manifest is the single artifact every other context primitive references.

## What a manifest records

```
ContextManifest
├── id, run_id, session_id
├── step_id?, model_call_id?, checkpoint_id?    ▸ bound after admission;
│                                                  null on HARD_STOP and
│                                                  on checkpoint-only snapshots
├── segments[]                  ▸ ordered ContextSegments (canonical order)
├── stable_prefix_id            ▸ prefix-tier identity for KV cache lookups
├── stable_context_id           ▸ compatibility-aware identity (digest cascade +
│                                 renderer + model family + tokenizer fp +
│                                 tenant / project / context scope)
├── renderer_version            ▸ which context renderer produced this shape
├── model_id, model_family
├── tokenizer_fingerprint       ▸ hash of the tokenizer assets
├── scope                       ▸ tenant / project / context_share_scope
├── token_counts
│   ├── stable_tokens           ▸ reused-prefix tokens
│   ├── volatile_tokens         ▸ likely-to-change tokens
│   ├── active_tokens           ▸ what counts against ContextBudget.active_token_limit
│   └── offloaded_tokens        ▸ moved to ContextArtifact storage
├── byte_counts
│   └── artifact_bytes          ▸ total bytes referenced via artifact_ref_id
├── compaction_records[]        ▸ CompactionRecord refs that produced any segment
├── admission
│   ├── decision                ▸ ALLOW | WARN | OFFLOAD_REQUIRED | ...
│   ├── transformations[]       ▸ what was applied to reach the admitted shape
│   └── timestamp
└── created_at
```

A manifest is written once and immutable thereafter. Every field is set at write time; the optional fields above (`step_id`, `model_call_id`, `checkpoint_id`) reflect the manifest's binding to other entities and are populated when those bindings exist.

## Stable identities

Two identity fields travel on every manifest. Both are pure functions of the manifest's content; they do not change between writes of the same shape.

| Identity | Purpose | Inputs |
|---|---|---|
| `stable_prefix_id` | Prefix-tier identity for KV cache lookups. The cache-aware router consults it to find reusable KV cache prefixes. | Hash of the canonical-ordered stable-prefix segments. |
| `stable_context_id` | Compatibility-aware identity used for manifest-tier reuse, KV index lookups, and cross-call correlation. | Digest cascade across all segments + renderer version + model family + tokenizer fingerprint + scope. |

Two requests that produce the same `stable_context_id` will route to the same warm placement. Two manifests that share `stable_prefix_id` but differ on `stable_context_id` (e.g. same prefix but different downstream segments) share the prefix tier of the cache but fan out at the manifest tier.

## Four storage tiers

GCM groups segments into **four prefix-tier buckets** on the `context_manifest_segments` join row, so different cache layers can find their own reuse boundary on the same data structure:

```
tier 0       tier 1        tier 2              tier 3            volatile tail
[ tools ][ system ][ history_prefix ][ summary ][ agent_state + tool_result ]
   ▲         ▲          ▲                ▲           ▲
   │         │          │                │           │
   tool      system     verbatim turn    bounded     fresh per-step state
   defs      prompt     history          summary     and latest tool output
```

Tier 0 carries [`persistent_context`](/docs/concepts/context/segment-types) segments whose content is tool definitions; tier 1 carries `system_prompt`; tier 2 carries `session_history`; tier 3 carries `summary` segments. The volatile tail (`agent_state`, `tool_result`) is appended after the four-tier prefix and is recomputed every call. Prefix-tier routing reuses the leftmost tiers; Anthropic-style prompt cache hints reuse tiers further right; manifest-tier reuse spans the full structure. Every tier produces its own digest, recorded on the manifest for downstream consumers.

## Segment ordering

Segments inside a manifest follow **canonical ordering**. The order is deterministic and renderer-invariant for the same input set, so two assemblies that consume the same inputs produce the same `stable_context_id`. The six canonical positions are:

1. `persistent_context` — long-lived material (tool definitions, RAG documents, [StateHandle](/docs/concepts/recovery/state-handles) references). Rendered first so the most-stable bytes anchor the cache prefix.
2. `system_prompt` — the runtime-supplied system prompt.
3. `session_history` — verbatim turn history back to the most recent compaction point.
4. `summary` — compaction outputs that replace older history; rendered after the verbatim history so its mutation on each compaction event invalidates only the summary tier, not the cache prefix above it.
5. `agent_state` — the runtime's per-step scratchpad, planner output, or current draft.
6. `tool_result` — the latest tool result that has not yet been summarized.

Runtimes provide *content*; GCM owns *position*. Runtimes do not influence the canonical position assigned to a segment. See [segment types](/docs/concepts/context/segment-types) for the closed enum and how the six values map onto the four storage tiers.

## Admission decision

Every manifest carries the admission outcome from the [ContextBudget](/docs/concepts/policy/context-budgets) admission service. The decision is one of:

| Decision | Meaning | Recorded transformations |
|---|---|---|
| `ALLOW` | Proceeded directly. | None. |
| `WARN` | Proceeded with a warning. | None. |
| `OFFLOAD_REQUIRED` | Re-admitted after artifact offload. | List of segments moved to artifact references. |
| `COMPACT_REQUIRED` | Re-admitted after compaction. | Reference to the CompactionRecord that reduced the manifest. |
| `DEGRADE` | Re-admitted under degraded shape. | Reduced `max_tokens`, dropped optional segments. |
| `HARD_STOP` | Rejected. The manifest is recorded but no ModelCall references it. |

The recorded transformations make the path from "raw assembly" to "admitted shape" reconstructible after the fact.

## Boundaries with the runtime

The runtime supplies content, retrieval strategy, and prompt intent on the request envelope; GCM normalizes shape, computes identity, persists the manifest, and records the admission outcome. Re-admission produces a *new* manifest with its own id and the prior manifest stays addressable, so timelines, checkpoints, and audit trails always resolve to the exact shape that was admitted.

The manifest stores segment metadata and identities; large segment bodies live in [ContextArtifacts](/docs/concepts/context/context-artifacts) and inline segment bodies live on the [ContextSegment](/docs/concepts/context/context-segments) row. Reading a manifest yields the structure; resolving its segments yields the bytes.
