The `context_type` field on every [ContextSegment](/docs/concepts/context/context-segments) names the segment's **canonical position** in a [ContextManifest](/docs/concepts/context/context-manifests). The set is closed and defined by ADR-003 — there are exactly six values, in a fixed canonical order. The order is deterministic and renderer-invariant: two assemblies that consume the same input set produce the same segment ordering, which is what makes the manifest's `stable_context_id` stable.

Runtimes provide *content*; GCM owns *position*. Runtimes do not influence the canonical position assigned to a segment.

## The six canonical types

```
Position 1: system_prompt        ▸ never changes (model instructions)
Position 2: persistent_context   ▸ rarely changes (RAG docs, tool definitions, activation handles)
Position 3: summary              ▸ changes on compaction (compacted older history)
Position 4: session_history      ▸ grows each turn (recent conversation)
Position 5: agent_state          ▸ changes each agent step (scratchpad, tool plan)
Position 6: tool_result          ▸ changes every request (latest tool output)
```

| `context_type` | Carries | Volatility | `memory_class` (typical) |
|---|---|---|---|
| `system_prompt` | The system prompt the runtime supplied. | Stable. | `tenant` or `project`. |
| `persistent_context` | Runtime-supplied long-lived material — RAG documents, tool definitions, persistent instructions, [StateHandle](/docs/concepts/recovery/state-handles) references. | Stable. | `tenant`, `project`, or `global`. |
| `summary` | Output of compaction. Replaces a span of older `session_history` or `tool_result` segments. Always carries a `compaction_record_id`. | Changes on compaction events. | `session`. |
| `session_history` | Verbatim turn history from the bound [Session](/docs/concepts/continuity/sessions), back to the most recent compaction point. | Grows each turn. | `session`. |
| `agent_state` | The runtime's per-step state — scratchpad, planner output, tool plan, current draft. | Changes each agent step. | `ephemeral` or `session`. |
| `tool_result` | A tool result that arrived since the last summarization point and is still inline. Large results are offloaded to a [ContextArtifact](/docs/concepts/context/context-artifacts) and the segment carries a preview. | Changes every request. | `session`. |

## Stable prefix versus volatile tail

```
[ stable prefix ─────────────────── ][ volatile tail ──────────────────────────── ]
  system_prompt  persistent_context    summary  session_history  agent_state  tool_result
```

Positions 1–2 form the **stable prefix** — if those segments have not changed, the substrate's KV cache is reusable. The legacy `stable_prefix_id` is computed over exactly this range (SHA-256 of the rendered token sequence for `system_prompt + persistent_context`). Positions 3–6 are **volatile** — expected to change call-to-call and appended after the cached prefix.

This is why `summary` sits at position 3, not folded into `persistent_context`: a compaction event changes the summary, and including it in the stable prefix would invalidate KV cache on every compression.

## The four storage tiers

GCM v2's manifest table groups the six `context_type` values into **four prefix-tier buckets** so the [Cache-Aware Router](/docs/concepts/routing/manifest-aware-routing) and Anthropic-style prompt cache hints can find their reuse boundary on the same data structure:

| Tier ordinal | Tier name | `context_type` values it carries |
|---|---|---|
| `0` | `tools` | `persistent_context` segments whose content carries tool definitions. |
| `1` | `system` | `system_prompt`. |
| `2` | `history_prefix` | `session_history`. |
| `3` | `summary` | `summary`. |

The tier ordinal lives on the `context_manifest_segments` join row alongside the segment's `ordinal`. The volatile tail (`agent_state`, `tool_result`) is not bucketed into a stable tier — those segments are appended after the four-tier prefix and recomputed every call.

## Position is canonical, not chosen

GCM v2 assigns `context_type` based on input shape:

- A `messages` entry with `role: system` becomes `system_prompt`.
- A runtime-supplied retrieval block, tool-definitions block, or [StateHandle](/docs/concepts/recovery/state-handles) reference becomes `persistent_context`.
- A compaction output becomes `summary`.
- A turn fetched from the [Session](/docs/concepts/continuity/sessions) Registry becomes `session_history`.
- The runtime's per-step scratchpad, planner output, or current draft becomes `agent_state`.
- A `messages` entry with `role: tool` becomes `tool_result`.

Runtimes do not pass `context_type` directly; it is derived. This is what keeps the manifest's identity stable across runtimes that supply input in different shapes — every reasonable shape resolves to the same canonical typing.

## Ordering rules

Within a `context_type` the order is also deterministic:

- `system_prompt` — single segment per manifest.
- `persistent_context` — multiple segments allowed; ordered by the runtime's supplied order, which GCM preserves as long as digests match across calls.
- `summary` — chronological, oldest first. Each carries `compaction_record_id`.
- `session_history` — turn order ascending. A long Session's history is broken into segments at turn boundaries so compaction can target precise spans.
- `agent_state` — runtime-supplied order; typically a single segment per call.
- `tool_result` — turn order ascending.

The order is enforced by GCM v2's serialization step, not by clients. Clients that supply inputs out of order have them re-ordered on their behalf.

## Mapping to wire `parts`

On the wire, segments serialize as `parts` (cross-vendor convergence — see [Context segments](/docs/concepts/context/context-segments)). The mapping to `parts.type` follows standard Anthropic / Gemini / Vercel AI SDK shapes:

| `context_type` | `parts.type` |
|---|---|
| `system_prompt`, `persistent_context`, `summary`, `session_history`, `agent_state` | `text` |
| `tool_result` | `tool_result` |

A wire `parts` array can be re-ingested into segments by reading `context_type` (when carried explicitly) or by inferring from `parts.type` plus position.

## Why the set is closed

A closed `context_type` enum is what makes manifest identity reproducible. If runtimes could invent local types, two semantically identical assemblies would produce different `stable_context_id`s, which would defeat manifest-tier reuse and prevent cross-runtime cache sharing. New types are added through ADR-003 amendment, never through ad-hoc runtime extension.

## What `context_type` is not

- **Not the wire type.** `parts.type` (`text`, `tool_use`, `tool_result`) is the wire shape; `context_type` is the canonical position. They are independent.
- **Not memory class.** [`memory_class`](/docs/concepts/context/context-segments) (`ephemeral` / `session` / `user` / `tenant` / `global`) is a separate axis carrying retention intent. A `system_prompt` segment can be `tenant` class; a `session_history` segment is `session` class; the two fields move independently.
- **Not the storage tier ordinal.** `context_type` is the typed position; the four-tier ordinal on `context_manifest_segments` is a bucketing of those types for prefix-cache reuse.
- **Not a free-form tag.** The enum is closed.
- **Not assigned by the runtime.** GCM v2 derives `context_type` from input shape; runtimes do not set it directly.
