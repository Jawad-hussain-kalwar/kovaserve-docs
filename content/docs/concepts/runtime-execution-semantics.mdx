The KovaServe runtime drives a [Run](/docs/concepts/runs) forward through **typed step progression** rather than an opaque internal loop. Every action a [Worker](/docs/concepts/workers) takes is recorded as a [Step](/docs/concepts/steps) of a known type from the canonical 19-value `StepType` enum, attached to the canonical [ContextManifest](/docs/concepts/context-manifests) for any model calls it issues, and mirrored to the [event taxonomy](/docs/concepts/events) so downstream consumers see the same progression the timeline does.

This page describes the *semantics* of that loop — what the runtime guarantees, what each phase does, and what the contract is between Worker, sandbox, control plane, and substrate.

```
            ┌────────────── canonical Run state machine (Layer 2) ──────────────┐
Run.status: │ CREATED ─▶ READY ─▶ RUNNING ─▶ ... ─▶ COMPLETED | FAILED | ...   │
            └───────────────────────────────────────────────────────────────────┘
                              ▲                  ▲
                              │                  │ records ModelCalls,
                              │                  │ Steps, events
                              │                  │
                       ┌──────┴──────┐    ┌──────┴──────┐
                       │  Worker     │    │  Sandbox    │
                       │  (lease)    │◀──▶│  (container)│
                       └──────┬──────┘    └─────────────┘
                              │
                              ▼
                       Runtime-local state
                       (planner scratchpad,
                        process handles,
                        last safe checkpoint)
```

## The progression model

Every Run is a sequence of Steps, and every Step has a type drawn from the [canonical 19-value enum](/docs/concepts/step-types) maintained in the control plane's `schemas/execution.py`. The runtime uses that enum directly — it does not invent a parallel one.

| Group | Step types | Who emits |
|---|---|---|
| Standard agent loop | `MODEL_CALL`, `TOOL_PLAN`, `TOOL_EXECUTE` | Every runtime |
| Workspace and process | `FILE_READ`, `FILE_WRITE`, `PROCESS_START` | KovaServe runtime in sandbox-bound modes |
| Reflection markers | `OBSERVE`, `THINK` | Optional, runtime's choice |
| Durability and lineage | `CHECKPOINT`, `PAUSE`, `RESUME`, `BRANCH` | Runtime + control plane |
| Terminal | `COMPLETE`, `FAIL` | Runtime |
| Context engineering | `COMPACT`, `MEMORY_READ`, `MEMORY_WRITE` | Runtime, when participating |
| Sub-run lineage | `SUB_RUN_DISPATCH`, `SUB_RUN_JOIN` | Runtime, on dispatch and join |

Validation is application-level — Step type is a `TEXT` column with an enum check — so new types extend the canonical list without a DDL migration. Step types are an enumeration distinct from the [event taxonomy](/docs/concepts/events); a runtime *event* like `ks.process.waiting` does not require its own Step type.

## The Worker loop

A Worker that owns a lease drives the Run forward through a deterministic sequence of phases. Each phase corresponds to canonical operations and emits the matching events.

### 1. Claim

The Worker requests a lease from the [Worker Manager](/docs/concepts/workers). The lease binds Worker → Run for a bounded time window. Lease renewal is the Worker's responsibility while it is making progress; lease loss triggers reassignment without data loss because all resume-relevant state lives in the [runtime state store](/docs/concepts/runtime-local-state), not Worker memory.

### 2. Materialize

The Worker reads the Run's [runtime-local state](/docs/concepts/runtime-local-state) — last safe checkpoint, planner scratchpad, active process handles, tool loop position, branch pointer — and prepares its in-memory view. If the Run was created fresh, the materialized view is empty; if the Worker is picking up after lease loss or a restore, the materialized view is reconstructed from the persisted state.

### 3. Step

The Worker decides the next Step type and dispatches the corresponding action:

| Phase | Action |
|---|---|
| `MODEL_CALL` | Build the prompt, dispatch through `POST /v1/chat/completions` with the canonical `kovaserve` extension envelope. The control plane assembles the [ContextManifest](/docs/concepts/context-manifests), runs [admission](/docs/concepts/admission), routes through the [Cache-Aware Router](/docs/concepts/manifest-aware-routing), executes on the substrate, and records the [ModelCall](/docs/concepts/model-calls). The Worker receives the response and the cost breakdown. |
| `TOOL_PLAN` | A `MODEL_CALL` whose response is structured tool selection. Often paired with a child `TOOL_EXECUTE` Step. |
| `TOOL_EXECUTE` | Dispatch the chosen [tool invocation](/docs/concepts/tool-invocations) through the Tool Gateway. Tool definitions are bound to a [ToolCatalog](/docs/concepts/tool-catalogs); tool outputs flow through [Tool Context Governance](/docs/concepts/tool-context-disposition) and may land in a [ContextArtifact](/docs/concepts/context-artifacts). |
| `FILE_READ` / `FILE_WRITE` | File operations against the [workspace](/docs/concepts/workspaces). Subject to workspace read/write policy; diffs surface as [WorkspaceArtifacts](/docs/concepts/workspace-artifacts). |
| `PROCESS_START` | Process invocation in the [sandbox](/docs/concepts/sandboxes). Stdout/stderr are captured into WorkspaceArtifacts; the process handle lives in runtime-local state. |
| `OBSERVE` / `THINK` | Optional progression markers. Zero ModelCalls. Make the timeline legible. |
| `MEMORY_READ` / `MEMORY_WRITE` | [Memory store](/docs/concepts/memory-store) operations under the chosen [MemoryScope](/docs/concepts/memory-scopes). |
| `COMPACT` | Compaction operation that produces a [CompactionRecord](/docs/concepts/compaction-records) — runtime-produced or KovaServe-managed. |
| `SUB_RUN_DISPATCH` / `SUB_RUN_JOIN` | [Sub-run](/docs/concepts/sub-runs) lineage Steps. |

### 4. Persist

Before considering the Step complete, the Worker writes runtime-local state — updated planner scratchpad, updated tool loop position, updated process handles, updated branch pointer. This is the durability invariant: a Worker may crash or lose its lease at any point after Step persistence, and the next Worker resumes from a consistent view.

### 5. Trigger checkpoint (conditional)

The [Checkpoint Integration Module](/docs/concepts/runtime-architecture#checkpoint-integration-module) evaluates whether the Step's completion crosses a checkpoint trigger threshold:

- before risky file mutations (`FILE_WRITE` against protected paths)
- before unsafe shell actions (`PROCESS_START` of `rm`, `git push`, network calls without policy)
- after stable analysis phases (a `THINK` Step that completes a planning block)
- on budget warning (the Layer 2 [Budget](/docs/concepts/budgets) emitted `ks.budget.warning`)
- on schedule (periodic, configured per agent)
- on explicit runtime request

When triggered, the Worker emits a `CHECKPOINT` Step. The actual capture goes through the canonical [Checkpoint Manager](/docs/concepts/checkpoints) — a `runtime_state` [StateHandle](/docs/concepts/state-handles) is added to the canonical Checkpoint record.

### 6. Renew or release

The Worker either renews the lease and continues to the next Step, or releases the lease (Run completed, paused, branched, or escalated). Lease release is graceful — the next Step's persisted state is the resume target.

## Idempotency and retry

Only safe and idempotent runtime steps retry in place. The default rules:

| Step type | Retry policy |
|---|---|
| `MODEL_CALL` | Retried automatically on transient substrate failures (timeouts, transient routing failures). The same [ContextManifest](/docs/concepts/context-manifests) is reused; idempotency keys at the [API level](/docs/api/idempotency) prevent double-billing on retried success. |
| `TOOL_EXECUTE` against an idempotent tool | Retried in place. The runtime marks tools as idempotent at registration time. |
| `TOOL_EXECUTE` against a non-idempotent tool | Not retried in place. The runtime escalates to a [resume](/docs/concepts/resume-flow) from the previous Checkpoint or fails the Run. |
| `FILE_WRITE` | Not retried in place. Resumes from the last Checkpoint that preceded the write. |
| `PROCESS_START` | Not retried in place. The process is considered observed; the runtime resumes from the previous Step. |
| `MEMORY_WRITE`, `COMPACT` | Idempotent — keyed by `(scope, namespace, key)` for memory and `(source_manifest, source_segment_ids)` for compaction. |

See [retries](/docs/concepts/retries) for the full retry contract and [resume flow](/docs/concepts/resume-flow) for the alternative path when in-place retry is unsafe.

## Pause, resume, and branch

These are explicit operations the runtime opts into; they go through canonical state transitions.

| Operation | Effect |
|---|---|
| Pause | Run transitions to `PAUSED`. Lease is released; runtime-local state is durable. The Run can be resumed at the same Step or from a Checkpoint. |
| Resume | Run transitions through `RECOVERING → RESUMING → RUNNING` (the [resume flow](/docs/concepts/resume-flow)). The Worker materializes runtime-local state, validates the [ContextManifest](/docs/concepts/context-manifests), and continues. |
| Branch | A `BRANCH` Step in the parent Run produces a new child Run lineage from a Checkpoint or stable point with bounded inherited budgets and policies. See [branch flow](/docs/concepts/branch-flow). |

## Sandbox boundary

Workers in modes that use a [sandbox](/docs/concepts/sandboxes) execute file and process steps inside it — never on the Worker host. The sandbox enforces:

- workspace mounting (read/write policy)
- file and network access policy
- output capture (stdout, stderr, written files)
- resource accounting (cpu, memory, time, disk)
- optional persistence across Steps within the same Run
- branchable snapshots

The Worker treats the sandbox as the execution surface for `FILE_READ`, `FILE_WRITE`, `PROCESS_START`, and any tool that touches the workspace. Tool invocations whose tool class is purely outbound (HTTP/API) may run from the Worker side under the same governance — but the default for a sandbox-bound agent is sandbox-side execution.

## Layer 2 round-trips

Every Step that interacts with the substrate or with canonical context state goes through Layer 2 — never around it.

| Action | Layer 2 surface |
|---|---|
| Issue a model call | `POST /v1/chat/completions` with `kovaserve` envelope |
| Capture a checkpoint | [Checkpoint Manager](/docs/concepts/checkpoints) |
| Read or write memory | [Memory Store](/docs/concepts/memory-store) |
| Register a tool catalog | [ToolCatalog Registry](/docs/concepts/tool-catalogs) |
| Record a CompactionRecord | [Compaction Lineage Service](/docs/concepts/compaction-records) |
| Offload a tool result or file | [ContextArtifact Store](/docs/concepts/context-artifacts) |
| Resolve a State Handle | [State Handle Registry](/docs/concepts/state-handles) |

This is the property that makes the runtime composable. Anything a [BYO runtime](/docs/concepts/byo-runtime) sees through these APIs, the KovaServe runtime sees through them too — same contract, same guarantees.

## Failure classification

A Run that fails ends in `FAILED`. The runtime classifies the failure on the terminal Step's metadata so the [timeline](/docs/concepts/runtime-observability) can present it without re-deriving from logs:

| Class | Cause |
|---|---|
| `BUDGET_HARD_STOP` | Run-level [Budget](/docs/concepts/budgets) hit `HARD_STOP`. |
| `CONTEXT_BUDGET_HARD_STOP` | [ContextBudget](/docs/concepts/context-budgets) admission returned `HARD_STOP`. |
| `POLICY_DENY` | [PolicyScope](/docs/concepts/policy-scopes) denied an action that had no degraded variant. |
| `TOOL_BLOCKED` | [Tool Context Governance](/docs/concepts/tool-context-disposition) blocked a tool definition or output. |
| `SUBSTRATE_FAILURE` | Substrate execution failed and exhausted retries. |
| `SANDBOX_FAILURE` | Sandbox crashed or violated resource limits. |
| `RUNTIME_ERROR` | Worker hit an unhandled error. |
| `CANCELED` | Operator or runtime canceled the Run. |
