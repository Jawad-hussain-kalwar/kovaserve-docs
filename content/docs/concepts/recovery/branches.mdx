A **Branch** is a new [Run](/docs/concepts/run-hierarchy/runs) created from a [Checkpoint](/docs/concepts/recovery/checkpoints) or a stable point on an existing Run. Where a [resume](/docs/concepts/recovery/resume-flow) replays the original Run forward as if the failure had not happened, a branch *forks* — the parent Run continues to exist (or remains terminal), and a new child Run carries on along an independent trajectory.

Branching is how KovaServe represents speculative paths, parallel hypotheses, and what-if exploration without contaminating the parent's audit trail. It is also the mechanism behind retry-from-known-good when the preferred re-entry point is a Checkpoint deeper than the most recent failure.

## When to branch

| Situation | Branch or resume? |
|---|---|
| The original Run failed; you want to continue forward exactly as it would have | Resume. |
| The original Run is healthy; you want a parallel exploration | Branch. |
| The original Run failed; you want to try a different approach from a known-good Checkpoint | Branch. |
| You want a sandbox to test a tool change without affecting the live Run | Branch. |
| You want the parent's full audit trail visible alongside the new trajectory | Branch — the parent is preserved untouched. |

Resume reconstructs; branch forks. The two are not interchangeable.

## `BRANCHING` on the Run FSM

The parent Run transitions through `BRANCHING` on the [Run state machine](/docs/concepts/run-hierarchy/run-state-machine) for the duration of the fork:

```
Parent Run:   ... ─▶ RUNNING ─▶ BRANCHING ─▶ RUNNING       (parent continues)
                              │
                              ├──▶ COMPLETED                (parent was already terminal at branch time)
                              ├──▶ FAILED
                              └──▶ CANCELED

Child Run:    CREATED ─▶ READY ─▶ RECOVERING ─▶ RESUMING ─▶ RUNNING
```

`BRANCHING` is the source-side state. The window is short — it covers only the metadata work to create the child Run row, allocate its budget, and validate the source Checkpoint. Substrate execution remains owned by the parent (if the parent is still running) and the child (after it reaches `RUNNING`).

If the parent Run is already terminal when the branch is requested, the FSM transition is recorded against the new child Run's lineage rather than the frozen parent. The parent row is not mutated.

## Lineage

```
Parent Run R_a
└── Checkpoint C (anchored to R_a, validated)
    │
    ├── Resume:   creates Run R_b with restored_from = C, branch_from = null
    └── Branch:   creates Run R_c with restored_from = C, branch_from = R_a
```

A branched child carries two distinct references:

| Field | Meaning |
|---|---|
| `restored_from` | The [Checkpoint](/docs/concepts/recovery/checkpoints) the child resumes from. Always populated on a branch. |
| `branch_from` | The parent [Run](/docs/concepts/run-hierarchy/runs) the child forked from. Populated only on branches; `null` on a plain resume. |
| `parent_run_id` | The [sub-run](/docs/concepts/run-hierarchy/sub-runs) parent, if any. Distinct from `branch_from` — branch lineage and sub-run lineage are independent and may both be set on the same Run. |

`branch_from` is preserved across the child's lifetime. It travels into events tagged to the child Run, into observability traces, and into the timeline UI so operators can navigate from any point in the branch tree back to its origin.

## Inherited but bounded budgets and policy

A branched child does **not** inherit the parent's running totals. It inherits the parent's *configuration* — Budget shape, ContextBudget dimensions, PolicyScope — but its consumption counters reset.

| Element | Inheritance behavior |
|---|---|
| [Budget](/docs/concepts/policy/budgets) shape (cost / token / step / time / tool limits) | Inherited as defaults; the branch request may override any limit downward. |
| Budget consumed-to-date | Reset to zero. The child gets its own fresh budget. |
| [ContextBudget](/docs/concepts/policy/context-budgets) dimensions | Inherited; overridable downward. |
| [PolicyScope](/docs/concepts/policy/policy-scopes) | Inherited unchanged. A branch cannot escape its parent's scope. |
| [ToolCatalog](/docs/concepts/tools/tool-catalogs) reference | Inherited unchanged. |
| [Session](/docs/concepts/continuity/sessions) binding | Inherited by default; may be overridden to bind the child to a fresh Session. |
| [`current_manifest_id`](/docs/concepts/continuity/current-manifest) on the bound Session | Set to the source Checkpoint's frozen manifest at branch time, identical to the resume case. |

Budgets cannot be *raised* on a branch. The intent is to scope speculative work, not to escape limits.

## Branch trees

Branches can be nested — a branched Run is itself branchable. The lineage forms a tree:

```
R_a (root) ──▶ R_b (resume)
            ├──▶ R_c (branch from C_1)
            │     └──▶ R_e (branch from C_2 on R_c)
            └──▶ R_d (branch from C_1)
```

Each node carries `branch_from` pointing at its immediate parent. The tree is reconstructable by recursive lookup. There is no aggregate "branch group" entity — the lineage is implicit in the per-Run pointers.

## Concurrent branches

Multiple branches from the same Checkpoint are permitted and run independently. Each branch holds its own `RESTORING` reference on the source Checkpoint; the Checkpoint moves to `RESTORED` only when every concurrent branch has reached `RUNNING`. See the [Checkpoint state machine](/docs/concepts/recovery/checkpoint-state-machine) for ref-counting details.

Branches sharing a [Session](/docs/concepts/continuity/sessions) are subject to the same rules as any concurrent Runs against a shared Session — the most recently bound Run becomes the Session's `current_run_id`, and ordering of independent branches is the runtime's responsibility.

## What branches are not

- **Not a clone of the parent's row.** The child has a fresh `run_id`, fresh timestamps, fresh execution events. It references the parent through `branch_from`; it does not duplicate it.
- **Not a substitute for [retries](/docs/concepts/recovery/retries).** A retry adds a new Step within the same Run for transient failures. A branch creates a new Run for divergent trajectories.
- **Not a substitute for [sub-runs](/docs/concepts/run-hierarchy/sub-runs).** A sub-run is a child Run dispatched by the parent's logic during execution and joined via summary handoff. A branch is an out-of-band fork from a Checkpoint. The two coexist; a branched Run can dispatch sub-runs, and a sub-run can be branched.
- **Not free.** Each branch creates its own budget consumption and its own observability surface. The control plane does not silently coalesce branches that converge.
