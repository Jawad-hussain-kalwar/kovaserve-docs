**Cost per completed run** is the unit of value for managed agents on KovaServe. It answers the question that matters in production: *what did it cost to finish this piece of work?* — not what a token cost, not what a model call cost, not what a GPU-second cost in isolation.

KovaServe is built so this number is always available, always attributable to a single managed run, and always defended against the runaway costs that happen when an agent loops, retries, or stalls without a control plane underneath it.

## Why the run is the unit

Agents do work over time. A single business task — fix a failing test, review a contract, investigate an alert — spans many model calls, many tool actions, pauses, recoveries, and decisions. A token-level metric tells you how many tokens flowed; it does not tell you whether the task got done, what the path looked like, or whether the same task next month will cost more or less.

Cost per completed run aligns the economic metric with the business outcome. The run starts with an objective; the run ends with a result; the cost is what the platform charged to bridge them. Everything else — token throughput, GPU utilization, cache hit rate — is an internal lever for *moving* this number, not a substitute for it.

## What goes into the number

A completed run rolls up every model call and every tool action it took:

```
cost_per_completed_run = SUM(
    model_call_cost              over all ModelCalls in the run
  + tool_invocation_cost         over all tool actions in the run
  + workspace_cost               for runtime-managed workspaces
)
                       − SUM(
    cache_savings                from KV cache reuse across the run
  + offloaded_segment_savings    from artifact offload, when priced
  + compaction_savings           from summary-driven context shrinkage
)
```

Every model call's slice is computed against frozen pricing — see [pricing snapshots](/docs/concepts/pricing-snapshots) — so the number is reproducible and auditable. Cache savings are real: KovaServe measures `cached_tokens` from the substrate, not an estimate, and the saving lands on the run total the same way the gross cost does.

The full breakdown is exposed on the run record and queryable at any time:

```json
{
  "id": "01J8X9Y2Z3K4M5N6P7Q8R9S0T1",
  "status": "COMPLETED",
  "cost": {
    "cost_cents": "1.4280",
    "gross_cost_cents": "2.0140",
    "cache_savings_cents": "0.5860",
    "cost_status": "ok"
  },
  "outcome": "SUCCEEDED"
}
```

The `cost_cents` field is the cost per completed run for this run.

## What KovaServe does to defend it

The metric is only useful if the platform actively keeps runs from going out of bounds. Cost per completed run is defended by the same primitives that make managed agents production-ready:

**Budgets** are active control objects. Every managed run carries a budget that evaluates before each step admission. A run that drifts above its threshold can be warned, degraded into a cheaper variant, or hard-stopped before the next model call — the budget is not a passive billing record.

**Context budgets** govern the working context size before GPU dispatch. A run whose context is about to balloon hits an admission decision that compacts, offloads, or rejects — the cost of an unbounded prompt is paid in the budget, not at the invoice.

**Cache reuse** is built into placement. Runs are routed to substrate replicas that have already paid prefill on the relevant prefix; the saving lands on the run as `cached_tokens × discount` and is reflected in `cache_savings_cents`. A long-running coding agent that loops over the same repository pays prefill once and reuses for the rest of the run.

**Artifact offload** keeps large tool outputs out of context. Tool outputs that would inflate every subsequent model call land in object storage and enter context as bounded references. The cost of the next call is computed against the reference, not the inlined payload.

**Compaction lineage** records when summarized segments replace raw transcripts, so the run reconstructs the same decisions on a smaller context. The cost shrinks; the work proceeds.

**Checkpoint recovery** prevents restart-from-zero billing. A failed run that recovers from a checkpoint pays for the new work; it does not pay again for everything before the checkpoint. The savings is in *not* being billed for work the platform already completed.

The combination is the product. Each primitive is documented elsewhere in these docs; the line they all draw is cost per completed run.

## How customers see it

The number surfaces at three layers:

| Layer | What it answers |
|---|---|
| Per-run | What did this specific run cost? |
| Per-managed-agent SKU | What does a typical CodeRuns / OpsRuns / DocRuns task cost? |
| Per-deployment | What do completed runs cost across all my managed agents this month? |

A neocloud, sovereign AI platform, or enterprise running KovaServe sees the same metric across every one of these layers because every layer rolls up the same underlying `usage_records` cost values. The aggregation is exact — KovaServe's rounding contract preserves the identity `gross = net + savings` across rows and across scopes, so a run total, a project total, and a tenant total never disagree by a cent.

## How it relates to the other economic metrics

Cost per completed run is the headline. Two adjacent metrics show the levers:

- **Cache reuse value** — the per-run cache savings, broken out so the team can see which workloads benefit most from KV reuse, manifest reuse, and stable-prefix routing.
- **Run completion rate** — the share of runs that finished successfully versus failed, canceled, or hard-stopped. A budget that prevents runaway cost is only the right answer if the work still completes.

Together they answer two questions that a token-level dashboard cannot: *did the agent do its job*, and *what did it cost to do it*.

## What cost per completed run is not

- **Not a token rate.** Tokens are the input; runs are the output. KovaServe shows both, but only the run-level number tracks the business outcome.
- **Not a GPU utilization metric.** GPU efficiency is one of the levers that moves cost per completed run downward; it is not the metric itself.
- **Not a SaaS subscription price.** Customers running KovaServe on private or neocloud infrastructure see the run cost computed from real consumption against frozen pricing — not a flat monthly fee.
- **Not estimated.** Every component — token counts, cache savings, pricing — is real and snapshotted on the row.
- **Not retroactively re-priced.** The cost a run shipped with is the cost the run keeps. New rates apply to new runs.
