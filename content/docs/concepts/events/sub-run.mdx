The **sub-run events** stream covers the lineage of [sub-runs](/docs/concepts/run-hierarchy/sub-runs) — child Runs dispatched by a parent Run with bounded budget and an inherited [PolicyScope](/docs/concepts/policy/policy-scopes). Sub-run events are emitted from both the parent's and the child's Run paths and tie them together via `parent_run_id` on the envelope.

Subjects: `ks.subrun.*`. Persisted in the `execution_events` JetStream stream. Consumed via [`GET /v1/ks/events/sub-run`](/docs/api/events/sub-run).

## Event types

| Event | Trigger | Emitted by |
|---|---|---|
| `ks.subrun.dispatched` | Parent dispatches a child via `POST /v1/ks/runs/{id}/subruns`. | Parent context. |
| `ks.subrun.summary_attached` | Child posts its summary via `POST /v1/ks/runs/{child_id}/summary`. | Child context. |
| `ks.subrun.joined` | Parent's next Step assembles a manifest that includes the child's summary segment. | Parent context. |

## Payload shapes

### `ks.subrun.dispatched`

```json
{
  "child_run_id": "01J8X9Y2Z3K4M5N6P7Q8R9CHD1",
  "parent_step_id": "01J8X9Y2Z3K4M5N6P7Q8R9SP15",
  "budget_allocation": {
    "cost_cents": "0.5000",
    "tokens": 100000,
    "step_count": 20,
    "time_seconds": 300
  },
  "inherited_policy_scope_id": "01J8X9Y2Z3K4M5N6P7Q8R9P0K0",
  "inherited_context_scope_id": "01J8X9Y2Z3K4M5N6P7Q8R9CS01",
  "objective_summary": "Investigate failing test in module/handler.py"
}
```

The envelope's `run_id` is the parent. `payload.child_run_id` is the new Run; `parent_step_id` is the Step on the parent that initiated dispatch.

### `ks.subrun.summary_attached`

```json
{
  "child_run_id": "01J8X9Y2Z3K4M5N6P7Q8R9CHD1",
  "summary_artifact_id": "01J8X9Y2Z3K4M5N6P7Q8R9SUM1",
  "summary_token_count": 412,
  "summary_segment_id": "01J8X9Y2Z3K4M5N6P7Q8R9SG09",
  "child_outcome": "SUCCEEDED"
}
```

The envelope's `run_id` is the child Run. `payload.summary_artifact_id` is the [ContextArtifact](/docs/concepts/context/context-artifacts) carrying the full summary; `summary_segment_id` is the ContextSegment that wraps the summary at preview length for inclusion in the parent's manifest.

### `ks.subrun.joined`

```json
{
  "child_run_id": "01J8X9Y2Z3K4M5N6P7Q8R9CHD1",
  "parent_step_id": "01J8X9Y2Z3K4M5N6P7Q8R9SP16",
  "manifest_id_with_summary": "01J8X9Y2Z3K4M5N6P7Q8R9MF12",
  "summary_segment_id": "01J8X9Y2Z3K4M5N6P7Q8R9SG09"
}
```

The envelope's `run_id` is the parent. The next Step on the parent — the one that consumed the child's summary — is `parent_step_id`. The manifest that included the summary is `manifest_id_with_summary`.

## Sequence

```
[parent context]
ks.subrun.dispatched          (parent emits; payload.child_run_id is the new Run)
ks.run.created                (child appears in execution stream with parent_run_id populated)
ks.run.started                (child)
ks.step.* / ks.model_call.*   (child's own work)
ks.run.completed              (child)

[child context]
ks.subrun.summary_attached    (child emits; the summary lands as a ContextArtifact)

[parent context]
ks.step.created               (parent's next Step)
ks.context.manifest.created   (parent's next manifest, including the summary segment)
ks.subrun.joined              (parent emits when manifest assembly includes the summary)
```

A child's full event timeline is visible by filtering execution-stream events on `parent_run_id = <parent>`. The `ks.subrun.*` events are the explicit lineage markers — they make the parent ↔ child relationship addressable without requiring consumers to walk every event.

## Identity fields

The envelope's `parent_run_id` is the key correlation field for sub-run analysis. It is populated on:

- Sub-run events themselves (carrying the parent's `run_id` in the envelope and the child's ID in payload, or vice versa as listed above).
- Every event emitted under the child Run — the child's [execution events](/docs/concepts/events/execution), [routing events](/docs/concepts/events/routing), [context engineering events](/docs/concepts/events/context-engineering), etc., all carry `parent_run_id` so a consumer reconstructing "the full work the parent caused" can filter the entire stream by it.

## V1 scope: single-child summary handoff

The v1 sub-run model ships single-child summary handoff: a parent Run dispatches one child Run that runs to completion and posts one summary that the parent consumes in its next Step's manifest. The summary handoff is the boundary contract — the parent's manifest does not inflate with the child's full transcript.

Multi-child fan-out and fan-in coordination follow the same event taxonomy: each child emits its own dispatched / summary_attached / joined cycle, and the parent assembles a manifest that references multiple child summary segments.

## Common consumers

- **Lineage view.** Walks `ks.subrun.dispatched` → `ks.subrun.joined` to render the parent → child tree.
- **Cost rollup.** A parent Run's "all-in" cost includes the child's `usage_records` rows, joined by `parent_run_id`. The sub-run events are the lineage anchor that makes the join correct.
- **Audit of fan-out.** Aggregating `ks.subrun.dispatched` per parent surfaces parents that consistently fan out — useful for capacity planning and Budget tuning.
