The **execution events** stream covers the lifecycle of [Runs](/docs/concepts/run-hierarchy/runs), [Steps](/docs/concepts/run-hierarchy/steps), and [ModelCalls](/docs/concepts/run-hierarchy/model-calls). It is the primary timeline of "what happened to my work" â€” every consumer that builds a Run explorer, a timeline UI, or an external trace reconstructs it from this stream.

NATS subjects: `ks.run.*`, `ks.step.*`, `ks.model_call.*`. Persisted in the `execution_events` JetStream stream under the `ks.>` filter. Consumed via [`GET /v1/ks/events/execution`](/docs/api/events/execution).

## Event types

| Event | Trigger | Run state at emit |
|---|---|---|
| `ks.run.created` | Run row inserted. | `CREATED` |
| `ks.run.started` | Run transitions out of `CREATED` into `RUNNING` (after admission). | `RUNNING` |
| `ks.run.paused` | Run transitions to `PAUSED` (operator or runtime request). | `PAUSED` |
| `ks.run.recovering` | Run enters `RECOVERING` after failure or operator-triggered resume. | `RECOVERING` |
| `ks.run.resumed` | Run transitions from `RESUMING` back to `RUNNING`. | `RUNNING` |
| `ks.run.completed` | Run transitions to `COMPLETED`. | `COMPLETED` |
| `ks.run.failed` | Run transitions to `FAILED`. | `FAILED` |
| `ks.step.created` | Step row inserted (post-admission). | (parent Run) |
| `ks.step.started` | Step transitions to `STARTED`. | (parent Run) |
| `ks.step.completed` | Step transitions to `SUCCEEDED`, `FAILED`, `SKIPPED`, or `CANCELED`. | (parent Run) |
| `ks.model_call.started` | ModelCall row inserted; substrate dispatch begins. | (parent Run) |
| `ks.model_call.completed` | ModelCall finishes â€” success, failure, or cancellation. | (parent Run) |

The Run state listed is the *resulting* state â€” the event marks the transition into it. Events are emitted in the same database transaction that updates the row, so the event existence implies the row state.

## Ordering guarantees

Within a single Run, events are emitted in transition order:

```
ks.run.created
ks.run.started
ks.step.created (Step 1)
ks.step.started (Step 1)
ks.model_call.started (Step 1's ModelCall)
ks.model_call.completed
ks.step.completed (Step 1)
...
ks.run.completed
```

Cross-Run ordering is not guaranteed â€” two Runs running concurrently produce interleaved events. Within a Run, `event_id` lexicographic order matches transition order.

## Common consumers

- **Run timeline UI.** Filters by `run_id` and merges with [routing](/docs/concepts/events/routing), [context engineering](/docs/concepts/events/context-engineering), and [runtime](/docs/concepts/events/runtime) events.
- **External tracing.** Consumes `ks.model_call.completed` for cost and token aggregation alongside OTel spans correlated by `trace_id`.
- **Run-completion webhooks.** External integrations subscribe to `ks.run.completed` and `ks.run.failed` (via the consumer's own webhook layer over the pull endpoint).
- **SLO dashboards.** Aggregate `ks.run.failed` rate and `ks.run.completed` duration for run-completion SLOs.
