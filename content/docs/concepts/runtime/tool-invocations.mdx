A **ToolInvocation** is the runtime's first-class record of a single tool execution. It is the wedge-specific runtime entity that pairs a `TOOL_EXECUTE` Step with the concrete details of what tool was called, with what arguments, against which [sandbox](/docs/concepts/runtime/sandboxes), and what came back.

Tool *invocation* вҖ” the actual call-and-result loop, sandbox dispatch, the wedge-specific tool classes вҖ” is owned by the Layer 3 runtime. The Layer 2 control plane owns the global governance and accounting around invocation: a [PolicyScope](/docs/concepts/policy/policy-scopes)'s `tool_policy_ref` decides which tools the resolved scope may invoke, and a [Budget](/docs/concepts/policy/budgets)'s `tool_limit` caps how many invocations a Run may make. The runtime calls into both before dispatch; ToolInvocation is the entity that records each call once admission is granted.

## What a ToolInvocation records

| Field | Purpose |
|---|---|
| `id` | ULID. |
| `run_id`, `step_id` | The Run and the `TOOL_EXECUTE` Step the invocation belongs to. |
| `tool_name` | The tool's name. |
| `tool_class` | The runtime tool class вҖ” see [Tool classes](#tool-classes). |
| `arguments` | The tool's input arguments (subject to redaction policy). |
| `sandbox_id` | The [sandbox](/docs/concepts/runtime/sandboxes) the invocation dispatched into (null for non-sandbox-bound tools). |
| `result` | The structured tool result вҖ” present inline when admission allows it. |
| `result_artifact_id` | The [ContextArtifact](/docs/concepts/context/context-artifacts) that holds the full result when admission required offload. |
| `result_summary` | A bounded summary segment when admission required summarization. |
| `cost` | Per-invocation cost (when the tool class is priced вҖ” e.g. `http_api` calls to billed endpoints). |
| `started_at`, `completed_at`, `duration_ms` | Timing. |
| `outcome` | `SUCCEEDED | FAILED | DENIED | TIMED_OUT`. |

## Tool classes

Layer 3 defines the standard tool classes. The class drives sandbox dispatch policy, governance defaults, and cost behavior.

| Class | Used for |
|---|---|
| `shell` | Arbitrary shell commands inside the sandbox. Subject to file/network policy. |
| `file` | File read, write, list, stat, hash. The runtime translates to `FILE_READ` / `FILE_WRITE` Steps when the operation is mutating. |
| `repo_git` | Git operations against the workspace's source binding вҖ” `clone`, `checkout`, `diff`, `commit`, `push`. Push is a checkpoint trigger. |
| `test_build` | Test runners and build commands вҖ” `pytest`, `npm test`, `cargo build`, custom runners. Output captured as a test report [WorkspaceArtifact](/docs/concepts/runtime/workspace-artifacts). |
| `http_api` | Outbound HTTP calls. Egress allowlist enforced by the sandbox network policy. |
| `structured_external` | Schema-typed external tools вҖ” anything from search APIs to internal company services with a registered JSON schema. |

The runtime dispatches each tool according to its class.

## Built-in tools

The runtime ships with a curated set of built-in tools that cover the wedge вҖ” coding, terminal, and tool-heavy long-running agents вҖ” out of the box. Custom tools register through the same registry and follow the same dispatch and governance contract.

| Tool | Class | Purpose |
|---|---|---|
| `file_read` | `file` | Read a file from the [Workspace](/docs/concepts/runtime/workspaces). Materializes a `FILE_READ` Step. |
| `file_write` | `file` | Write or patch a file in the Workspace. Materializes a `FILE_WRITE` Step and emits a `diff` [WorkspaceArtifact](/docs/concepts/runtime/workspace-artifacts). |
| `shell` | `shell` | Run a shell command inside the [sandbox](/docs/concepts/runtime/sandboxes). Materializes a `PROCESS_START` Step and captures stdout/stderr artifacts. |
| `search` | `structured_external` | Code or document search across the Workspace and registered indexes. |
| `git` | `repo_git` | Git operations against the Workspace's source binding вҖ” `clone`, `checkout`, `diff`, `commit`, `push`. `push` is a Checkpoint trigger. |
| `test_runner` | `test_build` | Run the Workspace's tests under the configured runner. Output is captured as a `test_report` WorkspaceArtifact. |

Custom tools register through `POST /v1/ks/runtime/tools` with a name, class, JSON schema for arguments, idempotency flag, and an optional cost rule. The runtime then dispatches them through the same Tool Gateway path as the built-ins.

## Layer boundary вҖ” Layer 3 invocation, Layer 2 governance

The split is deliberate and load-bearing.

| Concern | Owner | Surface |
|---|---|---|
| Tool registration, classes, dispatch semantics, sandbox bind | Layer 3 runtime | This page; Tool Registry; Tool Gateway. |
| Per-tool execution, output capture, artifact offload | Layer 3 runtime | [Sandboxes](/docs/concepts/runtime/sandboxes); [WorkspaceArtifacts](/docs/concepts/runtime/workspace-artifacts). |
| Which tools the resolved scope may invoke (allow/deny) | Layer 2 control plane | [PolicyScope](/docs/concepts/policy/policy-scopes) `tool_policy_ref`. |
| Cap on invocation count for a Run | Layer 2 control plane | [Budget](/docs/concepts/policy/budgets) `tool_limit`. |
| Audit trail across tenants and projects | Layer 2 control plane | [Events](/docs/concepts/events) вҖ” `ks.tool.invocation.started`, `ks.tool.invocation.completed`. |

A blocked invocation (policy or budget) never reaches dispatch вҖ” the Tool Gateway records the denial as the ToolInvocation's outcome (`DENIED` for policy, `FAILED` with `cause: tool_limit_exceeded` for budget) and the producing Step ends with the appropriate failure class.

## Wire shape

Recording a tool invocation explicitly:

```http
POST /v1/ks/tools/report
Authorization: Bearer ks_...
Content-Type: application/json

{
  "run_id":          "01J8X9Y2Z3K4M5N6P7Q8R9S0T1",
  "step_id":         "01J8X9Y2Z3K4M5N6P7Q8R9STP1",
  "tool_name":       "run_tests",
  "tool_class":      "test_build",
  "sandbox_id":      "01J8X9Y2Z3K4M5N6P7Q8R9SBX1",
  "arguments":       { "path": "tests/payments_test.py" },
  "started_at":      "2026-04-30T12:01:00.000Z",
  "completed_at":    "2026-04-30T12:01:42.180Z",
  "outcome":         "FAILED",
  "result":          { "exit_code": 1, "stdout_preview": "FAILED tests/payments_test.py::test_refund_partial" }
}
```

Response:

```json
{
  "id":                  "01J8X9Y2Z3K4M5N6P7Q8R9TIV1",
  "result_artifact_id":  "01J8X9Y2Z3K4M5N6P7Q8R9ART1",
  "result_summary":      "1 of 47 tests failed: test_refund_partial вҖ” assertion error on `tax_cents`.",
  "cost": { "cost_cents": "0.0000" }
}
```

The next `MODEL_CALL` Step in the Run includes the `result_summary` as a typed [context segment](/docs/concepts/context/context-segments) bound to `result_artifact_id`. The full stdout is fetchable through `GET /v1/ks/tools/results/{id}?mode=full` when an operator needs to inspect it on the [timeline](/docs/concepts/runtime/runtime-observability).

## Idempotency

Tools are marked idempotent or not at registration. The runtime uses this for [retry](/docs/concepts/recovery/retries) decisions:

| Idempotent | Retry behavior |
|---|---|
| Yes | Retried in place on transient failures. The same `(run_id, step_id, tool_name, arguments_hash)` is the idempotency key. |
| No | Not retried in place. The failure escalates through the [retry tier ladder](/docs/concepts/recovery/retries) вҖ” the runtime resumes from the previous Checkpoint, and if that does not clear the failure the tier escalates to a new Run lineage from the Checkpoint via [branch flow](/docs/concepts/recovery/branch-flow). The runtime does not define a parallel retry loop here; the contract lives in [retries](/docs/concepts/recovery/retries). |

`file` reads, `http_api` GETs, `repo_git` reads, `test_build` runs are typically idempotent. `shell` (without explicit override), `repo_git` writes (commit/push), and non-GET `http_api` are typically not.

## Events

Each invocation emits two [runtime events](/docs/concepts/events/runtime):

| Event | When |
|---|---|
| `ks.tool.invocation.started` | Dispatched into the sandbox (or into the Worker's outbound HTTP path for non-sandbox tools). |
| `ks.tool.invocation.completed` | Result captured. |

The events carry `tool_name`, `tool_class`, and `outcome` so dashboards and event consumers can build a per-Run tool action timeline without joining Postgres.

## Wedge-specific vs. general tools

Layer 3 owns the wedge-specific tool classes вҖ” shell, file, repo/git, test/build вҖ” because these are what coding, terminal, and tool-heavy long-running agents need. A team running a [BYO runtime](/docs/concepts/deployment/byo-runtime) that has its own tool gateway records ToolInvocations through the same `POST /v1/ks/tools/report` API; the runtime owns dispatch, but the canonical record shape is the same.
