title: Sandboxes

---

A **Sandbox** is the isolated execution environment a [Worker](/docs/concepts/runtime/workers) dispatches file, process, shell, and tool actions into during a Run in [KovaServe runtime mode](/docs/concepts/deployment/managed-runtime). The runtime is process-, file-, and tool-heavy, and sandboxes are first-class: every `FILE_WRITE`, `PROCESS_START`, and sandbox-bound tool runs inside a sandbox rather than on the Worker host. Sandboxes are container-based.

```
┌─ Worker (lease holder) ──────────────────────┐
│                                              │
│  decides next Step ──▶ dispatches into ─────▶│
│                                              │
└──────────┬───────────────────────────────────┘
           │
           ▼
┌─ Sandbox (container) ────────────────────────┐
│                                              │
│  ┌─ Workspace mount ─────────┐               │
│  │  /workspace (rw or ro)    │               │
│  │   ├── source bound        │               │
│  │   ├── mutation log        │               │
│  │   └── snapshots           │               │
│  └───────────────────────────┘               │
│                                              │
│  policy-limited file / network access        │
│  output capture (stdout / stderr / files)    │
│  resource accounting (cpu / mem / time)      │
│  optional persistence + branchable snapshots │
│                                              │
└──────────────────────────────────────────────┘
```

The Sandbox Manager is the [runtime subsystem](/docs/concepts/runtime/runtime-architecture#sandbox-manager) that owns the lifecycle.

## What a Sandbox provides

| Capability | Detail |
|---|---|
| Shell and process execution | The sandbox shell and PID namespace. `PROCESS_START` Steps run here. |
| Workspace mounting | The Run's [workspace](/docs/concepts/runtime/workspaces) mounts at a known path (default `/workspace`) under the read/write policy configured on the Run. |
| File access policy | Per-path allow/deny rules; writes outside the writable scope are rejected before reaching disk. |
| Network access policy | Egress allowlist. Tools that need outbound HTTP target the [Tool Gateway](/docs/concepts/runtime/tool-invocations) which enforces tool-level policy on top. |
| Output capture | Stdout, stderr, and any files the runtime declares "produced" are captured and recorded as [WorkspaceArtifacts](/docs/concepts/runtime/workspace-artifacts). |
| Resource accounting | Cpu, memory, wall-clock, disk, optional GPU. Limits enforced; usage reported on the Run. |
| Persistence | Optional. A persistent sandbox survives across Steps within the same Run (and, if the snapshot is captured, across resumes from a Checkpoint). |
| Branchable snapshots | A sandbox snapshot is the basis of [branch flow](/docs/concepts/recovery/branch-flow) — a child Run starts from the snapshot's filesystem state, not from a clean image. |

## Sandbox lifecycle

```
CREATED ─▶ STARTING ─▶ READY ─▶ RUNNING ─▶ PAUSED ─▶ RUNNING ─▶ STOPPED ─▶ ARCHIVED
                                  │
                                  ├──▶ SNAPSHOT_CAPTURING ─▶ SNAPSHOT_STORED
                                  │
                                  └──▶ FAILED
```

| State | Meaning |
|---|---|
| `CREATED` | Sandbox row exists; the container has not been scheduled yet. |
| `STARTING` | Container scheduled and booting. |
| `READY` | Container is up; workspace is mounted; the sandbox can accept Step dispatch. |
| `RUNNING` | Actively executing a Step. |
| `PAUSED` | Sandbox is held but not progressing; used during runtime pause. |
| `SNAPSHOT_CAPTURING` | A snapshot is being captured for a Checkpoint or branch point. |
| `SNAPSHOT_STORED` | The snapshot is durable in object storage; the running sandbox continues. |
| `STOPPED` | Sandbox has exited; bytes still in object storage if persistent. |
| `ARCHIVED` | Retention window expired; bytes evicted. |
| `FAILED` | Sandbox crashed or violated resource limits. |

## Sandbox profile

A Run is created against a **sandbox profile** that the agent spec carries. The profile fixes container image, resource limits, file/network policy, and persistence behavior.

| Field | Purpose |
|---|---|
| `image` | Container image reference (registry-pinned). |
| `cpu_limit`, `memory_limit_mb` | Compute caps. |
| `time_limit_seconds` | Wall-clock cap on a single Step's sandbox-side execution. |
| `disk_limit_mb` | Workspace + ephemeral disk cap. |
| `gpu`        | Optional GPU attachment for runtimes that need it. |
| `network_policy` | `none` / `allowlist[hosts]` / `open` (private deployments only). |
| `file_policy` | Read-only paths, writable paths, denied paths. |
| `persistence` | `none` (sandbox per Step), `run` (one sandbox for the Run), `session` (sandbox survives across Runs in a Session). |
| `snapshot_policy` | `never` / `on_checkpoint` / `manual`. |

Profiles are versioned. Agents pin a profile version so a Run is reproducible regardless of subsequent profile changes.

## Persistent sandboxes

A sandbox with `persistence: run` lives for the duration of the Run. Steps that would mount the workspace fresh on every dispatch instead reuse the same filesystem state:

- `FILE_WRITE` from Step 7 is visible to `PROCESS_START` in Step 8.
- A long-running build artifact compiled in Step 5 is still on disk for Step 12 to test.
- Tool output written to the workspace persists for the next tool to pick up.

A sandbox with `persistence: session` extends across Runs that share a [Session](/docs/concepts/continuity/sessions) — used for agents that should pick up where the last Run left off.

## Branchable snapshots

A snapshot is the byte-level state of the sandbox filesystem at a moment in time. Snapshots are taken:

- on every [Checkpoint](/docs/concepts/recovery/checkpoints) when `snapshot_policy = on_checkpoint` — captured as a `runtime_state` [StateHandle](/docs/concepts/recovery/state-handles)
- on explicit operator request
- automatically before [branches](/docs/concepts/recovery/branches) so the child Run starts from the parent's state

A child Run created via [branch flow](/docs/concepts/recovery/branch-flow) gets a fresh sandbox seeded from the snapshot. Two child Runs branched from the same snapshot evolve independently — the snapshot is the immutable common ancestor.

Snapshots are stored in object storage under tenant-scoped paths. Retention is governed by the [PolicyScope](/docs/concepts/policy/policy-scopes) `data_boundary` and the project's snapshot retention setting.

## Output capture

Three categories of output are captured automatically:

| Category | Captured as |
|---|---|
| Stdout / stderr of `PROCESS_START` | [WorkspaceArtifact](/docs/concepts/runtime/workspace-artifacts) of type `stdout` / `stderr`. |
| Files the runtime declares "produced" | WorkspaceArtifact of type `file`, with the file's path and content hash. |
| File diffs on `FILE_WRITE` Steps | WorkspaceArtifact of type `diff`, attached to the Step. |

Captured artifacts surface on the Run's [timeline](/docs/concepts/runtime/runtime-observability) and through `GET /v1/ks/runtime/runs/{id}/artifacts`.

## Sandbox events

Runtime events the sandbox produces:

| Event | When |
|---|---|
| `ks.process.started` | A `PROCESS_START` Step launches a process. |
| `ks.process.waiting` | A long-running process has been waiting on input or completion past a configured threshold. |
| `ks.file.write.applied` | A `FILE_WRITE` Step's bytes hit disk. |
| `ks.tool.invocation.started` / `ks.tool.invocation.completed` | Tool Gateway dispatches a tool through the sandbox. |

See [runtime events](/docs/concepts/events/runtime) for the full stream.

## Sandbox and substrate

A sandbox is the Worker's *execution* boundary for file, process, shell, and tool actions. Model calls follow a separate path: they dispatch through `POST /v1/chat/completions` to the [Cache-Aware Router](/docs/concepts/routing/manifest-aware-routing) and execute on a vLLM substrate node. Tools that talk to external HTTP services egress from the sandbox; tools that talk to in-cluster KovaServe APIs go through the Worker.

## Failure modes

| Failure | Effect |
|---|---|
| Container crash | Sandbox transitions to `FAILED`; Run's terminal Step records `SANDBOX_FAILURE`; the Worker releases the lease. |
| Resource limit violation | Same as crash — `SANDBOX_FAILURE` with a `cause` of `cpu` / `memory` / `disk` / `time`. |
| Policy violation (file or network) | The action is rejected before it reaches the sandbox runtime; the Step ends with a denial recorded as a [policy event](/docs/concepts/events/policy). |
| Snapshot capture failure | The Checkpoint that triggered the snapshot still completes — the canonical Checkpoint remains valid without the snapshot, since [resume](/docs/concepts/recovery/resume-flow) is logical-context-first. |
