A **Workspace** is the structured filesystem a [Run](/docs/concepts/run-hierarchy/runs) in [KovaServe runtime mode](/docs/concepts/deployment/managed-runtime) operates against. Every Run that touches files has exactly one Workspace, and every `FILE_READ`, `FILE_WRITE`, and `PROCESS_START` Step targets it. The Workspace gives the runtime a structured model — source binding, mutation log, snapshots, branch lineage, read/write policy — instead of an opaque mounted directory.

```
┌─ Workspace ────────────────────────────────────────────────┐
│                                                            │
│  source binding ─────▶ git@github.com:acme/payments        │
│                        ref = main, commit = a3f2c81        │
│                                                            │
│  workspace root      ▶ /workspace                          │
│                                                            │
│  read/write policy   ▶ writable: /workspace                │
│                        read-only: /etc, /usr               │
│                        denied:    /workspace/.kova-secrets │
│                                                            │
│  mutation log        ▶ FILE_WRITE Step 12: src/refund.py   │
│                        FILE_WRITE Step 17: tests/...       │
│                        ...                                 │
│                                                            │
│  snapshots           ▶ snap_01.. (Checkpoint at Step 14)   │
│                        snap_02.. (manual at Step 21)       │
│                                                            │
│  branch lineage      ▶ branched from snap_01.. by Run X    │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

The Workspace / Artifact Manager is the [runtime subsystem](/docs/concepts/runtime/runtime-architecture#workspace--artifact-manager) that owns the lifecycle.

## What a Workspace records

| Field | Purpose |
|---|---|
| `id` | ULID. |
| `run_id` | The Run the Workspace is bound to. A Run owns at most one Workspace. |
| `session_id` | The Session, when the Workspace persists across Runs in a Session. |
| `source` | The source binding — Git URL + ref + resolved commit, or a registered tarball, or an explicitly empty workspace. |
| `root` | The mount path inside the [sandbox](/docs/concepts/runtime/sandboxes) (default `/workspace`). |
| `read_write_policy` | Writable paths, read-only paths, denied paths. |
| `mutation_log_count` | How many `FILE_WRITE` Steps have committed mutations. |
| `snapshots[]` | Snapshot ids and the Step boundaries they were captured at. |
| `branch_lineage` | If this Workspace is a branch, the parent Workspace id and the snapshot it was seeded from. |
| `created_at` | When the Workspace was provisioned. |
| `archived_at` | When the Workspace was archived (after Run completion + retention). |

The Workspace itself is the metadata surface. The bytes on disk live inside the sandbox; the byte snapshots live in object storage.

## Source binding

A Workspace's source binding determines what the workspace looks like at provisioning time. The accepted shapes:

| Source type | Provisioning |
|---|---|
| `git` | `git clone <url>` + `git checkout <ref>`. The resolved commit hash is recorded so the Workspace is reproducible regardless of subsequent ref movement. |
| `tarball` | A previously registered [WorkspaceArtifact](/docs/concepts/runtime/workspace-artifacts) of type `workspace_snapshot`. |
| `branch` | Seeded from a parent Workspace's snapshot. Provisioned by the [branch flow](/docs/concepts/recovery/branch-flow). |
| `empty` | Empty `/workspace`. Used by document-review and ops-style runs that build the workspace from tool outputs. |

## Read/write policy

The policy is a list of path rules evaluated in order. The runtime rejects a `FILE_WRITE` that targets a path outside the writable scope before any bytes hit disk.

```json
{
  "writable_paths":  ["/workspace"],
  "read_only_paths": ["/etc", "/usr", "/workspace/vendor"],
  "denied_paths":    ["/workspace/.kova-secrets", "/workspace/**/.env"]
}
```

| Rule | Effect |
|---|---|
| `writable_paths` | Reads and writes allowed. The default for the workspace root. |
| `read_only_paths` | Reads allowed; writes denied. |
| `denied_paths` | All access denied; surfaces a `policy denied` failure on the Step. |

Denied access produces a [policy event](/docs/concepts/events/policy) and a `POLICY_DENY` failure class on the Step.

## Mutation log

The mutation log is the ordered record of every `FILE_WRITE` Step that committed bytes to the Workspace. Each entry records:

| Field | Purpose |
|---|---|
| `step_id` | The Step that produced the mutation. |
| `paths[]` | Paths written. |
| `diff_artifact_id` | The [WorkspaceArtifact](/docs/concepts/runtime/workspace-artifacts) holding the diff. |
| `applied_at` | When the bytes were committed. |

The mutation log is what makes the [timeline](/docs/concepts/runtime/runtime-observability) able to show "this Run changed these files in this order, and here is the diff for each." It is also what makes a Run reproducible from its source commit + the mutation sequence.

## Snapshots

A Workspace snapshot is the byte-level state of the writable scope at a moment in time. Snapshots are produced:

- on every [Checkpoint](/docs/concepts/recovery/checkpoints) when the [sandbox](/docs/concepts/runtime/sandboxes) profile sets `snapshot_policy = on_checkpoint`
- on explicit operator request via `POST /v1/ks/runtime/workspaces/{id}/snapshot`
- before [branches](/docs/concepts/recovery/branches), so the child Run starts from the parent's filesystem state

Snapshots are stored in object storage as `workspace_snapshot` [WorkspaceArtifacts](/docs/concepts/runtime/workspace-artifacts) and surface as `runtime_state` [StateHandles](/docs/concepts/recovery/state-handles) on Checkpoints. Two child Runs branched from the same snapshot evolve independently.

## Persistence across Runs

A Workspace's persistence is governed by the [sandbox profile](/docs/concepts/runtime/sandboxes) on the agent spec:

| Sandbox `persistence` | Workspace lifetime |
|---|---|
| `none` | A fresh Workspace per Step. Rare; used only by stateless tool runs. |
| `run` | One Workspace for the duration of the Run. The default. |
| `session` | The Workspace persists across Runs in the same Session — useful for agents that pick up where the last Run left off. |

A Workspace's archival is governed by the project's snapshot retention setting and the [PolicyScope](/docs/concepts/policy/policy-scopes) `data_boundary`.

## Endpoints

| Endpoint | Purpose |
|---|---|
| `GET /v1/ks/runtime/workspaces/{id}` | Fetch the Workspace metadata, mutation log count, and snapshot list. |
| `POST /v1/ks/runtime/workspaces/{id}/snapshot` | Capture a snapshot on demand. Returns a `workspace_snapshot` WorkspaceArtifact id. |

The Workspace itself is created implicitly when a Run is created with a `workspace.source` block — there is no separate "create workspace" call in the common path.

## Workspaces and adjacent concepts

A Workspace sits next to several related entities; the relationships are explicit:

- A Workspace holds the runtime's filesystem state. Bytes from the Workspace enter context only when a `FILE_READ` brings them into a [ContextSegment](/docs/concepts/context/context-segments), at which point the bytes are catalogued as a [ContextArtifact](/docs/concepts/context/context-artifacts).
- A Workspace's source is one of `git`, `tarball`, `branch`, or `empty` — Git is one of several source types.
- Each Run has a Workspace bound to itself; cross-Run continuity flows through Session-scoped persistence and snapshots, not through concurrent shared mounts.
- A Workspace is mounted *into* a [sandbox](/docs/concepts/runtime/sandboxes); the sandbox is the execution boundary around it.
