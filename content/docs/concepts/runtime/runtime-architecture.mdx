The **KovaServe runtime** sits on top of the [Layer 2](/docs/concepts/layers) control plane and turns canonical [Runs](/docs/concepts/run-hierarchy/runs), [Steps](/docs/concepts/run-hierarchy/steps), [ContextManifests](/docs/concepts/context/context-manifests), and [Checkpoints](/docs/concepts/recovery/checkpoints) into a coherent runtime experience for coding agents, terminal agents, tool-heavy long-running agents, and filesystem and process-heavy loops. It is the subject of the [managed runtime mode](/docs/concepts/deployment/managed-runtime); this page describes its internal architecture.

The runtime is intentionally focused. It is the surface where tight coupling between runtime semantics and control-plane semantics produces a coherent product experience — pause, resume, branch, retry, sandbox-bound execution, structured workspaces, and a runtime-native timeline, all driven from canonical control-plane entities.

## Subsystems

The runtime is composed of eleven subsystems.

### Runtime API Service

The runtime's public HTTP surface. Hosts `POST /v1/ks/runtime/runs`, the pause/resume/branch/retry endpoints, the timeline endpoint, and the artifacts endpoint. Resolves [tenancy](/docs/concepts/tenancy), authenticates Bearer keys, normalizes runtime metadata, and forwards into the orchestrator.

### Runtime Orchestrator

Routes new Runs onto available [Workers](/docs/concepts/runtime/workers) under lease-based ownership. Owns the runtime-side decisions: which Worker claims which Run, when a Worker's lease is renewed, when a Run is reassigned. Canonical state stays in the control plane; the orchestrator only holds the runtime-side cursor on it.

### Worker Manager

Manages the Worker pool — claim runnable work, materialize runtime state, execute step transitions, call into the control plane for model calls and checkpoints, dispatch sandboxes and tools, renew or release leases, and emit runtime events. Everything required for resume is persisted in [runtime-local state](/docs/concepts/runtime/runtime-local-state) outside Worker memory, so reassignment after lease loss reconstructs a consistent view.

### Sandbox Manager

Manages the [sandbox](/docs/concepts/runtime/sandboxes) lifecycle. The runtime is process- and file-heavy by design, so isolated execution environments are first-class. Responsibilities include shell and process execution, workspace mounting, policy-limited file and network access, output capture, resource accounting, persistent run sandboxes, and branchable sandboxes from snapshot points. Sandboxes are container-based.

### Tool Registry and Tool Gateway

The runtime [tool invocation](/docs/concepts/runtime/tool-invocations) layer. Built-in tool classes — shell and command, file, repo and git, test and build, HTTP and API, and structured external tools — cover the curated set of `file_read`, `file_write`, `shell`, `search`, `git`, and `test_runner` tools the runtime ships with, alongside any custom tools registered for an agent spec. The Tool Gateway dispatches each invocation, enforces sandbox boundaries, and emits `ks.tool.invocation.*` events. The runtime owns tool registration, dispatch, and runtime-side invocation semantics; the control plane owns global policy enforcement (`tool_policy_ref` on the resolved [PolicyScope](/docs/concepts/policy/policy-scopes)) and budget enforcement (`tool_limit` on the resolved [Budget](/docs/concepts/policy/budgets)).

### Workspace / Artifact Manager

Owns the structured [workspace](/docs/concepts/runtime/workspaces) model — workspace root, source binding, mutation log, snapshots, branch lineage, read/write policy — and the [WorkspaceArtifact](/docs/concepts/runtime/workspace-artifacts) catalog: stdout, stderr, file diffs, patches, test reports, tool results, model outputs, workspace snapshots, checkpoint notes. Artifact bytes land in object storage; the catalog rows live in Postgres alongside the canonical entity tables.

### Runtime State Store

Holds [runtime-local state](/docs/concepts/runtime/runtime-local-state): planner scratchpad, active process handles, tool loop position, workspace refs, branch pointers, last safe checkpoint. Anything required for resume must live here — outside Worker memory — so Worker reassignment is safe and lease loss does not destroy progress.

### Checkpoint Integration Module

Layered on top of the canonical [Checkpoint Manager](/docs/concepts/recovery/checkpoints). Defines the runtime-aware trigger rules: before risky file mutations, before unsafe shell actions, after stable analysis phases, on budget warning, manual checkpoints, and periodic checkpoints. The actual checkpoint capture goes through the canonical control-plane path — what the runtime adds is the *when*. Checkpoints attach a `runtime_state` [StateHandle](/docs/concepts/recovery/state-handles) to the canonical record so resume can reconstitute the runtime view.

### Retry / Resume / Branch Manager

Owns the runtime UX for [retries](/docs/concepts/recovery/retries), [resumes](/docs/concepts/recovery/resume-flow), and [branches](/docs/concepts/recovery/branches). Only safe and idempotent runtime steps retry in place. Resume can target the current step, the previous step, the last checkpoint, or a different worker via restore-based relocation. Branch creates a new run lineage from a checkpoint or stable point with bounded inherited budgets and policies — see [branch flow](/docs/concepts/recovery/branch-flow).

### Timeline / Event Projection Service

Projects the canonical [Step](/docs/concepts/run-hierarchy/steps), [ModelCall](/docs/concepts/run-hierarchy/model-calls), [tool invocation](/docs/concepts/runtime/tool-invocations), and [checkpoint event](/docs/concepts/events/checkpoint) streams into a runtime-native timeline view: typed step list, model and tool actions, [diffs](/docs/concepts/runtime/workspace-artifacts), [checkpoint](/docs/concepts/recovery/checkpoints) tree, [branch lineage](/docs/concepts/recovery/branch-flow), failure classification, retry/resume reasons. See [runtime observability](/docs/concepts/runtime/runtime-observability).

### Runtime SDK

The Python-first SDK that authors and operates Runs against the runtime. `KovaClient` wraps inference with the canonical `kovaserve` extension envelope; `KovaControlClient` covers control-plane CRUD for runs, budgets, manifests, artifacts, and tenants; `ContextBuilder`, `ArtifactClient`, and `ContextBudgetClient` cover context shape, offload, and admission; `kovaserve.adapters.langgraph` is the LangGraph integration. The SDK surfaces the same APIs the runtime uses internally — anything an agent author can do through the SDK, the runtime does through the same paths.

## Ownership boundary

The boundary is precise — and is enforced by what the runtime persists locally vs. what it resolves through the control plane.

| The runtime owns | The control plane owns |
|---|---|
| Agent definition model | Canonical Run, Session, Step, and ModelCall persistence and state machines |
| Worker lease and assignment | [PolicyScope](/docs/concepts/policy/policy-scopes), tenant resolution, [Budget](/docs/concepts/policy/budgets) and [ContextBudget](/docs/concepts/policy/context-budgets) admission |
| [Sandbox](/docs/concepts/runtime/sandboxes) lifecycle | Routing across the substrate fleet |
| [Tool registration and invocation](/docs/concepts/runtime/tool-invocations) semantics | Global policy and budget enforcement |
| [Workspace](/docs/concepts/runtime/workspaces) and [WorkspaceArtifact](/docs/concepts/runtime/workspace-artifacts) model | [ContextManifest](/docs/concepts/context/context-manifests), [ContextSegment](/docs/concepts/context/context-segments), [ContextArtifact](/docs/concepts/context/context-artifacts), [CompactionRecord](/docs/concepts/context/compaction-records) |
| [Runtime-local state](/docs/concepts/runtime/runtime-local-state) | Canonical entity persistence |
| Checkpoint trigger rules at runtime boundaries | [Checkpoint Manager](/docs/concepts/recovery/checkpoints) capture, validation, and restore |
| [Retry, resume, and branch](/docs/concepts/recovery/branches) UX | Run [state machine](/docs/concepts/run-hierarchy/run-state-machine) transitions |
| Runtime-native [timeline](/docs/concepts/runtime/runtime-observability) UX | Event taxonomy and [event envelope](/docs/concepts/events/event-envelope) |

When the runtime compacts context, it produces a [CompactionRecord](/docs/concepts/context/compaction-records) through the canonical API. When the runtime offloads a tool result, it writes a [ContextArtifact](/docs/concepts/context/context-artifacts). The control-plane surface is what keeps the runtime thin.

## Wire shape — runtime API

The runtime exposes a small surface over the canonical control plane:

| Endpoint | Purpose |
|---|---|
| `POST /v1/ks/runtime/projects` | Create or get a runtime project. |
| `POST /v1/ks/runtime/agents` | Register an agent spec — model, system prompt, tool list, sandbox profile, defaults. |
| `POST /v1/ks/runtime/runs` | Trigger a Run against an agent and a workspace source. |
| `POST /v1/ks/runtime/runs/{id}/pause` | Pause a Run. |
| `POST /v1/ks/runtime/runs/{id}/resume` | Resume a Run from current step, previous step, or last checkpoint. |
| `POST /v1/ks/runtime/runs/{id}/branch` | Branch a new Run lineage from a Checkpoint or stable point. |
| `POST /v1/ks/runtime/runs/{id}/steps/{step_id}/retry` | Retry a safe/idempotent step. |
| `GET  /v1/ks/runtime/runs/{id}/timeline` | The runtime-native timeline view. |
| `GET  /v1/ks/runtime/runs/{id}/artifacts` | The WorkspaceArtifact catalog for the Run. |

## Deployment

The runtime ships as part of the same Helm chart as the control plane. On [managed cloud](/docs/concepts/deployment/managed-cloud) the runtime workers and sandboxes run inside the hosted cluster; on [private deployment](/docs/concepts/deployment/private-deployment) they run inside the customer's Kubernetes cluster, on the same nodes — or on dedicated runtime-pool nodes — as the rest of the control plane.
