The **KovaServe runtime** is the Layer 3 product that sits on top of the Layer 2 control plane and turns canonical [Runs](/docs/concepts/runs), [Steps](/docs/concepts/steps), [ContextManifests](/docs/concepts/context-manifests), and [Checkpoints](/docs/concepts/checkpoints) into a coherent runtime experience for the wedge — coding agents, terminal agents, tool-heavy long-running agents, and filesystem/process-heavy loops. It is the subject of the [managed runtime mode](/docs/concepts/managed-runtime); this page describes its internal architecture.

The runtime is intentionally focused. It does not aim to be a general-purpose orchestrator. It exists where tighter coupling between runtime semantics and control-plane semantics produces clearly better outcomes than a [BYO runtime](/docs/concepts/byo-runtime) integration could achieve.

## Subsystems

The runtime is composed of eleven Layer 3 subsystems.

### Runtime API Service

The runtime's public HTTP surface. Hosts `POST /v1/ks/runtime/runs`, the pause/resume/branch/retry endpoints, the timeline endpoint, and the artifacts endpoint. Resolves [tenancy](/docs/concepts/tenancy), authenticates Bearer keys, normalizes runtime metadata, and forwards into the orchestrator.

### Runtime Orchestrator

Routes new Runs onto available [Workers](/docs/concepts/workers) under lease-based ownership. Owns the runtime-side decisions: which Worker claims which Run, when a Worker's lease is renewed, when a Run is reassigned. The orchestrator never holds canonical state — that lives in Layer 2.

### Worker Manager

Manages the Worker pool — claim runnable work, materialize runtime state, execute step transitions, interact with Layer 2 for model calls and checkpoints, interact with sandboxes and tools, renew or release leases, emit runtime events. A Worker is never the sole holder of state needed for resume; everything required for resume is persisted in [runtime-local state](/docs/concepts/runtime-local-state) outside Worker memory.

### Sandbox Manager

Manages the [sandbox](/docs/concepts/sandboxes) lifecycle. The runtime is process- and file-heavy by design, so isolated execution environments are first-class. Responsibilities include shell/process execution, workspace mounting, policy-limited file and network access, output capture, resource accounting, optional persistent run sandboxes, and branchable sandboxes from snapshot points. v1 sandboxes are container-based.

### Tool Registry and Tool Gateway

The wedge-specific [tool invocation](/docs/concepts/tool-invocations) layer. Built-in tool classes — shell/command, file, repo/git, test/build, HTTP/API, structured external. Tool definitions hash into a [ToolCatalog](/docs/concepts/tool-catalogs) so cache breakpoints stay stable. The Tool Gateway dispatches each invocation, enforces sandbox boundaries, and emits `ks.tool.invocation.*` events. Layer 3 owns wedge-specific invocation semantics; Layer 2 owns global policy and budget enforcement.

### Workspace / Artifact Manager

Owns the structured [workspace](/docs/concepts/workspaces) model — workspace root, source binding, mutation log, snapshots, branch lineage, read/write policy — and the [WorkspaceArtifact](/docs/concepts/workspace-artifacts) catalog: stdout, stderr, file diffs, patches, test reports, tool results, model outputs, workspace snapshots, checkpoint notes. Artifact bytes land in object storage; the catalog rows live in Postgres alongside the canonical entity tables.

### Runtime State Store

Holds [runtime-local state](/docs/concepts/runtime-local-state): planner scratchpad, active process handles, tool loop position, workspace refs, branch pointers, last safe checkpoint. Anything required for resume must live here — outside Worker memory — so Worker reassignment is safe and lease loss does not destroy progress.

### Checkpoint Integration Module

Layered on top of the Layer 2 [Checkpoint Manager](/docs/concepts/checkpoints). Defines the wedge-aware trigger rules: before risky file mutations, before unsafe shell actions, after stable analysis phases, on budget warning, manual checkpoints, periodic checkpoints. The actual checkpoint capture goes through the canonical Layer 2 path — what the runtime adds is the *when*. Checkpoints attach a `runtime_state` [StateHandle](/docs/concepts/state-handles) to the canonical record so resume can reconstitute the runtime view.

### Retry / Resume / Branch Manager

Owns the runtime UX for [retries](/docs/concepts/retries), [resumes](/docs/concepts/resume-flow), and [branches](/docs/concepts/branches). Only safe and idempotent runtime steps retry in place. Resume can target the current step, the previous step, the last checkpoint, or a different worker via restore-based relocation. Branch creates a new run lineage from a checkpoint or stable point with bounded inherited budgets and policies — see [branch flow](/docs/concepts/branch-flow).

### Timeline / Event Projection Service

Projects the canonical [Step](/docs/concepts/steps), [ModelCall](/docs/concepts/model-calls), [tool invocation](/docs/concepts/tool-invocations), and [checkpoint event](/docs/concepts/events/checkpoint) streams into a runtime-native timeline view: typed step list, model and tool actions, [diffs](/docs/concepts/workspace-artifacts), [checkpoint](/docs/concepts/checkpoints) tree, [branch lineage](/docs/concepts/branch-flow), failure classification, retry/resume reasons. See [runtime observability](/docs/concepts/runtime-observability).

### Runtime SDK

Python-first SDK that emits the canonical entities through ergonomic decorators — `@kova.run`, `@kova.session`, `@kova.budget`, `@kova.step`. The SDK is a thin authoring surface; it does not replace the runtime API but makes the common case concise.

## What the runtime owns vs. does not own

The boundary is precise — and is enforced by what the runtime persists vs. what it resolves through Layer 2.

| Layer 3 owns | Layer 2 owns |
|---|---|
| Agent definition model | Canonical Run / Session / Step / ModelCall persistence and FSMs |
| Worker lease and assignment | [PolicyScope](/docs/concepts/policy-scopes), tenant resolution, [Budget](/docs/concepts/budgets) and [ContextBudget](/docs/concepts/context-budgets) admission |
| [Sandbox](/docs/concepts/sandboxes) lifecycle | Routing across the substrate fleet |
| [Tool registration and invocation](/docs/concepts/tool-invocations) semantics | [Tool catalog](/docs/concepts/tool-catalogs) versioning + [tool context governance](/docs/concepts/tool-context-disposition) |
| [Workspace](/docs/concepts/workspaces) and [WorkspaceArtifact](/docs/concepts/workspace-artifacts) model | [ContextManifest](/docs/concepts/context-manifests), [ContextSegment](/docs/concepts/context-segments), [ContextArtifact](/docs/concepts/context-artifacts), [CompactionRecord](/docs/concepts/compaction-records) |
| [Runtime-local state](/docs/concepts/runtime-local-state) | [Memory store](/docs/concepts/memory-store) and [MemoryScope](/docs/concepts/memory-scopes) |
| Checkpoint trigger rules at runtime boundaries | [Checkpoint Manager](/docs/concepts/checkpoints) capture, validation, restore |
| [Retry / resume / branch](/docs/concepts/branches) UX for the wedge | Run [state machine](/docs/concepts/run-state-machine) transitions |
| Runtime-native [timeline](/docs/concepts/runtime-observability) UX | Event taxonomy and [event envelope](/docs/concepts/event-envelope) |

The runtime never invents parallel context-engineering primitives. When it needs to compact context, it produces a [CompactionRecord](/docs/concepts/compaction-records) through the canonical API. When it needs to offload a tool result, it writes a [ContextArtifact](/docs/concepts/context-artifacts). When it needs to remember something across runs, it writes a [MemoryEntry](/docs/concepts/memory-entries) under the right [MemoryScope](/docs/concepts/memory-scopes). The Layer 2 surface is what makes the runtime *thin*.

## Wire shape — runtime API

The runtime exposes a small surface over the canonical control plane:

| Endpoint | Purpose |
|---|---|
| `POST /v1/ks/runtime/projects` | Create or get a runtime project. |
| `POST /v1/ks/runtime/agents` | Register an agent spec — model, system prompt, tool list, sandbox profile, defaults. |
| `POST /v1/ks/runtime/runs` | Trigger a Run against an agent and a workspace source. |
| `POST /v1/ks/runtime/runs/{id}/pause` | Pause a Run. |
| `POST /v1/ks/runtime/runs/{id}/resume` | Resume a Run from current step, previous step, or last checkpoint. |
| `POST /v1/ks/runtime/runs/{id}/branch` | Branch a new Run lineage from a Checkpoint or stable point. |
| `POST /v1/ks/runtime/runs/{id}/steps/{step_id}/retry` | Retry a safe/idempotent step. |
| `GET  /v1/ks/runtime/runs/{id}/timeline` | The runtime-native timeline view. |
| `GET  /v1/ks/runtime/runs/{id}/artifacts` | The WorkspaceArtifact catalog for the Run. |

## Deployment

The runtime ships as part of the same Helm chart as the Layer 2 control plane. On [managed cloud](/docs/concepts/managed-cloud) the runtime workers and sandboxes run inside the hosted cluster; on [private deployment](/docs/concepts/private-deployment) they run inside the customer's Kubernetes cluster, on the same nodes (or on dedicated runtime-pool nodes) as the rest of the control plane.
