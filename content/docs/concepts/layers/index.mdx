KovaServe is a three-layer system. **Layer 1** is the open inference substrate that executes model computation and exposes execution mechanics, KV state lifecycle, and restore hooks. **Layer 2** is the licensed control plane вҖ” the semantic and commercial center вҖ” that turns substrate signals into continuity, checkpoints, routing, policy, budgets, observability, and managed/private portability. **Layer 3** is a focused runtime wedge for coding, terminal, and tool-heavy long-running agents, built on top of Layer 2 without inventing parallel context-engineering primitives. The layers share a single execution vocabulary: `Run`, `Session`, `Step`, `ModelCall`, `ContextManifest`, `Checkpoint`, `Budget`, `PolicyScope`, `StateHandle`.

## Why three layers

The AI infrastructure stack is splitting into three distinct strata. At the bottom, open inference substrate (vLLM, llm-d) is improving rapidly at throughput, distributed serving, and cache handling. At the top, runtimes for agents and durable workflows (LangGraph, Temporal, custom) are solving orchestration, retries, and tool loops. The middle layer вҖ” the execution control plane вҖ” is the missing piece, and is the layer KovaServe productizes.

The three-layer model reflects that split. Layer 1 stays aligned with the open ecosystem and contributes upstream where contributions are generally useful. Layer 2 is the commercial center where continuity, resumability, economics, and governance become first-class. Layer 3 is a narrow product wedge that only exists because tighter coupling between runtime semantics and control-plane semantics produces clearly better outcomes for one class of workload.

## Layer 1 вҖ” open inference substrate

Layer 1 makes serving infrastructure visible and controllable enough for Layer 2 to reason about continuity, resumability, and execution economics using real substrate signals. It is intentionally substrate-focused вҖ” it carries semantics, it does not own them.

| Layer 1 owns | Layer 1 does not own |
|---|---|
| Model execution mechanics | Canonical execution model |
| Execution metadata propagation through the engine | Semantics of `run_id`, `session_id`, `checkpoint_id` |
| KV cache lifecycle hooks (store, access, evict, promote, demote, restore, transfer) | Cluster-wide KV index |
| Prefix/state visibility (hot, warm, cool, cold, partial) | Route scoring |
| Restore / import / export primitives | Restore strategy and planning |
| Tier movement integration (HBM, host RAM, NVMe, object) | Tier movement policy |
| Scheduler/admission advisory cooperation | Admission decisions |
| Node-local telemetry and `kv_events` emission | Cross-node aggregation |

Layer 1 is the subject of [`execution-substrate`](/docs/concepts/layers/execution-substrate).

## Layer 2 вҖ” licensed control plane

Layer 2 is the heart of KovaServe and the main commercial layer. It sits above the open substrate and below application/runtime layers, converting model serving and execution signals into continuity, checkpoints, routing, policy, budgets, governance, observability, and deployment portability. It is the canonical semantic center of the system вҖ” the layer of record for every entity in the canonical model.

| Layer 2 owns | Layer 2 does not own |
|---|---|
| Canonical execution model вҖ” `Run` / `Session` / `Step` / `ModelCall` persistence and FSMs | Model execution mechanics |
| Continuity-aware context normalization (`ContextManifest`, `ContextSegment`, `ContextArtifact`) | Prompt composition |
| State index and route planning | Raw KV cache placement |
| Checkpoints and logical-context-first restore planning | Per-engine state representation |
| Budgets and `ContextBudget` admission | Application-level cost analytics |
| Tenancy and chargeback | Identity provider implementation |
| Dashboards, APIs, SDK support, and managed/private portability | Customer agent code |

Layer 2 is the subject of [`control-plane`](/docs/concepts/layers/control-plane).

## Layer 3 вҖ” focused runtime wedge

Layer 3 is a focused runtime product built on the Layer 2 control plane. It exists where tighter coupling between runtime semantics and control-plane semantics produces clearly better outcomes вҖ” coding agents, terminal agents, tool-heavy long-running agents, filesystem and process-heavy loops. It uses the canonical entities as its foundation and adds wedge-specific semantics on top, never inventing parallel context-engineering primitives.

| Layer 3 owns | Layer 3 does not own |
|---|---|
| Agent definition model | Canonical execution truth |
| Worker model and lease ownership | `PolicyScope` / `Budget` / `ContextBudget` |
| Sandbox lifecycle (container-based, persistent, branchable) | Cluster-wide routing |
| Tool registration and invocation semantics | Global policy and budget enforcement |
| Workspace and `WorkspaceArtifact` model | `ContextManifest` / `ContextArtifact` / `CompactionRecord` |
| Runtime-local state (planner scratchpad, tool loop position) | Canonical entity persistence |
| Checkpoint trigger rules at runtime boundaries | Checkpoint capture, validation, restore |
| Retry / resume / branch UX for the wedge | `Run` state machine transitions |
| Runtime-native timeline UX | Event taxonomy and event envelope |

Layer 3 is the subject of [`runtime-wedge`](/docs/concepts/layers/runtime-wedge).

## What flows across the boundaries

The layers communicate through a small, fixed set of interfaces.

### Layer 1 вҶ” Layer 2

| Direction | Surface | Carries |
|---|---|---|
| Down | Execution metadata envelope on the engine request path | `run_id`, `session_id`, `step_id`, `model_call_id`, `context_manifest_id`, `stable_context_id`, `checkpoint_id`, `placement_hint`, `policy_scope_ref`, `memory_class_hints`, `trace_id` |
| Down | Restore / import / export instructions | KV state handles, target node, target tier |
| Down | Scheduler advisory hints | Admission preference, tier preference, locality preference |
| Up | `kv_events` substrate stream | `kv.state.stored`, `kv.state.accessed`, `kv.state.partial_match`, `kv.state.demoted`, `kv.state.promoted`, `kv.state.restored`, `kv.state.transferred`, `kv.state.evicted`, `kv.state.invalidated` |
| Up | Node telemetry (gRPC) | Queue depth, HBM pressure, KV occupancy, transfer latency, restore latency |
| Up | Usage primitives | Real input/output token counts, `cached_tokens`, model id, latency |

### Layer 2 вҶ” Layer 3

| Direction | Surface | Carries |
|---|---|---|
| Up | Canonical entity reads | `Run`, `Session`, `Step`, `ModelCall`, `ContextManifest`, `Checkpoint`, `Budget`, `PolicyScope` |
| Up | Admission decisions | `ContextBudget` decision (`ALLOW` / `WARN` / `OFFLOAD_REQUIRED` / `COMPACT_REQUIRED` / `DEGRADE` / `HARD_STOP`), policy decisions, run-level budget pressure |
| Down | Step transitions | `MODEL_CALL`, `TOOL_PLAN`, `TOOL_EXECUTE`, `FILE_READ`, `FILE_WRITE`, `PROCESS_START`, `OBSERVE`, `THINK`, `CHECKPOINT`, `PAUSE`, `RESUME`, `BRANCH`, `COMPLETE`, `FAIL`, `COMPACT` |
| Down | Checkpoint requests | Trigger reason, attached `runtime_state` `StateHandle` |
| Down | Tool invocations and `ContextArtifact` writes | Sandbox refs, artifact bytes |
| Down | Runtime events | `ks.tool.invocation.*`, `ks.process.*`, `ks.file.write.applied`, `ks.branch.created` |

## The shared execution vocabulary

The layers do not invent overlapping object models. The canonical vocabulary, defined by Layer 2, is what makes integration tractable across all three layers:

| Category | Entities |
|---|---|
| Execution | [Run](/docs/concepts/run-hierarchy/runs), [Session](/docs/concepts/continuity/sessions), [Step](/docs/concepts/run-hierarchy/steps), [ModelCall](/docs/concepts/run-hierarchy/model-calls) |
| Context | [ContextManifest](/docs/concepts/context/context-manifests), [ContextSegment](/docs/concepts/context/context-segments), [ContextArtifact](/docs/concepts/context/context-artifacts), [CompactionRecord](/docs/concepts/context/compaction-records) |
| Continuity | [Checkpoint](/docs/concepts/recovery/checkpoints), [StateHandle](/docs/concepts/recovery/state-handles) |
| Governance | [PolicyScope](/docs/concepts/policy/policy-scopes), [Budget](/docs/concepts/policy/budgets), [ContextBudget](/docs/concepts/policy/context-budgets) |
| Routing | [PlacementHint](/docs/concepts/routing/placement-hints) |

Layer 2 is the system of record for every one of these. Layer 1 carries identifiers through the engine path and emits events tied to them. Layer 3 reads them, transitions them, and produces wedge-specific entities (Workers, Sandboxes, Workspaces, WorkspaceArtifacts) that reference them.

## Where the boundaries are drawn

Three load-bearing rules govern what goes where.

**Substrate signals are real, not synthetic.** Routing, tiering, checkpointing, and economics rely on real Layer 1 events, not approximations. Layer 1 is what makes the control plane credible.

**Context engineering lives at the execution boundary, not in the runtime.** Layer 2 owns segment identity, manifest durability, admission, offload, compaction lineage, retention, restore, governance, and observability. Runtimes own prompt composition, planner logic, retrieval strategy, and agent memory authoring. Layer 3 follows the same rule вҖ” it consumes Layer 2 context primitives, it does not duplicate them.

**Managed and private deployments share one semantic model.** A `Run`, a `Checkpoint`, a `Budget`, a `PolicyScope`, and a `StateHandle` mean the same thing across [managed cloud](/docs/concepts/deployment/managed-cloud) and [private deployment](/docs/concepts/deployment/private-deployment). The architecture avoids hidden cloud-only assumptions; the layers are the same in both topologies.
