The **runtime wedge** is Layer 3 of the KovaServe layered model — a focused runtime product built on the [Layer 2 control plane](/docs/concepts/layers/control-plane) for one class of workload: coding agents, terminal agents, tool-heavy long-running agents, and filesystem and process-heavy loops. It exists because tighter coupling between runtime semantics and control-plane semantics produces clearly better outcomes for this wedge than a [BYO runtime](/docs/concepts/deployment/byo-runtime) integration could achieve. The product surface that runs on the wedge is [CodeRuns](/docs/concepts/managed-agents/coderuns), with [OpsRuns](/docs/concepts/managed-agents/opsruns) and [DocRuns](/docs/concepts/managed-agents/docruns) as adjacent SKUs in the same shape.

```
┌─ Layer 3 — runtime wedge ─────────────────────────────────────────────┐
│                                                                       │
│   Runtime API Service       ── /v1/ks/runtime/* (pause/resume/branch) │
│           │                                                           │
│           ▼                                                           │
│   Runtime Orchestrator ──▶ Worker Manager ──▶ Workers (leases)        │
│                                  │                  │                 │
│                                  ▼                  ▼                 │
│                           Sandbox Manager ──▶ Sandboxes (containers)  │
│                                  │                  │                 │
│                                  ▼                  ▼                 │
│                           Tool Registry ────▶ Tool Gateway            │
│                                  │                                    │
│                                  ▼                                    │
│                           Workspace / Artifact Manager                │
│                                  │                                    │
│                                  ▼                                    │
│                           Runtime State Store                         │
│                                                                       │
│           ├──▶ Checkpoint Integration Module                          │
│           ├──▶ Retry / Resume / Branch Manager                        │
│           └──▶ Timeline / Event Projection Service                    │
│                                                                       │
│   Runtime SDK (Python: @kova.run, @kova.session, @kova.budget,        │
│                        @kova.step)                                    │
└────────────────────────────▲──────────────────────────────────────────┘
                             │ canonical entities
                             ▼
                    Layer 2 — control plane
```

The wedge uses the canonical entities — `Run`, `Session`, `Step`, `ModelCall`, `ContextManifest`, `Checkpoint`, `Budget`, `PolicyScope`, `StateHandle` — as its foundation and adds wedge-specific semantics on top. Workspaces, Sandboxes, WorkspaceArtifacts, and runtime-local state are the wedge's native entities; everything else resolves through Layer 2.

## Why a wedge

The default integration mode for KovaServe is [BYO runtime](/docs/concepts/deployment/byo-runtime): the user keeps LangGraph, Temporal, or a custom runtime, and KovaServe sits underneath as the execution control plane. That mode is correct for most workloads.

For one class of workload — coding, terminal, tool-heavy, filesystem and process-heavy long-running agents — BYO is not enough. These agents need:

- A first-class **Workspace** model with branchable lineage.
- First-class **Sandboxes** for shell and file access.
- A **Tool Gateway** that dispatches each invocation, enforces sandbox boundaries, and emits per-invocation events.
- Wedge-aware **checkpoint trigger rules** — before risky file mutations, before unsafe shell actions, after stable analysis phases, on budget warning.
- **Retry / resume / branch** UX that targets the current step, the previous step, the last checkpoint, or a different worker.
- A **runtime-native timeline** rich enough to inspect every step, diff, tool result, and checkpoint.

These are not framework concerns. They are runtime concerns that compose better when they sit directly on top of the control plane than when they sit on top of a generic orchestrator. The wedge productizes that integration.

## Subsystems

The wedge is composed of eleven Layer 3 subsystems. Each one has a stable boundary against the control plane and against the other subsystems.

### Runtime API Service

The wedge's public HTTP surface. Hosts `POST /v1/ks/runtime/runs`, the pause / resume / branch / retry endpoints, the timeline endpoint, the artifacts endpoint, and the project / agent registration endpoints. Resolves [tenancy](/docs/concepts/tenancy), authenticates Bearer keys, normalizes runtime metadata, forwards into the orchestrator. The shape of the surface mirrors the control-plane API style.

### Runtime Orchestrator

Routes new `Run`s onto available [`Workers`](/docs/concepts/runtime/workers) under lease-based ownership. Owns the wedge-side decisions: which Worker claims which Run, when a Worker's lease is renewed, when a Run is reassigned. The orchestrator never holds canonical state — that lives in the [Canonical Execution Service](/docs/concepts/layers/control-plane#canonical-execution-service).

### Worker Manager

Manages the Worker pool. Workers claim runnable work, materialize runtime state, execute step transitions, interact with Layer 2 for model calls and checkpoints, interact with sandboxes and tools, renew or release leases, and emit runtime events. Everything required for resume is persisted in the [Runtime State Store](/docs/concepts/runtime/runtime-local-state) outside Worker memory; Worker reassignment is safe and progress survives lease loss.

### Sandbox Manager

Manages the [`Sandbox`](/docs/concepts/runtime/sandboxes) lifecycle. The wedge is process- and file-heavy by design, so isolated execution environments are first-class.

| Capability | Detail |
|---|---|
| Shell / process execution | Run shell commands inside the sandbox; capture stdout / stderr as artifacts. |
| Workspace mounting | Mount the [`Workspace`](/docs/concepts/runtime/workspaces) at a writable path. |
| Policy-limited access | File and network access governed by the [`PolicyScope`](/docs/concepts/policy/policy-scopes). |
| Output capture | Stdout, stderr, exit code, durations. |
| Resource accounting | CPU, memory, disk usage. |
| Optional persistence | Sandboxes can persist across the lifetime of a Run. |
| Branchable | Sandboxes can branch from a snapshot point so [branch flow](/docs/concepts/recovery/branch-flow) carries workspace state. |

Sandboxes are container-based.

### Tool Registry and Tool Gateway

The wedge-specific [tool invocation](/docs/concepts/runtime/tool-invocations) layer. Built-in tool classes:

| Class | Purpose |
|---|---|
| `shell` / command | Run a shell command inside the sandbox. |
| `file` | Read or write a file in the workspace. |
| `repo` / `git` | Operate on the bound Git repository. |
| `test` / `build` | Execute test or build runners; surface reports as artifacts. |
| `http` / `api` | Make outbound HTTP calls under policy. |
| `structured external` | Tool definitions provided by the customer or partner. |

The Tool Gateway dispatches each invocation, enforces sandbox boundaries, and emits `ks.tool.invocation.*` events.

The wedge owns wedge-specific invocation semantics. Layer 2 owns global policy and budget enforcement.

### Workspace / Artifact Manager

Owns the structured [`Workspace`](/docs/concepts/runtime/workspaces) model — workspace root, source binding, mutation log, snapshots, branch lineage, read/write policy — and the [`WorkspaceArtifact`](/docs/concepts/runtime/workspace-artifacts) catalog:

| Artifact class | Examples |
|---|---|
| Process outputs | stdout, stderr |
| Code changes | File diffs, patches |
| Verification | Test reports, build reports |
| Tool outputs | Tool call results, structured responses |
| Model outputs | Plan documents, model decisions |
| Snapshots | Workspace snapshots at checkpoint time |
| Notes | Checkpoint notes |

Artifact bytes land in object storage; the catalog rows live in Postgres alongside the canonical entity tables.

### Runtime State Store

Holds [runtime-local state](/docs/concepts/runtime/runtime-local-state) — the state that is the wedge's responsibility but that is not part of the canonical execution model. Examples: planner scratchpad, active process handles, tool loop position, workspace refs, branch pointers, last safe checkpoint. **Anything required for resume must live here**, outside Worker memory.

### Checkpoint Integration Module

Layered on top of the Layer 2 [Checkpoint Manager](/docs/concepts/recovery/checkpoints). The actual checkpoint capture goes through the canonical Layer 2 path; what the wedge adds is the *when*. Wedge-aware trigger rules:

| Trigger | Reason |
|---|---|
| Before risky file mutations | Workspace state may need to be reverted. |
| Before unsafe shell actions | Process side effects may need to be rolled back. |
| After stable analysis phases | The thinking is reusable; cache it. |
| On budget warning | Preserve progress before degradation. |
| Manual | Operator-initiated. |
| Periodic | Configurable interval. |

Checkpoints attach a `runtime_state` [`StateHandle`](/docs/concepts/recovery/state-handles) to the canonical record so resume can reconstitute the runtime view alongside the logical context.

### Retry / Resume / Branch Manager

Owns the runtime UX for [retries](/docs/concepts/recovery/retries), [resumes](/docs/concepts/recovery/resume-flow), and [branches](/docs/concepts/recovery/branches).

| Operation | Targets |
|---|---|
| Retry | Only safe and idempotent runtime steps retry in place. |
| Resume | Current step, previous step, last checkpoint, or a different Worker via restore-based relocation. |
| Branch | A new `Run` lineage from a `Checkpoint` or stable point with bounded inherited budgets and policies. |

See [branch flow](/docs/concepts/recovery/branch-flow) for the full branch transition.

### Timeline / Event Projection Service

Projects the canonical [`Step`](/docs/concepts/run-hierarchy/steps), [`ModelCall`](/docs/concepts/run-hierarchy/model-calls), [tool invocation](/docs/concepts/runtime/tool-invocations), and [checkpoint event](/docs/concepts/events/checkpoint) streams into a runtime-native timeline view: typed step list, model and tool actions, [diffs](/docs/concepts/runtime/workspace-artifacts), [checkpoint](/docs/concepts/recovery/checkpoints) tree, [branch lineage](/docs/concepts/recovery/branch-flow), failure classification, retry/resume reasons. See [runtime observability](/docs/concepts/runtime/runtime-observability).

### Runtime SDK

Python-first. Emits the canonical entities through ergonomic decorators:

| Decorator | Purpose |
|---|---|
| `@kova.run` | Mark a function as a Run; SDK creates the canonical Run row and binds budgets, policy, session. |
| `@kova.session` | Bind execution to a Session for continuity. |
| `@kova.budget` | Attach a Budget at function scope. |
| `@kova.step` | Mark an inner function as a typed Step. |

The SDK is a thin authoring surface; it does not replace the runtime API but makes the common case concise.

## Step types

The wedge uses **typed step progression** — the canonical `StepType` enum, drawn from the same definition Layer 2 owns. The wedge surfaces a richer set of step types than a generic orchestrator because the workload demands it.

| Group | Step types |
|---|---|
| Standard agent loop | `MODEL_CALL`, `TOOL_PLAN`, `TOOL_EXECUTE` |
| Workspace / process | `FILE_READ`, `FILE_WRITE`, `PROCESS_START` |
| Reflection | `OBSERVE`, `THINK` |
| Durability and lineage | `CHECKPOINT`, `PAUSE`, `RESUME`, `BRANCH` |
| Terminal | `COMPLETE`, `FAIL` |
| Context | `COMPACT` |

Validation is application-level (TEXT column with enum check), so new types extend the canonical list without DDL changes. The wedge may also surface a `process_waiting` runtime *event* (`ks.process.waiting`) without a corresponding StepType — events and step types are separate enumerations. See [step types](/docs/concepts/run-hierarchy/step-types) and [runtime execution semantics](/docs/concepts/runtime/runtime-execution-semantics).

## Where things live

The boundary is enforced by what the wedge persists vs. what it resolves through Layer 2.

| Layer 3 owns | Layer 2 owns |
|---|---|
| Agent definition model | Canonical `Run` / `Session` / `Step` / `ModelCall` persistence and FSMs |
| Worker lease and assignment | [`PolicyScope`](/docs/concepts/policy/policy-scopes), tenant resolution, [`Budget`](/docs/concepts/policy/budgets) and [`ContextBudget`](/docs/concepts/policy/context-budgets) admission |
| [`Sandbox`](/docs/concepts/runtime/sandboxes) lifecycle | Routing across the substrate fleet |
| [Tool registration and invocation](/docs/concepts/runtime/tool-invocations) semantics | Global policy and budget enforcement |
| [`Workspace`](/docs/concepts/runtime/workspaces) and [`WorkspaceArtifact`](/docs/concepts/runtime/workspace-artifacts) model | [`ContextManifest`](/docs/concepts/context/context-manifests), [`ContextSegment`](/docs/concepts/context/context-segments), [`ContextArtifact`](/docs/concepts/context/context-artifacts), [`CompactionRecord`](/docs/concepts/context/compaction-records) |
| [Runtime-local state](/docs/concepts/runtime/runtime-local-state) | Canonical entity persistence |
| Checkpoint trigger rules at runtime boundaries | [Checkpoint Manager](/docs/concepts/recovery/checkpoints) capture, validation, restore |
| [Retry / resume / branch](/docs/concepts/recovery/branches) UX for the wedge | Run [state machine](/docs/concepts/run-hierarchy/run-state-machine) transitions |
| Runtime-native [timeline](/docs/concepts/runtime/runtime-observability) UX | Event taxonomy and [event envelope](/docs/concepts/events/event-envelope) |

When the wedge needs to compact context, it produces a `CompactionRecord` through the canonical API. When it needs to offload a tool result, it writes a `ContextArtifact`. The Layer 2 surface is what makes the wedge *thin*.

## API surface

The wedge exposes a small, focused surface over the canonical control plane.

| Endpoint | Purpose |
|---|---|
| `POST /v1/ks/runtime/projects` | Create or get a runtime project. |
| `POST /v1/ks/runtime/agents` | Register an agent spec — model, system prompt, tool list, sandbox profile, defaults. |
| `POST /v1/ks/runtime/runs` | Trigger a Run against an agent and a workspace source. |
| `POST /v1/ks/runtime/runs/{id}/pause` | Pause a Run. |
| `POST /v1/ks/runtime/runs/{id}/resume` | Resume a Run from current step, previous step, or last checkpoint. |
| `POST /v1/ks/runtime/runs/{id}/branch` | Branch a new Run lineage from a Checkpoint or stable point. |
| `POST /v1/ks/runtime/runs/{id}/steps/{step_id}/retry` | Retry a safe / idempotent step. |
| `GET  /v1/ks/runtime/runs/{id}/timeline` | The runtime-native timeline view. |
| `GET  /v1/ks/runtime/runs/{id}/artifacts` | The `WorkspaceArtifact` catalog for the Run. |

Wire shapes share the [error envelope](/docs/api/using-the-api/errors), [pagination](/docs/api/using-the-api/pagination), [idempotency](/docs/api/using-the-api/idempotency), [cost headers](/docs/api/using-the-api/cost-headers), and [SSE streaming](/docs/api/using-the-api/streaming) conventions used elsewhere in the API.

## Wedge-specific events

The wedge participates in the canonical [event taxonomy](/docs/concepts/events) and contributes the runtime stream:

| Event | Meaning |
|---|---|
| `ks.tool.invocation.started` | Tool call dispatched to the gateway. |
| `ks.tool.invocation.completed` | Tool call returned. |
| `ks.process.started` | A sandboxed process started. |
| `ks.process.waiting` | A process is blocked on input or external state. |
| `ks.file.write.applied` | A file write landed in the workspace. |
| `ks.branch.created` | A branch lineage was created from a checkpoint or stable point. |

Combined with the canonical execution and checkpoint events, these power the runtime-native timeline.

## Product packaging

The wedge ships under managed-agent SKUs that compose the runtime, default policies, default budgets, default tool catalogs, and default sandbox profiles into a productized offering. The reference SKUs:

| SKU | Workload |
|---|---|
| [CodeRuns](/docs/concepts/managed-agents/coderuns) | Software engineering — bug fixing, test repair, repo analysis, migrations, CI failure investigation. |
| [OpsRuns](/docs/concepts/managed-agents/opsruns) | IT, cloud, and security operations — incident investigation, log analysis, runbook execution, diagnostics, policy-controlled remediation. |
| [DocRuns](/docs/concepts/managed-agents/docruns) | Document-heavy enterprise workflows — contract review, compliance checks, redline assistance, clause extraction, document reconciliation. |

The SKUs are configurations of the same wedge. The wedge is the engine; the SKU is the configured product.

## Deployment

The wedge ships as part of the same Helm chart as the [Layer 2 control plane](/docs/concepts/layers/control-plane). On [managed cloud](/docs/concepts/deployment/managed-cloud) the runtime workers and sandboxes run inside the hosted cluster; on [private deployment](/docs/concepts/deployment/private-deployment) they run inside the customer's Kubernetes cluster, on the same nodes (or on dedicated runtime-pool nodes) as the rest of the control plane.
