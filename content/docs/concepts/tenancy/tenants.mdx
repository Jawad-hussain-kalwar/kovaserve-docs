A **Tenant** is the billing root and top-level governance container in the [tenancy hierarchy](/docs/concepts/tenancy). It owns the contract relationship, the default model ACLs, the default rate limits, the default data-boundary, and every [Project](/docs/concepts/tenancy/projects) underneath. One Tenant corresponds to one billed customer.

Every other entity in KovaServe — Projects, Environments, Keys, Sessions, Runs, ModelCalls, ContextArtifacts — resolves up to exactly one Tenant.

## What a Tenant owns

| Field | Purpose |
|---|---|
| `id` | Bare 26-char Crockford ULID. |
| `name` | Human-readable identifier. Not used for routing or auth. |
| `status` | `ACTIVE` or `SUSPENDED`. Suspended Tenants reject all admission with `403 Forbidden`. |
| `default_policy_scope_id` | The PolicyScope applied to any Project that does not declare its own override. |
| `default_data_boundary` | `tenant_local` / `region_local` / `unrestricted` — the residency floor for ContextArtifacts and KV exports across all Projects. |
| `default_rate_limits` | Tenant-wide ceiling on requests-per-second and tokens-per-second. Sums across all Projects. |
| `default_model_acl` | Allowlist of model IDs the Tenant is contracted for. Project-level ACLs can subset this list but not extend it. |
| `created_at` | Timestamp. |

The Tenant-level fields are **defaults and ceilings**, not per-request enforcement. The actual enforcement happens at admission time after the full hierarchy overlays — see [PolicyScope](/docs/concepts/policy/policy-scopes#resolution).

## Wire shape

```http
GET /v1/ks/tenants/{id}
Authorization: Bearer ks_...
```

```json
{
  "id": "01J8X9Y2Z3K4M5N6P7Q8R9TEN1",
  "name": "Acme Corp",
  "status": "ACTIVE",
  "default_policy_scope_id": "01J8X9Y2Z3K4M5N6P7Q8R9P0K0",
  "default_data_boundary": "tenant_local",
  "default_rate_limits": {
    "requests_per_second": 200,
    "tokens_per_second": 4000000
  },
  "default_model_acl": ["qwen3-coder-30b", "qwen3-coder-7b"],
  "created_at": "2026-01-15T09:00:00Z"
}
```

A Bearer key resolves to exactly one Tenant. The `GET /v1/ks/tenants/{id}` endpoint accepts only the resolved `tenant_id` for the authenticated key — calls for other IDs return `404 Not Found`.

## Identity scope

The Tenant is the **opaque boundary** of identity:

- A Key in Tenant A cannot read, write, or reference any entity in Tenant B. Cross-tenant access uniformly returns `404 Not Found`, not `403`, so probing leaks no information.
- The Run, Session, ContextArtifact, and ModelCall ID spaces are tenant-scoped — the same ULID could in principle exist in two Tenants and neither would see the other's row.
- Cache reuse never crosses Tenants. The [Cache-Aware Router](/docs/concepts/routing/manifest-aware-routing) treats `tenant_id` as a hard scope filter regardless of how `context_share_scope` is configured at narrower levels.

## What sums up at the Tenant level

Cost, usage, and budget surfaces aggregate up the hierarchy. Tenant-level totals are the sum across all Projects and Environments:

| Surface | Endpoint |
|---|---|
| Usage and cost | `GET /v1/ks/metering/usage?scope=tenant` |
| Per-day breakdown | `GET /v1/ks/metering/usage?scope=tenant&group_by=day` |
| Per-project breakdown | `GET /v1/ks/metering/usage?scope=tenant&group_by=project` |
| Per-model breakdown | `GET /v1/ks/metering/usage?scope=tenant&group_by=model` |
| JSONL export | `GET /v1/ks/metering/export?scope=tenant` |

`scope=tenant` returns `200` with zero counts when there is no traffic in the requested period — see [cost-model aggregation](/docs/concepts/cost/cost-model#aggregation) for the full error envelope.

## Suspension

A Tenant can be moved to `SUSPENDED`. The effect is immediate:

| Surface | Behavior under SUSPENDED |
|---|---|
| New ModelCall admission | Reject with `403 Forbidden`, `error.code = "tenant_suspended"`. |
| In-flight Runs | Continue until next Step admission, where they reject and the Run transitions to `FAILED` with `error.code = "tenant_suspended"`. |
| Read-only endpoints | Continue to serve. Suspended customers can still inspect their data and exports. |
| Data retention | Unchanged — suspension is operational, not a deletion event. |

Suspension is set out-of-band by the platform operator. There is no API for self-suspension.

## Inheritance

| Operation | Tenant-level effect |
|---|---|
| Create [Project](/docs/concepts/tenancy/projects) | Inherits `default_policy_scope_id`, `default_data_boundary`, `default_rate_limits`, `default_model_acl`. The Project may narrow but not widen any of these. |
| Create [Environment](/docs/concepts/tenancy/environments) | Same — Environment-level overrides narrow against the Project's resolved values, which already narrow against the Tenant. |
| Create [Key](/docs/concepts/tenancy/keys) | A Key cannot be issued for a Tenant that is `SUSPENDED`. |

The Tenant's `default_model_acl` is a **hard ceiling**: a Project ACL cannot list a model not on the Tenant's ACL. The resolution rule (more-specific can narrow, never widen) makes this property robust regardless of how many overlay levels are involved.

## Lifecycle

The Tenant lifecycle is a two-state surface: `ACTIVE` and `SUSPENDED`. Suspension is operational — data, exports, and read-only endpoints continue to serve while admission rejects. Tenants are reactivated by clearing the suspension out-of-band; the data retention surface is governed separately. A small customer with one application still has one Tenant with one Project beneath, since Tenants carry billing and Projects carry workloads — the levels exist independently for that reason.
