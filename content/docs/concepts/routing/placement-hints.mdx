A **PlacementHint** is an execution preference supplied by the runtime or by an operator that the [Cache-Aware Router](/docs/concepts/routing/manifest-aware-routing) considers as a **soft** input to scoring. Hints express intent — "prefer this node", "prefer the replica that warmed the prior turn", "avoid this region" — without overriding hard scope filters or substrate health constraints.

Hints are KovaServe's mechanism for runtimes to communicate placement intent. The router treats them as scoring weights and combines them with continuity, economics, and policy signals to choose a replica.

## What hints carry

| Field | Purpose |
|---|---|
| `prefer_node_id` | Soft preference for a specific replica node. |
| `prefer_session_affinity` | Boolean. When true, the router weights the Session's most-recent replica more heavily. |
| `prefer_warmness_minimum` | Minimum acceptable warmness class — `hot` / `warm` / `cool`. The router prefers candidates at or above this class but may fall through. |
| `avoid_node_ids[]` | Soft denylist for replica nodes. Used during partial drains. |
| `avoid_regions[]` | Soft denylist for regions. Subordinate to `policy_scope.data_boundary`. |
| `region_affinity` | Soft preference for a region (when `data_boundary` permits). |
| `latency_class` | `interactive` / `batch`. `interactive` weighs queue depth heavily; `batch` weighs cache reuse heavily. |

Hints left unset have no effect. Multiple hints combine — the router scores against the full vector.

## Wire shape

Hints travel on the inference path through the `kovaserve` extension envelope:

```json
{
  "kovaserve": {
    "placement_hint": {
      "prefer_session_affinity": true,
      "prefer_warmness_minimum": "warm",
      "latency_class": "interactive"
    }
  }
}
```

The field name is **singular** (`placement_hint`) — runtimes supply one hint object per request, not an array. Multiple preferences combine inside the single object.

## Hard filter vs soft hint

Hints are **soft**. They affect scoring weight but cannot override hard filters:

| | Source | Hard or soft |
|---|---|---|
| Model availability | manifest's model_id | hard |
| Tokenizer / renderer compatibility | manifest fields | hard |
| `policy_scope.context_share_scope` | PolicyScope | hard |
| `policy_scope.data_boundary` | PolicyScope | hard |
| `prefer_node_id` | hint | soft |
| `avoid_node_ids[]` | hint | soft |
| `avoid_regions[]` | hint | soft |
| `prefer_warmness_minimum` | hint | soft |
| `region_affinity` | hint | soft |
| `latency_class` | hint | soft |

A hint that conflicts with a hard filter is silently ignored. A request with `avoid_node_ids: ["node-X"]` whose only compatible replica is on `node-X` still routes there — the alternative would be a hard failure for a soft preference.

## Operator hints

Operators express placement intent through cluster-level overrides, not per-request hints. The operator surface is documented under [`POST /v1/ks/routing/plan`](/docs/api/cluster/routing-plan); typical uses are draining a node for maintenance or pinning a Run during an incident investigation.

## How the router applies hints

```
final_score = (
    base_continuity_score
    + manifest_reuse_weight
    - queue_cost
    + hint_match_bonus       ◀── soft contribution
    - hint_avoid_penalty     ◀── soft contribution
)
```

`hint_match_bonus` and `hint_avoid_penalty` are bounded weights. Their magnitude is calibrated so that strong continuity signals (a hot KV match) override weak hint preferences, but a clear hint (matching `prefer_node_id` exactly) tips the balance between similarly scored candidates.

Hint match outcomes are recorded on the routing rationale:

```json
{
  "rationale": {
    "hint_applied": "prefer_session_affinity",
    "hint_match": true,
    "selected_node_id": "01J8X9Y2Z3K4M5N6P7Q8R9N002"
  }
}
```

`hint_match: false` is also valid — the hint was considered, did not align with the router's preferred candidate, and was overridden by stronger signals.

## Latency class

`latency_class` is the most consequential hint for runtimes that mix interactive and batch work:

| Value | Effect |
|---|---|
| `interactive` | Queue depth and warmness weighted heavily; cache reuse subordinate to "available now". |
| `batch` | Cache reuse and prefill savings weighted heavily; queue depth tolerated. |

A long-running batch Run that prefers cache reuse over latency benefits from `latency_class: batch`. An interactive coding-agent Run benefits from `latency_class: interactive`.

## Session affinity

`prefer_session_affinity: true` is the most common hint. It causes the router to score the Session's most-recently-used replica higher, capturing the common case where conversational continuity benefits from sticky placement. The hint is soft — a heavily loaded prior replica yields to a fresh cold one when the queue cost outweighs reuse.

## Lifecycle and persistence

Hints are scoped to the request that carries them. Each subsequent request restates whichever hints the runtime still wants applied; absent hints contribute nothing to scoring. Hints land on the routing audit event payload for the request and are not persisted on the Run row.
