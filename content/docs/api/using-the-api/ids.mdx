KovaServe identifies entities with two formats, both server-generated, both tenant-scoped:

- **Bare 26-character Crockford ULIDs** — used for almost every entity on the surface.
- **Prefixed Session IDs** (`ses_<24hex>`) — the one deliberate exception.

This page is the API-side reference for what those formats look like on the wire and how the gateway validates them. For the rationale and full identity model — including idempotency keys, `external_run_id`, and trace correlation — see [Identity and IDs](/docs/concepts/run-hierarchy/identity-and-ids) in the concepts section.

## ULID format

A KovaServe ULID is a 26-character uppercase string drawn from Crockford's base32 alphabet:

```
01J8X9Y2Z3K4M5N6P7Q8R9S0T1
```

| Property | Value |
|---|---|
| Length | exactly 26 characters |
| Alphabet | `0–9`, `A–Z` excluding `I`, `L`, `O`, `U` |
| Case on the wire | uppercase canonical |
| Case on input | accepted in either case; normalized to uppercase before validation |
| Sortability | lexicographic sort = creation-time sort (first 48 bits encode a millisecond timestamp) |
| Entropy | 128 bits |

The ULID format applies to: `run_id`, `step_id`, `model_call_id`, `checkpoint_id`, `context_manifest_id`, `context_segment_id`, `context_artifact_id`, `compaction_record_id`, `budget_id`, `context_budget_id`, `policy_scope_id`, `state_handle_id`, `tenant_id`, `project_id`, `environment_id`, `key_id`, `event_id`, and `X-Request-Id` response headers.

## Session ID format

Session IDs are the literal prefix `ses_` followed by 24 lowercase hex characters:

```
ses_a7f3c2b8d4e1f0925a1b6c3d
```

| Property | Value |
|---|---|
| Length | exactly 28 characters (`ses_` + 24 hex) |
| Alphabet | `0–9`, `a–f` after the prefix |
| Case on the wire | lowercase canonical |
| Case on input | accepted in either case; normalized to lowercase before validation |
| Entropy | 96 bits |

Sessions get a distinguishable prefix because they are the long-lived continuity container that developers paste between systems — into log lines, support tickets, runbooks. The prefix saves a glance. Every other entity uses a bare ULID.

## On the wire

Every JSON response emits IDs in canonical form — uppercase Crockford for ULIDs, lowercase hex for Session IDs:

```json
{
  "id": "01J8X9Y2Z3K4M5N6P7Q8R9S0T1",
  "session_id": "ses_a7f3c2b8d4e1f0925a1b6c3d",
  "context_manifest_id": "01J8X9Y2Z3K4M5N6P7Q8R9S0T7",
  "policy_scope_id": "01J8X9Y2Z3K4M5N6P7Q8R9S0T4"
}
```

Every JSON request and every URL path accepts IDs in either case. The gateway normalizes inputs before validation.

## Validation

A malformed ID — wrong length, wrong alphabet, missing or wrong prefix on a Session field — returns `400 Bad Request` with `error_code: "invalid_id"` *before* any tenancy or scope resolution runs:

```http
HTTP/1.1 400 Bad Request
Content-Type: application/json

{
  "error_code": "invalid_id",
  "message": "Path parameter 'run_id' is not a valid 26-character Crockford ULID.",
  "details": { "field": "run_id", "value": "01J8X9Y2Z3K4M5N6P7Q8R9S0!!" }
}
```

Validation is strict: a 27-character value, a value containing `I`/`L`/`O`/`U`, a Session ID without the `ses_` prefix, and a Session ID with uppercase hex are all rejected at the gateway boundary. Bad IDs never reach storage and never count against rate limits or budgets.

## Tenant scoping

Every ID is scoped to its owning `(tenant_id, project_id)` pair. A read, write, transition, or reference targeting an ID owned by another Tenant returns `404 Not Found` — the same response as a genuinely missing entity.

This is uniform across every entity on the surface: probing another Tenant's IDs leaks no information. The two consequences:

- **Authorization requires the API key's Tenant scope.** The ID identifies a row; the caller's [AuthContext](/docs/api/authentication) determines whether that row is visible.
- **Tenants are isolated.** Even if two Tenants generated colliding ULIDs (entropy makes this unlikely), each Tenant only ever sees rows in its own scope.

## Server-generated by default

Every KovaServe entity ID is **server-generated** at insert time. Clients do not supply ULIDs or Session IDs on create requests; the gateway generates them and returns them in the `Location` header (for `201` responses) and the response body.

There are two specific places where clients **do** supply identifiers:

- **`Idempotency-Key`** — a request-level header for safe retries. See [Idempotency](/docs/api/idempotency).
- **`external_run_id`** — a runtime's own identifier for a logical task, recorded on the [Run](/docs/concepts/run-hierarchy/runs#how-a-run-is-created) row. Idempotency-keyed per `(tenant_id, external_runtime, external_run_id)`.

Neither is a ULID. Neither is addressable.

## Request IDs and trace correlation

Every response carries `X-Request-Id`, a 26-character ULID assigned by the gateway:

```http
HTTP/1.1 200 OK
X-Request-Id: 01J8X9Y2Z3K4M5N6P7Q8R9S0XX
```

Include it when reporting an issue. The gateway records `X-Request-Id` on every execution event and audit row, so a single value reconstructs everything that happened during the request.

W3C `traceparent` is also honored when supplied:

```
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
```

When absent, the gateway generates one. The trace ID is recorded on the [ModelCall](/docs/concepts/run-hierarchy/model-calls) and propagated through every downstream module, so a single trace stitches together the gateway request, the manifest assembly, the routing decision, and the substrate execution. See [Identity and IDs — Trace correlation](/docs/concepts/run-hierarchy/identity-and-ids#trace-correlation) for the propagation contract.

## Quick reference

| Identifier | Format | Source | Addressable | Use |
|---|---|---|---|---|
| Entity IDs (`run_id`, etc.) | 26-char Crockford ULID | server | yes | identifies a row |
| Session ID | `ses_<24 lowercase hex>` | server | yes | continuity container ID |
| `external_run_id` | free-form string, ≤ 255 chars | client | no | runtime's own task ID, recorded on the Run |
| `Idempotency-Key` | free-form string, 1–255 chars | client | no | safe-retry header on mutating requests |
| `X-Request-Id` | 26-char Crockford ULID | server (response) | no | per-request correlation across logs |
| `traceparent` | W3C trace context | client or server | no | OpenTelemetry trace correlation |
