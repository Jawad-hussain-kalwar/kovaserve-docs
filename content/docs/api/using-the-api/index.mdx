KovaServe's HTTP API is the public interface to the [control plane](/docs/concepts/layers/control-plane). It exposes two coexistent families on the same base URL:

- **OpenAI-compatible inference** — `/v1/chat/completions` and `/v1/models`. Drop-in usage from any OpenAI SDK works without code changes; the `kovaserve` extension envelope opts into native features when you want them.
- **Native control plane** — `/v1/ks/*` for [Runs](/docs/concepts/run-hierarchy/runs), [Sessions](/docs/concepts/continuity/sessions), [Checkpoints](/docs/concepts/recovery/checkpoints), context engineering, budgets, policy, metering, tenancy, runtime, events, and cluster operations. These endpoints use KovaServe's native vocabulary.

Both families share authentication, idempotency, ID conventions, and cost accounting. Where they diverge is the error envelope and the streaming convention — see [Errors](/docs/api/using-the-api/errors) and [Streaming](/docs/api/using-the-api/streaming).

## Base URL

The managed cloud base URL is:

```
https://api.kovaserve.com
```

Self-hosted deployments expose the gateway on a configurable HTTP endpoint set during installation; consult your cluster operator for the local URL. The path layout (`/v1/...`, `/v1/ks/...`) is identical across managed and private deployments — the same SDK, same paths, same request shapes work in both.

## How a request flows

Every authenticated request resolves through the same admission ordering before it reaches a model:

```
client request
   │
   ▼
authentication                  resolve API key → AuthContext
   │
   ▼
tenancy resolution              Tenant → Project → Environment → Deployment
   │
   ▼
context assembly                build the ContextManifest for this call
   │
   ▼
context admission               ContextBudget evaluates the manifest
   │
   ▼
run-budget pre-check            Budget evaluates against the admitted manifest
   │
   ▼
routing                         Cache-Aware Router selects placement
   │
   ▼
substrate execution             Layer 1 runs the model
   │
   ▼
response + metering             cost headers, execution events, usage records
```

Each stage is observable through the [event taxonomy](/docs/concepts/events) and recorded on the [ContextManifest](/docs/concepts/context/context-manifests) the call consumed.

## What every page on this surface gives you

API pages on `docs.kovaserve.com` follow one shape:

1. A one-line description of the endpoint.
2. The HTTP method and path.
3. The authentication scope required.
4. Request schema with a JSON example.
5. Response schema with a JSON example.
6. Status-code semantics — what each non-2xx response means, including the error envelope.
7. A `See also` block linking to the relevant concept page, SDK method, and CLI subcommand.

Every example uses concrete IDs, shapes, and headers that match the wire format the gateway emits.

## Compatibility

KovaServe versions its API additively. New optional fields and new endpoints may appear within `/v1/`; existing fields and existing endpoints retain their shape. Breaking changes ship under a new major version path.

The `kovaserve` extension envelope on `/v1/chat/completions` is the seam for native features that have no OpenAI counterpart. Clients that ignore the envelope continue to work as plain OpenAI clients; clients that opt in get [run identity](/docs/concepts/run-hierarchy/runs), [context manifests](/docs/concepts/context/context-manifests), [policy scopes](/docs/concepts/policy/policy-scopes), [placement hints](/docs/concepts/routing/placement-hints), and the rest of the native surface inline with their existing call.
